# Phase 38.3 Plan 01: Bk2Kombi::CommitInning Service Summary

**One-liner:** New thin service `Bk2Kombi::CommitInning` implements point-entry paradigm (Variante B) with D-11 DZ opponent-credit and D-12 SP signed-additive scoring, delegating set-close/match-close logic to AdvanceMatchState via Pattern A send().

## What Was Built

`Bk2Kombi::CommitInning` is a new service in `app/services/bk2_kombi/commit_inning.rb` that:

- Accepts `table_monitor:`, `player:`, `inning_total:`, `shot_sequence_number: nil`
- Applies a whole-Aufnahme signed integer to `bk2_state` per phase rules (D-11/D-12)
- Reuses AdvanceMatchState's `close_set_if_reached!` / `close_match_if_reached!` via Pattern A delegation (`send`)
- Returns `{state:, transitions:}` — same shape as AdvanceMatchState.call
- Idempotency via `shot_sequence_number` stored in `bk2_state["applied_commit_sequences"]`
- Guards on `player` whitelist, `Integer` type, and `-999..999` range

## Pattern Chosen

**Pattern A (delegation)** — `CommitInning` instantiates an `AdvanceMatchState` helper and calls its private helpers via `send(:close_set_if_reached!, state)` and `send(:close_match_if_reached!, state)`.

Pattern B (module extraction) was not chosen because the helpers access `@tm` instance variables and read from `bk2_options` through instance methods — extracting them to a shared module would require a more complex interface with no current second consumer beyond this plan.
