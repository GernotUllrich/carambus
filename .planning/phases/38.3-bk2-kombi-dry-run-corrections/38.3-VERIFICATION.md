### Gaps

#### I6 (Blocker) — Shootout → Playing transition fails with GAP-05 fallback

**Severity:** Blocker — all further manual testing (HUMAN-UAT 1/2/3, I7 scoring verification) is impossible until fixed.

**Observed:** After clicking any of the 4 BK2 shootout buttons (e.g., "Spieler A — Direkter Zweikampf"), the scoreboard renders the GAP-05 fallback banner "Spiel nicht initialisiert — Bitte das Spiel über die Startansicht neu starten." instead of the playing scoreboard.

**Expected:** After the shootout button click, AASM transitions to `playing`, `bk2_state` is fully initialized (current_phase reflects the chosen first_set_mode, set_scores populated, innings/shots counters at configured max), and the scoreboard renders with phase chip + remaining badge + zero scores.

**Root cause hypothesis:**
- Plan 38.3-05's reflex augmentation writes `bk2_options["first_set_mode"]` from the dataset, then the existing AASM transition to `playing` fires.
- The playing-state scoreboard calls `bk2_state_uninitialized?`, which returns true — so `bk2_state` is empty at the moment of render.
- Either `AdvanceMatchState#initialize_bk2_state!` (or equivalent) is never called on the shootout → playing transition, or it runs before the `bk2_options["first_set_mode"]` write persists.
- The existing `start_game` / `switch_players_and_start_game` reflex flow handles Karambol's `set_scores` init but lacks a BK2-specific initialization hook.

**Likely files:** `app/reflexes/table_monitor_reflex.rb`, `app/services/advance_match_state.rb`, `app/models/table_monitor.rb`.
