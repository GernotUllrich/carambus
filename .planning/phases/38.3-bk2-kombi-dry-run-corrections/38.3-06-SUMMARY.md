## Threat Surface

No new security surface introduced beyond what the plan's threat model covers. The two new fields read from `params[:bk2_options]` (not the slice-filtered `p`) and are immediately CLAMPed to `(1..99)` before any persistence — silent fallback to defaults (2/5) on out-of-range or missing input. No JSONB bloat risk (integer values, trivially small).

T-38.3-06-02 (set_target_points pre-existing CLAMP) verified via grep — not re-implemented.
T-38.3-06-03 (Alpine scope-lift) verified via `grep -c 'x-data="{' == 1`.

## Deviations from Plan

None — plan executed exactly as written.

The plan suggested reading `params[:bk2_options]` via an `ActionController::Parameters` permit block (Step B pseudocode). Implementation instead reads the nested hash directly from `params` (not `p`) and applies the CLAMP immediately — this is equivalent in security (CLAMP is the boundary) and simpler (avoids double-handling). Both quick-game and free-game branches follow the same pattern.

## Known Stubs

None. Both new fields are wired end-to-end: Alpine state → hidden inputs → controller CLAMP → `bk2_options` hash → persisted to `table_monitor.data`.
