<objective>
Close I6 (blocker): clicking any of the 4 shootout buttons transitions AASM to `playing` but `bk2_state` is never initialized — the scoreboard renders the GAP-05 fallback banner "Spiel nicht initialisiert" instead of the playing layout. Root cause confirmed by code reading: **Candidate 1** — `AdvanceMatchState#init_state_if_missing!` is private and only invoked when a scoring event arrives (`Bk2Kombi::AdvanceMatchState.call(...)` is only called from `ScoreShot` paths, which are dormant in Variante B). The shootout→playing path (Plan 38.3-05) writes `bk2_options[first_set_mode]` correctly but never triggers the init.

**Fix:** expose `init_state_if_missing!` via a new public class method `Bk2Kombi::AdvanceMatchState.initialize_bk2_state!(table_monitor)`, and call it from all THREE reflex paths that reach `finish_shootout!`:

1. `start_game` — shootout-button click for Spieler A (mode passed via dataset)
2. `switch_players_and_start_game` — shootout-button click for Spieler B (mode passed via dataset)
3. `key_d` — keyboard equivalent (`d` key) while the shootout modal is open — NO mode payload, so the init relies on a previously-written `bk2_options["first_set_mode"]` OR falls back to `DEFAULT_FIRST_SET_MODE = "direkter_zweikampf"` via `derive_first_set_mode`'s whitelist

In the button paths, the init runs **after** the Plan 38.3-05 `bk2_options["first_set_mode"]` write block (so `derive_first_set_mode` picks up the just-written mode) and **before** `finish_shootout!` / `do_play` (so the playing state broadcast sees a populated bk2_state). In `key_d`, the init runs **before** `finish_shootout!` inside the existing `shootout_modal_should_be_open?` branch — the default fallback is acceptable because the dry-run scenario for this key is "accept whatever mode is currently configured". No ordering bug risk: Ruby is single-threaded per request, the in-memory `@table_monitor.data` hash already carries the mode by the time init runs.

Why not inline the init in the reflex: `init_state_if_missing!` contains the canonical schema (8 keys, 3-set scaffold, derive_* helpers) and is the ONLY source of truth for bk2_state shape. Duplicating that into the reflex creates drift risk. Exposing it publicly costs one new 3-line wrapper method.

Unblocks all 4 items in `38.3-HUMAN-UAT.md` (1, 2, 3 blocked directly by I6; 4 is build-artifact cleanup, unaffected).

**Not in scope** (Phase 38.4): I1 (discipline-list restructuring), I2 ("Direkter Zweikampf" → "BK-2plus" rename), I3/I4 (conditional detail-view inputs), I5 (shootout layout reorder), I7 (opponent-credit mode-scoping). This plan ONLY fixes the state-initialization gap.
