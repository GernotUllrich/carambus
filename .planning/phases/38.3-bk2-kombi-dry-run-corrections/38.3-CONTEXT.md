through a simplification pass:

   38.2-03's `_show_bk2_kombi.html.erb` and `_scoreboard_bk2_kombi.html.erb` are

3. **Move the first_set_mode 4-button matrix from the detail-view to the shootout
   screen**, where it replaces the Karambol-generic "Spieler A/B stößt an". The
   detail-view BK2 branch is repurposed to configure Ballziel (50/60/70) + DZ-max-shots
   + SP-max-innings.

4. **Fix the dead "Stoß erfassen" Reflex round-trip (DR-05)** by deleting it — the new
   entry paradigm makes it irrelevant.

5. **Validate rule-interpretation defaults (DR-02)** with Landessportwart + BK2 Excel
   tool. No code change required unless the Excel-tool behavior reveals a missing
   `bk2_options` field — Plan 38.2-01's schema is already configurable.

Out of scope:
- Changes to Plan 38.2-01's service/config/i18n foundations (`AdvanceMatchState`

- Full Variante A (event-based Schreiber+Schiri mode) — explicitly deferred. Not

<decisions>
## Implementation Decisions

### Overall Strategy

- **D-01:** **Karambol-branch approach over parallel partials.** Phase 38.2 Plan 03
  created `_show_bk2_kombi.html.erb` and `_scoreboard_bk2_kombi.html.erb` as 1:1
  structural copies. 38.3 **reverts** this: both partials are deleted, and BK2 is
  expressed as conditional branches inside `_show.html.erb`, `_scoreboard.html.erb`,
  and `_player_score_panel.html.erb` — the same pattern Pool / Snooker / Eurokegel
  use alongside Karambol. Rationale: with Variante B the BK2 deltas reduce to three
  narrow concerns (negative values, hidden GD/HS, set-phase alternation) — separate
  partials would be pure duplication.

- **D-02:** **Variante B (point-entry) is the only live-path input method.** The
  event-based Stimulus controller + 3-row form from Plan 38.2-04 is deleted.
  `Bk2Kombi::ScoreShot` is no longer reached from the live reflex; its code remains
  for a potential later Variante A revival but has no live callers.

- **D-03:** **Scope-softening of Phase 38.2 D-02 is explicit and intentional.**
  Phase 38.2 CONTEXT.md D-02 said "ScoreShot / BK2_DISCIPLINE_MAP / carambus.yml /
  start_game / reflex stay as shipped in 38.1 Plan 06 — only augmentations allowed".
  38.3 softens the "reflex stays" clause: `TableMonitorReflex#bk2_kombi_submit_shot`
  becomes unused and is deleted. Discipline / carambus.yml / start_game remain
  protected.

### Scoreboard Visual Layout

- **D-04:** **Top-center (outside player-card frames):** discipline + set-win
  counter next to each player name, Karambol-identical (e.g., "Christian 0 …
  Gernot 1" with the digit in a rounded box). This slot already comes for free from
  the Karambol branch.

- **D-05:** **Player-card frame** (green for active, gray for inactive):
  encompasses score + logo + Ziel block + inning-score indicator. Negative values
  allowed: both large set-score and small inning-score may render with a minus
  sign in their natural position. No new "negative-score" widget — the existing
  number slot simply tolerates `-` prefix.

- **D-06:** **Large score** (center of player card): current **set score**, may be
  negative (red/gray minus prefix). Updated live during the in-progress Aufnahme
  as point-entry buttons click (preview mode — value before Aufnahmewechsel-Commit).

- **D-07:** **Small red inning-score** (top-left of player card, Karambol-identical
  position): current Aufnahme running total. May be negative (displayed as e.g.
  `-3`, still in red). On Aufnahmewechsel (commit), this value is applied to the
  set score per the phase rules (D-11, D-12) and resets to 0 for the new player.

- **D-08:** **Ziel slot** (top-right of player card, Karambol-identical position):
  shows `Ziel: N` where N is `bk2_state['set_target_points']` (50 / 60 / 70). **GD
  and HS rows are suppressed for BK2-Kombi** via a `unless @free_game_form ==
  "bk2_kombi"` branch in `_player_score_panel.html.erb`.

- **D-09:** **Center strip:**
  - **Big "Satz 1 / 3" header is REMOVED.** Set progress is already visible in the
    sets_won counter next to the player names (D-04).
  - **Phase chip stays** — colored chip showing "Direkter Zweikampf" or
    "Serienspiel" for the current set; drives the remaining-badge wording (D-10).
  - Discipline name + table name + clock remain Karambol-identical.

- **D-10:** **Phase-sensitive remaining badge** (same slot as Karambol's inning
  counter "2 of 20"):
  - DZ set: `Stöße übrig: N` (from `bk2_state["shots_left_in_turn"]`, reset per
    Aufnahmewechsel from `bk2_options["direkter_zweikampf_max_shots_per_turn"]`)
  - SP set: `Aufnahmen übrig: M` (from `bk2_state["innings_left_in_set"]`, reset
    per set-start from `bk2_options["serienspiel_max_innings_per_set"]`)
  - **Note:** In Variante B the DZ shot-counter meaning shifts from
    "individual shots left" to "inning-commits-until-turn-ends" if the
    rule-interpretation (DR-02) maps max-shots to max-commits. To be confirmed
    with Landessportwart. Default Variante B assumption: **DZ max-shots counts
    committed Aufnahmen within a turn, SP max-innings counts completed Aufnahmen
    within a set.**

- **D-11:** **DZ scoring — Aufnahmewechsel with negative inning total:**
  The absolute value of the negative total is credited **positively** to the
  opponent. Commit trigger = swap-icon click OR opponent-panel click (D-14).
  - Player A closes his Aufnahme with `-3` → Spieler B's set-score increases by
    `+3`; Spieler A's set-score stays at the pre-Aufnahme value.
  - Player A closes his Aufnahme with `+7` → Spieler A's set-score increases by
    `+7`; Spieler B unchanged.

- **D-12:** **SP scoring — Aufnahme added with sign:**
  The inning total is added to the own set-score, **including negative values**
  (set-score can decrease).
  - Player A closes his Aufnahme with `+12` → his set-score += 12.
  - Player A closes his Aufnahme with `-4` → his set-score += -4 (i.e. decreases
    by 4, may go negative).
  - Opponent unaffected.

### Shot-Entry Bar (Variante B)

- **D-13:** **Karambol-identical bottom bar layout.** Button row:
  `[clipboard] -1 -5 -10 [swap-icon] +10 +5 +1 [calculator]`.
  All buttons preserve their Karambol `data-*` wiring; only server-side
  interpretation adapts (see D-15).

- **D-14:** **Commit triggers — Karambol-identical:**
  - **Click on swap-icon in the middle of the bottom bar** → commit inning +
    switch active player
  - **Click on opponent's player-card** → same (Karambol already supports this;
    inherit verbatim)
  - No separate "Aufnahme fertig" button. No timeouts. User gesture only.

- **D-15:** **Number-popup (Rechner-Icon):** Reuse the Karambol popup **as-is for
  positive values**. For **negative Aufnahme totals**, the Karambol popup does not
  directly support minus sign — the ±N buttons cover those paths. Alternative
  acceptable: add a minus-toggle to the existing popup (small UI delta). Decision
  deferred to planner — either path preserves the live UX.

- **D-16:** **Mittelpin / echter Karambol / unechter Karambol / Bande flags —
  NOT tracked in Variante B.** These are purely Protokoll-relevant and do not
  influence the live score. Retrofit optional if Variante A is revived later.

### Start-Game Flow (Detail-View + Shootout)

- **D-17:** **Detail-view BK2 config fields** (replaces the 4 first_set_mode
  buttons from Plan 38.2-02). Inside the `bk2_selected` branch of
  `app/views/locations/scoreboard_free_game_karambol_new.html.erb`:
  - **Ballziel:** radio or dropdown with options 50 / 60 / 70. Writes to hidden
    `bk2_options[set_target_points]`.
  - **DZ max Stösse pro Aufnahme:** number input, default 2. Writes to
    `bk2_options[direkter_zweikampf_max_shots_per_turn]`.
  - **SP max Aufnahmen pro Satz:** number input, default 5 (or 5+1 flag pending
    DR-02). Writes to `bk2_options[serienspiel_max_innings_per_set]`.
  - **Removed:** the 4 first_set_mode buttons (Plan 38.2-02 payload). Remove the
    related Alpine state slots (`bk2_first_set_mode`) from hidden inputs.
  - **Preserved:** Alpine-scope lift from Plan 38.2-02 (GAP-02 fix) — do NOT
    regress.

- **D-18:** **Shootout-screen branch in `_shootout.html.erb`** — when
  `show_data["free_game_form"] == "bk2_kombi"`, render 4 buttons instead of 2:
  - `Spieler A stösst an — Direkter Zweikampf` → `first_break_choice=1`,
    `bk2_first_set_mode=direkter_zweikampf`
  - `Spieler A stösst an — Serienspiel` → `first_break_choice=1`,
    `bk2_first_set_mode=serienspiel`
  - Analog for Spieler B (2 more).
  - Submits the Karambol shootout-to-playing transition with the BK2-specific
    payload included.
  - **Satz 3** (played only at 1:1) uses the **same mode as Satz 1** (Plan 38.2-01
    alternation logic handles this).

### Set / Match Transitions

- **D-19:** **Set-end + match-end use Karambol-identical AASM transitions:**
  `playing → set_over → final_set_score → ready_for_new_match` (or
  `final_match_score` at 2:0 / 2:1). These states already exist on TableMonitor;
  Plan 38.2-01's `Bk2Kombi::AdvanceMatchState` already wires set-close and
  match-close. Phase alternation for the next set (DZ ↔ SP) is driven by the
  existing `first_set_mode` + `phase_for_set(set_number)` logic from Plan 38.2-01.

- **D-20:** **Responsive fit:** BK2-Kombi scoreboard must fit without scroll on
  1024×768 landscape tablets (BCW club-table target). Same constraint as Karambol;
  if Karambol passes, the branched BK2 variant should pass too — but this is an
  explicit acceptance criterion, not an implicit assumption.

### Service Layer

- **D-21:** **`Bk2Kombi::CommitInning`** — new service:
  - Signature: `call(table_monitor:, player:, inning_total:)` where `player` is
    `"playera"` | `"playerb"` and `inning_total` is a signed integer (may be
    negative).
  - Reads `bk2_state["current_phase"]` to decide DZ (D-11) vs SP (D-12) scoring.
  - Applies the delta, increments the Aufnahme counter, decrements `shots_left`
    or `innings_left` as appropriate, closes the set if target reached
    (delegates to existing `AdvanceMatchState#close_set_if_reached!` logic —
    keep that code path reused, don't re-implement).
  - Returns `{state:, transitions:}` — same shape as `AdvanceMatchState#call` so
    CableReady broadcasts remain compatible.
  - Idempotency guard: optional `shot_sequence_number` parameter, mirrored from
    the existing guard in `AdvanceMatchState`.

- **D-22:** **`Bk2Kombi::ScoreShot`** — no code changes. No live callers after
  this phase. Left in repo as dormant code.

- **D-23:** **`TableMonitorReflex#bk2_kombi_submit_shot`** — **deleted.** The
  new point-entry path calls an existing Karambol-style reflex (adapted to route
  through `Bk2Kombi::CommitInning` when `free_game_form == "bk2_kombi"`).

- Exact DOM name for the branch hook in `_shootout.html.erb` / `_scoreboard.html.erb`

- Plan 38.2-01 service foundations are the **single source of truth** for BK2 state
  structure. `bk2_state` schema, `first_set_mode` alternation,
  `bk2_state_uninitialized?` predicate — all consume-as-is.

  the same transitions.
- `_player_score_panel.html.erb` — per-player card is factored out; add a single
  `unless bk2?` branch around the GD/HS rows, allow negative rendering in the
  score slot.

### Established Patterns

- **Pool / Snooker pattern** = parallel location-level scoreboard file +
  dispatcher branch. Phase 38.2 Plan 03 accidentally followed this pattern;
  Phase 38.3 **rejects it** for BK2 because the delta is too small.
- **Config-driven branching in views:** `_quick_game_buttons.html.erb` branches
  on `is_pool` / `is_snooker` via carambus.yml-driven flags — this is the
  conceptual reference for how to branch `_scoreboard.html.erb` on BK2.
- **Alpine x-data scope:** the Plan 38.2-02 scope-lift fix must be preserved when
  restructuring the detail-view — hidden inputs that depend on Alpine state must
  stay inside the scope.

### Integration Points

- **DB (JSONB):** `TableMonitor#data["bk2_state"]` shape is set by Plan 38.2-01
  and must stay backward-compatible. New fields allowed; no field removals.
- **Reflex:** the new point-entry path routes through
  `TableMonitorReflex`'s existing Karambol score-update action (adapted with a
  BK2 branch), not a dedicated BK2 reflex. This is cleaner than the Plan 38.2
  `bk2_kombi_submit_shot` which is being retired.
- **CableReady broadcast:** unchanged — `tm.save!` in after_commit drives the
  same morph, just with updated BK2 state.
