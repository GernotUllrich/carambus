### I1 — Detail-View Disziplin-Liste: alle BK-Varianten als eigene Disziplinen listen, Ballziele separat als Punktziel wählen

**Scope:** Deferred to Phase 38.4.

**Ist:**
- Die Detail-View listet BK2-Ballziel-Varianten (BK2 50, BK2 60, BK2 70) getrennt als eigene Disziplinen.
- BK-2, BK-2plus erscheinen nicht konsistent mit Ballziel-Auswahl.
- Punktziel-Reihe und Disziplin-Liste sind nicht sauber getrennt.

**Soll:**
- Disziplin-Liste enthält: **BK50, BK100, BK-2, BK-2plus, BK-2kombi** (keine Ballziel-Varianten innerhalb der Disziplin-Liste).
- **Punktziel-Reihe** wählt das Ballziel:
  - BK50: automatisch 50 (nur lesen / disabled)
  - BK100: automatisch 100 (nur lesen / disabled)
  - BK-2, BK-2plus: wählbar aus {50, 60, 70, 80, 90, 100}
  - BK-2kombi: wählbar aus {50, 60, 70} (gilt für beide Sätze)
- Bei BK-2kombi: **Mehrsatzspiel automatisch aktiviert**, Sätze unbegrenzt (`*`), Winner (when) = 2.

**Root cause:** Plan 38.1 hat BK-Disziplinen mit eingebauten Ballzielen als separate Records modelliert. Die Detail-View zeigt diese 1:1 an.

**Scope-Erweiterung:** Affects all BK-\* disciplines, not only BK-2kombi. Requires rethinking how disciplines + point-targets are represented in the detail view.

### I6 — Blocker: Shootout → Playing-Transition führt zu GAP-05 Fallback-Banner

**Screenshot:** `/Users/gullrich/Desktop/Bildschirmfoto 2026-04-24 um 18.37.23.png`

**Status:** ✓ **CLOSED** by Plan 38.3-08 (2026-04-24). Manual UAT 2026-04-24T22:00 confirmed: "Start nach shootout landet jetzt im scoreboard".

**Scope:** **Fix in 38.3 gap closure** (blocker — ohne Fix kann keine BK-2kombi-Partie gestartet/getestet werden).

**URL (approximiert):** `http://0.0.0.0:3007/locations/<md5>?sb_state=table_monitor&table_id=<N>` nach Klick auf einen der 4 Shootout-Buttons.

**Ist:**
- Klick auf einen der 4 Shootout-Buttons ("Spieler A stößt an — Direkter Zweikampf" etc.) löst die `start_game` / `switch_players_and_start_game` Reflex-Aktion aus.
- Plan 38.3-05 schreibt `@table_monitor.data["bk2_options"]["first_set_mode"]` aus dem Dataset-Attribut, dann AASM-Transition shootout → playing.
- Scoreboard rendert — zeigt aber die GAP-05 Fallback-Banner **"Spiel nicht initialisiert — Bitte das Spiel über die Startansicht neu starten."** statt des eigentlichen Scoreboards.
- `bk2_state_uninitialized?` feuert, weil `@table_monitor.data["bk2_state"]` leer/nil ist bei playing-Zustand.

**Root cause hypothesis:**
- Die AASM-Transition shootout → playing (guard `advance_match_state!` oder ähnlich) läuft, aber initialisiert `bk2_state` nicht mit dem per Plan 38.3-05 gesetzten `bk2_options["first_set_mode"]`.
- Vermutlich fehlt im BK-2kombi start_game-Pfad der Call zu `AdvanceMatchState#initialize_bk2_state!` oder einer äquivalenten Initialisierung, die `bk2_state["current_phase"]`, `set_scores`, `innings_left_in_set`, `shots_left_in_turn` setzt.
- Alternativ: `initialize_bk2_state!` wird aufgerufen, liest aber `bk2_options["first_set_mode"]` bevor Plan 38.3-05s Write zum `data`-Hash persistiert wurde (race oder falsche Reihenfolge innerhalb derselben Reflex-Action).

**Soll:**
- Klick auf Shootout-Button → AASM-Transition zu playing → Scoreboard erscheint vollständig initialisiert mit:
  - `bk2_state["current_phase"]` = gewählter `first_set_mode` (`direkter_zweikampf` oder `serienspiel`)
  - `bk2_state["set_scores"]` korrekt initialisiert
  - `bk2_state["innings_left_in_set"]` bzw. `shots_left_in_turn` auf Max-Wert
  - Phasen-Chip + Remaining-Badge + Scores zeigen korrekte Startwerte
- Keine Fallback-Banner.

**Blockiert:** Alle weiteren manuellen Tests (I7-Scoring, Reflex-Round-Trip, Visual-Layout).

### I7 — Negativ-Punkte-Regel: Opponent-Credit nur für BK-2plus, nicht generell

**Scope:** Deferred to Phase 38.4.

**Ist:**
- Plan 38.3-01 (`Bk2Kombi::CommitInning`) implementiert die D-11-Regel: in Direkter-Zweikampf-Phase werden negative Inning-Totals dem Gegner als positive Punkte gutgeschrieben.
- Diese Regel wurde ursprünglich als **DZ-spezifisch** (Direkter Zweikampf) kodiert.
- Nach Rücksprache mit BK-Pro-Spieler: die Regel gilt offiziell nur für **BK-2plus** (bisher "Direkter Zweikampf" genannt, s. I2).

**Soll:**
- **BK-2plus** (und BK-2kombi während BK-2plus-Phase): negative Inning-Totals → Opponent-Credit (Punkte beim Gegner positiv).
- **BK-2, BK50, BK100** (und BK-2kombi während BK-2-Phase): negative Inning-Totals **bleiben beim Spieler selbst akkumuliert** (Spieler-Score kann negativ werden — Minus-Werte werden am Spieler ausgewiesen).
- D-11-Decision muss in Phase 38.4 präzisiert / re-interpretiert werden:
  - Alter Wortlaut: "DZ negatives → opponent credit" (impliziert alle Modi mit DZ-Namen)
  - Neuer Wortlaut: "BK-2plus negatives → opponent credit" (mode-specific, nach I2-Rename)

**Root cause:** Missverständnis der Regel während 38.3 Planning. `Bk2Kombi::CommitInning` behandelt Phase `direkter_zweikampf` einheitlich als opponent-credit — das stimmt für BK-2kombi, aber nicht für BK-2 als Einzelmodus.

**Scope-Erweiterung:**
- `Bk2Kombi::CommitInning` muss differenzieren nach `free_game_form` (Discipline), nicht nur nach `current_phase`.
- Oder: Scoring-Service wird generalisiert (`Bk2::CommitInning`?) und deckt alle BK-Disziplinen ab.
- Depends on I1 (discipline modeling) and I2 (terminology).

### I8 — TableMonitor kann nicht aus Webapp gelöscht werden (id=50000001)

**Scope:** Deferred to Phase 38.4.

**Reported:** Manual UAT 2026-04-24T22:00 (post-38.3-08).

**Ist:**
- User kann `table_monitors/50000001` nicht über die Web-UI löschen.
- ID 50000001 ≥ `MIN_ID` (50_000_000) → ist ein **lokaler** Record, LocalProtector sollte `destroy` nicht blockieren.

**Soll:**
- TableMonitor mit id ≥ MIN_ID muss über Web-UI löschbar sein (lokaler Record, kein Global-Protection greift).

**Root cause hypothesis:**
- UI-Button/Link auf destroy-Action fehlt oder ist nicht sichtbar in aktueller Ansicht.
- Oder: `TableMonitorsController#destroy` wirft einen stillen Redirect/Error (Pundit/CanCanCan policy, missing DELETE route, oder Turbo-confirm-Problem).
- Oder: LocalProtector-Logik ist falsch konfiguriert und blockiert fälschlicherweise auch lokale Records.

**Next step (for 38.4):**
- Reproduktion in dev: `TableMonitor.find(50000001).destroy` im Rails-Console → OK? Dann ist es ein UI/Controller-Problem, nicht Model.
- Check `TableMonitorsController#destroy` (line 364) Flow + route + policy + button-handler.

### I9 — Ballziel (set_target_points) wird in Serienspiel nicht berücksichtigt

**Scope:** Deferred to Phase 38.4.

**Reported:** Manual UAT 2026-04-24T22:00 (post-38.3-08).

**Ist:**
- User hat Ballziel über Detail-View konfiguriert (50/60/70 per Plan 38.3-06).
- In Serienspiel-Modus wird das konfigurierte Ballziel ignoriert — Satz schließt nicht oder schließt bei Default-Wert.

**Soll:**
- `bk2_options[:set_target_points]` aus Detail-View wird persistent gespeichert und in `Bk2Kombi::AdvanceMatchState#derive_set_target` korrekt gelesen.
- Serienspiel-Satz schließt wenn Spieler-Score cumulativ `set_target_points` erreicht.

**Known wiring (needs trace):**
- Form hidden input: `app/views/locations/scoreboard_free_game_karambol_new.html.erb:228` → `<input name='set_target_points' :value="bk2_selected ? bk2_target : ''" type='hidden'/>`
- Controller: `app/controllers/table_monitors_controller.rb:125,143` → `p.delete(:set_target_points).to_i` → `bk2_options["set_target_points"] = set_target`
- Service: `app/services/bk2_kombi/advance_match_state.rb:102` → `(@tm.data.dig("bk2_options", "set_target_points") || DEFAULT_SET_TARGET).to_i`
- Service: `app/services/bk2_kombi/advance_match_state.rb:157` → `target = state["set_target_points"].to_i` in set-close guard.

**Root cause hypothesis:**
- Alpine.js `bk2_target` binding evaluates to empty string when BK2 ball-target radio not actively clicked → `to_i → 0` → `bk2_options["set_target_points"] = 0` persisted.
- Or: form never reaches controller with non-empty `set_target_points` (Alpine conditional).
- Or: `derive_set_target` default `DEFAULT_SET_TARGET` masks the zero and set does close at default (50), so user sees "wrong target" rather than "no target".

**Next step (for 38.4):**
- End-to-end trace: browser DevTools Network tab → confirm POST includes `set_target_points=60` (or 70) when selected.
- DB check: after start-game, `TableMonitor.find(id).data["bk2_options"]["set_target_points"]` → expected 60/70, observed value.
- System test: add assertion that set-close fires at configured target, not at DEFAULT_SET_TARGET.
