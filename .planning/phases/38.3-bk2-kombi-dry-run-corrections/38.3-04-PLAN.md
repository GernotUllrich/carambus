Replace the deprecated Plan 38.2-04 event-based shot-entry bar with a Karambol-parallel point-entry path:

1. **Delete** `TableMonitorReflex#bk2_kombi_submit_shot` and the `bk2_kombi_shot_controller.js` Stimulus controller (D-23).
2. **Add** a new Karambol-identical bottom-bar BK2 branch inside `_scoreboard.html.erb`: same `-1 -5 -10 | swap | +10 +5 +1 | popup` layout, same Stimulus controller wiring, but the buttons accumulate into a client-side running total that's committed to the server via a new `TableMonitorReflex#bk2_commit_inning` reflex action only on swap-icon click or opponent-panel click (D-14).
3. **Add** the new reflex action `bk2_commit_inning` which reads `player_at_table` from server-side `bk2_state` (security — never trust client), reads the accumulated `n` from `element.dataset[:n]` (client-supplied delta, coerced + validated by CommitInning), then delegates to `Bk2Kombi::CommitInning.call(...)`.

Critical design choice — client-side accumulation model:
- The `+1/+5/+10/-1/-5/-10` buttons in Karambol currently each fire their own reflex round-trip (`add_n`/`minus_n`). For BK2 we want the SAME user gesture but with inning-level accumulation: each button press increments a client-side counter displayed in the small red inning slot (already a `<span class="inning-score">` in the merged player panel from Plan 38.3-03), and NO server call happens until Aufnahmewechsel (swap icon or opponent-panel tap).
- However to minimize Stimulus surface area + keep Karambol parity, we DELEGATE this to the server: each ±N click fires an existing `add_n`/`minus_n` reflex which calls `add_n_balls` on TableMonitor (adding to `innings_redo_list[-1]` — exactly what Karambol does). On `next_step` (swap-icon) or `key_a`/`key_b` (opponent-panel) the current inning total is read, then delegated to `Bk2Kombi::CommitInning`.

Two implementation routes — choose based on reading existing code:

**Route A (hijack existing next_step/key_a/key_b reflexes):** In `TableMonitorReflex#next_step` / `#key_a` / `#key_b`, add an early branch: if `@table_monitor.data["free_game_form"] == "bk2_kombi"`, extract the current inning total from `@table_monitor.data["playera"]["innings_redo_list"][-1]` (or playerb), delegate to `Bk2Kombi::CommitInning.call(table_monitor:, player:, inning_total:)`, then return without calling the Karambol `next_step` path. Zero new reflex actions; no new Stimulus controller code.

**Route B (new reflex action):** Add a new reflex `bk2_commit_inning` called from a new `@click` handler on the swap icon and opponent panels when `is_bk2`. More explicit, but duplicates wiring.

Plan mandates **Route A** — minimizes diff, reuses existing Karambol Stimulus + reflex scaffolding (add_n / minus_n / next_step / key_a / key_b), zero new JS code.

Output: reflex + view modifications + 2 file deletions.

    # live match is BK2-Kombi and the user triggered an Aufnahmewechsel gesture
    # (swap-icon via next_step, OR opponent-panel via key_a/key_b). Returns true if

    def bk2_kombi_commit_if_active(gesture_id:)
      return false unless @table_monitor.data["free_game_form"] == "bk2_kombi"

      bk2_state = @table_monitor.data["bk2_state"] || {}
      player = bk2_state["player_at_table"].to_s

      inning_total = @table_monitor.data.dig(player, "innings_redo_list").andand[-1].to_i

      Rails.logger.info "[bk2_commit_inning gesture=#{gesture_id}] player=#{player} n=#{inning_total}" if DEBUG

      Bk2Kombi::CommitInning.call(
        table_monitor: @table_monitor,
        player: player,
        inning_total: inning_total,
        shot_sequence_number: SecureRandom.uuid

      @table_monitor.data[player]["innings_redo_list"] ||= []
      @table_monitor.data[player]["innings_redo_list"][-1] = 0

    end
