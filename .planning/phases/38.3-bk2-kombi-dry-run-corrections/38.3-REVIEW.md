# Phase 38.3: Code Review Report

**Reviewed:** 2026-04-24
**Depth:** standard
**Files Reviewed:** 15
**Status:** issues_found

## Summary

This is a phase-wide re-review of the BK2-Kombi dry-run corrections phase, covering plans
38.3-01 through 38.3-08 including the newly added `Bk2Kombi::AdvanceMatchState.initialize_bk2_state!`
public entry point (Plan 38.3-08 / I6) and the three reflex call sites wired into the
shootout → playing transition (`start_game`, `switch_players_and_start_game`, and the
keyboard `key_d` path).

Overall the implementation is well-structured and carefully documented with inline Plan
references. The CLAMP posture (T-38.1-06-01) applied to all user-supplied BK2 parameters is
consistent between the detail-form, quick-form, and new `bk2_options` fields. The public
`initialize_bk2_state!` entry point properly delegates to the existing private guard and is
idempotent. Tests cover both the positive and negative (I6) paths for the new init entry
point, including a direct `bk2_options = {}` test for the `key_d` keyboard path (T4).

Key concerns flagged:
- **Pattern A private-method delegation via `send`** — `CommitInning` reaches into
  `AdvanceMatchState`'s private `close_set_if_reached!`/`close_match_if_reached!`/
  `init_state_if_missing!` APIs. Works today but couples the two classes through their
  private contracts (WR-01, WR-02).
- **Duplicated `four_ball` guard block** in `TableMonitorsController#start_game` with
  subtly different side effects between the two occurrences (WR-03).
- **`bk2_kombi_commit_if_active` rescues `ArgumentError` broadly** and performs
  post-commit mutation of `innings_redo_list` outside any transactional guarantee (WR-04).
- **Repetitive BK2 init block** across three reflex methods (WR-05).
- Several info-level nits around log level, comment drift, and minor redundancies.

No critical security vulnerabilities or data-loss bugs found. The CLAMP / whitelist /
server-side-truth posture for BK2 parameters is intact.

## Warnings

### WR-01: `CommitInning` couples to `AdvanceMatchState` private API via `send`

**File:** `app/services/bk2_kombi/commit_inning.rb:49-54, 83-87`
**Issue:** `CommitInning#call` and `ensure_initialized!` both instantiate `AdvanceMatchState`
purely to call its private methods via `send(:close_set_if_reached!, state)`,
`send(:close_match_if_reached!, state)`, and `send(:init_state_if_missing!)`. The comment
explicitly labels this "Pattern A delegation via send" — acknowledged coupling — but it
creates a fragile contract: a refactor that renames or changes the signature of those
private methods in `AdvanceMatchState` will silently break `CommitInning` with no
compile-time warning, because `send` bypasses visibility checks. Tests cover happy paths
but a private-method rename is a realistic refactoring accident.

**Fix:** Extract the three shared primitives into a separate, explicitly-public module
(e.g. `Bk2Kombi::StateTransitions` with `close_set_if_reached!`, `close_match_if_reached!`,
`init_state_if_missing!` as `module_function` methods), then include/call from both
services. Makes the API contract explicit and keeps the current delegation semantics.
```ruby
# app/services/bk2_kombi/state_transitions.rb
module Bk2Kombi::StateTransitions
  module_function
  def close_set_if_reached!(state, tm)
    # ...extracted body...
  end
  # etc.
end
```
Alternatively, make the three methods `public` in `AdvanceMatchState` and call them
directly (still delegates, but stops laundering through `send`).

### WR-02: `AdvanceMatchState.initialize_bk2_state!` uses `send` to call private guard

**File:** `app/services/bk2_kombi/advance_match_state.rb:37-41`
**Issue:** The new public entry-point calls `send(:init_state_if_missing!)` on itself — a
minor code smell because within the same class there is no need for `send`; private calls
with an implicit receiver are already allowed. Using `send` here masks a small opportunity
to make the method protected (accessible from the same class family) and call it cleanly.

**Fix:** Either mark `init_state_if_missing!` as protected and call it directly, or bump
its visibility with a descriptive reason in a comment. Example:
```ruby
def self.initialize_bk2_state!(table_monitor)
  new(table_monitor: table_monitor, shot_payload: {}).__send__(:init_state_if_missing!)
  table_monitor.save!
  table_monitor.data["bk2_state"]
end
```
Or pair with WR-01's fix — move to `Bk2Kombi::StateTransitions.ensure_initialized!(tm)`
and have this class method become a thin wrapper.

### WR-03: Duplicated `four_ball` guard block in `start_game` controller action

**File:** `app/controllers/table_monitors_controller.rb:167-173` and `305-315`
**Issue:** The `four_ball` guard runs twice — once inside the `unless p[:quick_game_form]`
branch at line 167, then again unconditionally at line 305 (this time also setting
`color_remains_with_set`, `allow_overflow`, `timeouts`, and `timeout`). The two blocks are
not identical — the first lacks the timer overrides. For a POST with `free_game_form=pool`
+ `four_ball=1` the first block is skipped (it's inside the `unless quick_game_form`
branch so runs only for detail-forms) but the second block fires and overwrites the
pool-specific params. The comment at line 304 cross-references the "first occurrence" but
the two are not semantically equivalent.

**Fix:** Consolidate into a single block at the end of the method, after all form-specific
processing. Remove the first occurrence (167-173) and keep the more complete second one:
```ruby
# AFTER all form branches:
if p[:four_ball].present? && p[:quick_game_form] != "bk2_kombi" && p[:free_game_form] != "bk2_kombi"
  p[:discipline_a] = p[:discipline_b] = "4 Ball"
  p[:balls_goal_a] = p[:balls_goal_b] = 120
  p[:innings_goal]  = 0
  p[:sets_to_play]  = 1
  p[:sets_to_win]   = 1
  p[:color_remains_with_set] = true
  p[:allow_overflow]         = false
  p[:timeouts] = 0
  p[:timeout]  = 0
end
```

### WR-04: `bk2_kombi_commit_if_active` swallows `ArgumentError` and mutates state outside any transaction

**File:** `app/reflexes/table_monitor_reflex.rb:968-1001`
**Issue:** The helper reads `inning_total` from `@table_monitor.data.dig(player, "innings_redo_list").andand[-1].to_i` (line 982),
then invokes `Bk2Kombi::CommitInning.call(...)`, then unconditionally mutates
`@table_monitor.data[player]["innings_redo_list"]` at lines 994-996 (reset transient Karambol
running total). The `rescue ArgumentError` at line 998 catches `CommitInning.validate!`
failures and returns `false` — but by then any successful call will have already saved
(via `CommitInning`'s `tm.save!`). The caller then does `@table_monitor.save!` (line 125 /
189) on top of the just-mutated `innings_redo_list`, which is fine in the happy path but
means the "reset the running total" step is NOT atomic with the commit: a crash between
`CommitInning.call` and `@table_monitor.save!` would leave the JSONB bk2_state advanced but
the `innings_redo_list` stale, which on next commit would re-apply the same points.
Additionally, the `.to_i` on `andand[-1]` silently coerces `nil` (empty list) to `0`,
masking "no inning to commit" cases.

**Fix:** (a) Guard on list emptiness BEFORE calling `CommitInning`:
```ruby
redo_list = @table_monitor.data.dig(player, "innings_redo_list")
if redo_list.blank?
  Rails.logger.warn "[bk2_commit_inning gesture=#{gesture_id}] no inning to commit for #{player}"
  return false
end
inning_total = redo_list.last.to_i
```
(b) Move the `innings_redo_list` reset INTO `CommitInning.call` so it's atomic with the
bk2_state mutation (both persisted together via the single `tm.save!` inside the service).
(c) Log `inning_total` with its source so a nonsensical reset of a non-integer value can
be diagnosed.

### WR-05: Three-way code duplication in reflex BK2 init blocks

**File:** `app/reflexes/table_monitor_reflex.rb:283-290, 361-375, 394-408`
**Issue:** The BK2 init block — "if this is BK2, persist `bk2_first_set_mode` from the
element dataset, then init state" — appears in nearly-identical form across `key_d`,
`switch_players_and_start_game`, and `start_game`. The `key_d` variant is slightly different
(no dataset read, relies on the `DEFAULT_FIRST_SET_MODE` fallback). This is a DRY violation
that will bite the next time the init flow needs to change (e.g., a new `bk2_options`
field) — one of the three sites could get missed.

**Fix:** Extract a private helper:
```ruby
private

def bk2_kombi_init_from_shootout!(write_mode_from_dataset: true)
  return unless @table_monitor.data["free_game_form"] == "bk2_kombi"

  if write_mode_from_dataset
    bk2_mode = element.andand.dataset[:bk2_first_set_mode].to_s
    if %w[direkter_zweikampf serienspiel].include?(bk2_mode)
      @table_monitor.data["bk2_options"] ||= {}
      @table_monitor.data["bk2_options"]["first_set_mode"] = bk2_mode
    end
  end
  Bk2Kombi::AdvanceMatchState.initialize_bk2_state!(@table_monitor)
end
```
Then replace each of the three inline blocks. `key_d` calls
`bk2_kombi_init_from_shootout!(write_mode_from_dataset: false)`; the other two call with
default `true`. Also consolidates the three slightly divergent Plan-reference comments
(see IN-04).

## Info

### IN-01: `DEBUG = true` class constant with no config override

**File:** `app/reflexes/table_monitor_reflex.rb:25`
**Issue:** `DEBUG = true` is hard-coded; changing log verbosity requires a code deploy.
Many `Rails.logger.info` calls gated on `DEBUG` are high-frequency (one per key press).
**Fix:** Read from `Rails.application.config.x.table_monitor_reflex_debug` with a false
default, or use `Rails.logger.debug { "..." }` blocks (evaluated lazily when the log level
is debug).

### IN-02: Unicode emoji in log messages

**File:** `app/reflexes/table_monitor_reflex.rb:41, 336, 430, 475, 586, 621` (and others)
**Issue:** Log messages contain emojis (🔔, 🚫). Harmless but inconsistent with the rest
of the codebase and may cause encoding issues in older syslog pipelines.
**Fix:** Replace with plain ASCII prefixes, e.g. `"[BLOCKED]"`, `"[SKIPPED]"`.

### IN-03: `close_set_if_reached!` depends on shared `@tm` instance for `derive_first_set_mode` fallback

**File:** `app/services/bk2_kombi/advance_match_state.rb:172` (called from
`app/services/bk2_kombi/commit_inning.rb:49-54`)
**Issue:** `close_set_if_reached!` uses `state["first_set_mode"].presence || derive_first_set_mode`
as fallback. `derive_first_set_mode` reads from `@tm.data.dig("bk2_options", ...)`. When
`CommitInning` calls `close_set_if_reached!` on a fresh `AdvanceMatchState.new`, that `@tm`
is the same TableMonitor (passed in at line 49-52 of `commit_inning.rb`), so it reads the
correct options. But this coupling is subtle — future refactors could accidentally pass a
different `@tm` or a stale copy.
**Fix:** Add a one-line comment at `commit_inning.rb:49-52` noting that the `advance_helper`
instance shares the same `@tm` reference so `derive_first_set_mode` reads the correct
`bk2_options`. Or accept an explicit `first_set_mode:` keyword to make the dependency
explicit.

### IN-05: Broad `rescue StandardError` inconsistently re-raises

**File:** `app/reflexes/table_monitor_reflex.rb:349-352, 444-447, 754-756, 782-784`
**Issue:** `rescue StandardError => e` re-raises in `add_n`/`minus_n` but swallows silently
in `up`/`down` (just logs with `Rails.logger.info`). Inconsistent failure handling is hard
to reason about when debugging. Per the CLAUDE.md conventions, broad rescues are
acknowledged as common in this codebase — this is informational only.
**Fix:** Make `up`/`down` follow the pattern in `warmup_state_change` at line 1042:
`raise StandardError unless Rails.env == "production"`. At minimum, log with `.error` not
`.info` so failures are visible in log-level production configurations.

### IN-06: Double-`andand` on a guaranteed-non-nil receiver

**File:** `app/controllers/table_monitors_controller.rb:35`
**Issue:** `@table_monitor.andand.table.andand.id` — the `@table_monitor` is guaranteed
non-nil at this point (the `unless @table_monitor` guard in `set_table_monitor` would have
redirected already). The first `andand` is redundant; minor readability nit.
**Fix:** Simplify to `@table_monitor.table.andand.id`.

### IN-07: Test fixture `free` has `state: "playing"` but empty `data`

**File:** `test/fixtures/table_monitors.yml:19-28`
**Issue:** Fixture `free` has `state: "playing"` and `data: '{}'`. Many tests update `data`
before exercising the service, but a `playing`-state TableMonitor with no bk2_state is
inconsistent with production invariants — normally `playing` implies a populated
bk2_state. Test-fixture smell, not a production bug.
**Fix:** Either flip to `state: "new"` for `free` (tests that need `playing` can
`update_columns(state: "playing")`) or add a minimal placeholder `bk2_state` seed. The
latter risks tests passing for the wrong reason, so the former is preferred.

### IN-09: Misleading "dormant code" comment for `ScoreShot`

**File:** `app/services/bk2_kombi/commit_inning.rb:5-7` (doc) and
`app/services/bk2_kombi/advance_match_state.rb:54-57` (live call)
**Issue:** `CommitInning`'s docstring says "Ersetzt den event-basierten live-Pfad
(Bk2Kombi::ScoreShot + AdvanceMatchState). ScoreShot bleibt als dormant code für eine
spätere Variante-A-Wiederbelebung". But `AdvanceMatchState#call` still actively invokes
`ScoreShot.call` at line 54. `ScoreShot` is therefore not dormant — it's still the code
path for `AdvanceMatchState.call`. Only the live event-based Stimulus flow has been
replaced; `initialize_bk2_state!` (the new 38.3-08 entry) correctly bypasses ScoreShot
because it calls `init_state_if_missing!` directly without going through `call`.
**Fix:** Clarify the comment: `"AdvanceMatchState.call remains live as the single-shot
path used by auto-pin-counter (Variante A); only the live event-based Stimulus reflex
flow has been replaced by Variante B CommitInning."`
