# Phase 38.3: Code Review Report

**Reviewed:** 2026-04-24
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Phase 38.3 refactored the BK2-Kombi scoreboard back into shared Karambol partials, replaced the event-based shot-entry form with a point-entry reflex (CommitInning), and relocated the first_set_mode decision from the detail form to the shootout screen. The overall structure is sound — the security-sensitive CLAMPs in `start_game` are thorough and the `bk2_kombi_commit_if_active` helper correctly reads inning data from server-side state rather than client-supplied attributes.

One critical bug was found: the Spieler B player-select dropdown in the detail form pre-selects using `player_a.andand.id` (copy-paste error), meaning both dropdowns default to the same player. Five warnings cover a double-save race condition in the BK2 commit path, a missing `data_will_change!` after inline hash mutation in `bk2_kombi_commit_if_active`, an unguarded `locked_scoreboard` bypass in the BK2 early-return path, a client-controlled `bk2_first_set_mode` write without data-dirtying, and a hardcoded production `DEBUG = true` constant that floods logs. Five info items address code quality.

## Critical Issues

### CR-01: Spieler B dropdown defaults to Player A's ID (copy-paste bug)

**File:** `app/views/locations/scoreboard_free_game_karambol_new.html.erb:261`
**Issue:** The `select_tag :player_b_id` uses `player_a.andand.id || default_guest_a.player.id` as its pre-selected value. This is a copy-paste from the Player A widget immediately above. Every time the form loads, both dropdowns default to the same player — Spieler B pre-selects Spieler A's record. Games started without manually changing Spieler B will record both roles to the same player.
**Fix:**
```erb
<%= select_tag :player_b_id,
  options_for_select(Array(players), player_b.andand.id || default_guest_b.player.id),
  class: "py-3 px-4 text-2vw ..." %>
```

## Warnings

### WR-01: Double-save race condition — CommitInning.save! then caller.save!

**File:** `app/reflexes/table_monitor_reflex.rb:122-123` (key_a), `187-188` (key_b), `773-774` (next_step)
**Issue:** `bk2_kombi_commit_if_active` calls `Bk2Kombi::CommitInning.call(...)`, which already performs `@tm.save!` (commit_inning.rb:58). Immediately after, the helper further mutates `@table_monitor.data[player]["innings_redo_list"]` in memory (reflex.rb:974-976) and returns `true`. The caller then does `@table_monitor.save!` a second time. This is two `save!` calls on the same in-memory object for the same logical operation.

The second `save!` writes the post-CommitInning state plus the innings_redo_list reset. Because `data` is a serialized text column (not JSONB), the second save is needed to persist the in-memory innings_redo_list change — but the ordering means there is a window between the two saves where another process could read stale data. Additionally, CommitInning sets `@tm.data["bk2_state"] = state` (line 57) directly on the in-memory hash, and the caller's subsequent innings_redo_list mutation on the same object may not be tracked by ActiveRecord's dirty-tracking for a text-serialized column.

**Fix:** Either move the innings_redo_list reset into CommitInning so there is a single authoritative save, or mark the data dirty explicitly before the second save:
```ruby
# In bk2_kombi_commit_if_active, after CommitInning.call:
@table_monitor.reload  # pull the CommitInning-committed state
@table_monitor.data_will_change!
@table_monitor.data[player]["innings_redo_list"] ||= []
@table_monitor.data[player]["innings_redo_list"][-1] = 0
@table_monitor.data[player]["innings_redo_list"] << 0
# caller's save! then persists this mutation cleanly
```

### WR-02: Missing data_will_change! before inline innings_redo_list mutation

**File:** `app/reflexes/table_monitor_reflex.rb:974-976`
**Issue:** After `CommitInning.call` returns, the helper mutates `@table_monitor.data[player]["innings_redo_list"]` in-place. Because `data` is serialized via `serialize :data, coder: JSON, type: Hash` on a text column, ActiveRecord's dirty-tracking may not detect the nested mutation without an explicit `data_will_change!` call. If the dirty check misses the mutation, the caller's `@table_monitor.save!` writes nothing to the database and the innings_redo_list reset is silently lost — the next inning starts with a non-zero running total.
**Fix:**
```ruby
@table_monitor.data_will_change!
@table_monitor.data[player]["innings_redo_list"] ||= []
@table_monitor.data[player]["innings_redo_list"][-1] = 0
@table_monitor.data[player]["innings_redo_list"] << 0
```

### WR-03: locked_scoreboard guard bypassed for BK2 early-return path

**File:** `app/reflexes/table_monitor_reflex.rb:122-128` (key_a), `187-193` (key_b)
**Issue:** The BK2-Kombi early-return block in `key_a` and `key_b` commits an inning and returns before the `locked_scoreboard` guard is reached:
```ruby
if bk2_kombi_commit_if_active(gesture_id: "key_a", clicked_player: "playera")
  @table_monitor.save!
  return           # ← guard never reached
end
@table_monitor.suppress_broadcast = true
return if @table_monitor.locked_scoreboard
```
If `locked_scoreboard` is intended to freeze all scoreboard mutations (e.g., tournament admin lock), it is silently bypassed for BK2-Kombi games. This may be intentional (BK2 games are free-play only), but there is no comment documenting the exclusion, and the `block_tournament_manipulation` before_action guards only the controller-level `start_game`.
**Fix:** Either add the check before the BK2 commit or document the intentional bypass:
```ruby
return if @table_monitor.locked_scoreboard  # guard BEFORE bk2 check

if bk2_kombi_commit_if_active(gesture_id: "key_a", clicked_player: "playera")
  @table_monitor.save!
  return
end
```

### WR-04: bk2_first_set_mode written from client dataset without data_will_change!

**File:** `app/reflexes/table_monitor_reflex.rb:354-359` (switch_players_and_start_game), `380-385` (start_game)
**Issue:** Both reflex methods write `@table_monitor.data["bk2_options"]["first_set_mode"]` via a nested in-place assignment (`@table_monitor.data["bk2_options"]["first_set_mode"] = bk2_mode`) without calling `data_will_change!`. The write is then persisted by `@table_monitor.save!`. On a text-serialized column this is the same dirty-tracking hazard as WR-02: if AR does not detect the nested mutation, the mode selection is silently discarded and the game launches with the previously stored mode (or no mode).

The whitelist guard (`%w[direkter_zweikampf serienspiel].include?(bk2_mode)`) is correctly applied before the write — this is a persistence reliability issue only.
**Fix:**
```ruby
if %w[direkter_zweikampf serienspiel].include?(bk2_mode)
  @table_monitor.data_will_change!
  @table_monitor.data["bk2_options"] ||= {}
  @table_monitor.data["bk2_options"]["first_set_mode"] = bk2_mode
end
```

### WR-05: Remote-IP whitelist in remote_request? excludes non-192.168.x.x private ranges

**File:** `app/reflexes/table_monitor_reflex.rb:934-939`
**Issue:** The `remote_request?` helper guards `add_n` and `minus_n` against unauthenticated admin operations. It allows only `127.0.0.1`, `::1`, `localhost`, and `192.168.*` as "local". RFC 1918 also defines `10.0.0.0/8` and `172.16.0.0/12` as private ranges; clients connecting from those subnets (e.g., a Raspberry Pi on a `10.x` LAN) are treated as remote and will be blocked unless they are admins. This could break legitimate LAN scoreboards.
**Fix:**
```ruby
def remote_request?
  return false unless request.present?
  remote_ip = request.remote_ip
  local_ranges = ['127.0.0.1', '::1', 'localhost']
  return false if local_ranges.include?(remote_ip)
  return false if remote_ip.start_with?('192.168.', '10.', '172.16.', '172.17.',
                                         '172.18.', '172.19.', '172.2', '172.3')
  true
end
```
Or use Ruby's `IPAddr` for proper range checks.

## Info

### IN-01: DEBUG = true hardcoded constant causes log flooding in all environments

**File:** `app/reflexes/table_monitor_reflex.rb:25`
**Issue:** `DEBUG = true` is a top-level class constant unconditionally set to `true`. Every reflex action (key_a, key_b, nnn_0..9, etc.) emits `Rails.logger.info` banners because the guard `if DEBUG` is always true. This floods production logs and obscures real errors.
**Fix:**
```ruby
DEBUG = Rails.env.development?
```
Or remove the constant and use `Rails.logger.debug { ... }` with block form (already used in the controller).

### IN-02: params.permit(params.keys) anti-pattern in start_game

**File:** `app/controllers/table_monitors_controller.rb:67`
**Issue:** `p = params.permit(params.keys)` permits all top-level params before slicing. This is an unusual pattern — `params.permit(params.keys)` with scalar keys works, but nested hashes (e.g., `bk2_options`) would be unpermitted attributes and raise `ActionController::UnpermittedParameters` in strict mode unless the nested key is explicitly listed. The subsequent `.slice(...)` does provide the effective whitelist (line 68-84), but the intent is obscured and the pattern would fail for any future nested param added to the slice list without also being listed in the initial permit call. The `bk2_options` nested param is correctly read via `params[:bk2_options]` directly (lines 136, 254) rather than through `p`, so this works today — but it is a subtle convention break.
**Fix:** Permit explicitly including the nested hash:
```ruby
p = params.permit(
  :player_a_id, :player_b_id, ...,
  bk2_options: [:direkter_zweikampf_max_shots_per_turn, :serienspiel_max_innings_per_set]
)
```

### IN-03: Duplicate four_ball block

**File:** `app/controllers/table_monitors_controller.rb:162-173` and `305-315`
**Issue:** The `four_ball` branch (sets discipline to "4 Ball", etc.) appears twice: once before the `free_game_form` elsif chain (lines 167-173) and once after it (lines 305-315). The second block has additional fields (`color_remains_with_set`, `allow_overflow`, `timeouts`, `timeout`) that the first block omits. The first block is therefore either redundant or incomplete — if it fires, the second block fires too and overwrites it. The comments claim this is intentional ("same guard as the earlier four_ball branch") but that means the first block at lines 167-173 is dead code because it sets identical fields that are then overwritten again at 305-315.
**Fix:** Remove the first `four_ball` block (lines 162-173) and keep only the complete one at lines 303-315, or consolidate both into one block before the free_game_form chain.

### IN-04: show action accepts 500 as valid in controller test

**File:** `test/controllers/table_monitors_controller_test.rb:44`
**Issue:** `assert_includes [200, 302, 500], response.status` accepts a 500 server error as a valid outcome for the `show` action. This means the test would pass even on an unhandled exception. A 500 should not be an accepted response for any production action.
**Fix:**
```ruby
assert_includes [200, 302], response.status,
  "show should respond with success or redirect"
```

### IN-05: Test coverage gap — bk2_kombi_commit_if_active with locked_scoreboard state

**File:** `test/controllers/table_monitors_controller_test.rb`, `test/system/bk2_kombi_scoreboard_test.rb`

Per WR-03, the locked_scoreboard guard is currently bypassed for BK2 games. If this is intentional, a comment and regression test would prevent future confusion. If it is unintentional, a test would catch the bypass.
**Fix:** Add a unit test in `bk2_kombi_scoreboard_test.rb` or a new reflex unit test that sets `locked_scoreboard` on a BK2 monitor and asserts the correct behavior (either commit is blocked, or the bypass is documented as deliberate).
