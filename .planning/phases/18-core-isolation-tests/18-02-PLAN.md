**Test name:** `test "ISOL-02: score:update dispatch event path isolation — event blocked on wrong scoreboard"`

**Data initialization:** The `score_data` branch in `TableMonitorJob` reads `table_monitor.data["playera"]["innings_redo_list"]`. With `data: {}`, this will be `nil` and raise `NoMethodError`. Before triggering the score_data broadcast, update TM-B's data column:
```ruby
@tm_b.update_columns(
  data: {
    "playera" => { "innings_redo_list" => [5], "result" => 10 },
    "playerb" => { "innings_redo_list" => [3], "result" => 7 }
  }.to_json
)
```

Also ensure TM-B is in "ready" state (the job may need a valid state for `get_options!`):
```ruby
@tm_b.update_columns(state: "ready")
```

**Flow:**
1. `in_session(:scoreboard_a)` — visit TM-A scoreboard, wait for ActionCable connection, then install a JS event listener to detect if score:update for TM-B fires on this session:

window._wrongScoreUpdateReceived = false;
