## Key Confirmations (per plan output spec)

### `_radio_select.html.erb` DOM contract

Confirmed: `_radio_select.html.erb` L6 emits
`<input type="radio" x-model.number="<varname>" name="<varname>_choice" ...>` — **no `data-varname` attribute**. For the BK2 `varname: "bk2_choice"`, the submitted
form field is `bk2_choice_choice` and the Alpine state is `bk2_choice`.
The Task 4 Capybara selector would be `input[name='bk2_choice_choice'][value='…']`.

### Alpine binding approach used

Used **approach (a): computed getters inside `x-data`** — no `x-effect`, no
wrapper divs. The x-data object literal now ends with:

```javascript
,
bk2_choice: null,
get bk2_selected() { return this.bk2_choice !== null && this.bk2_choice !== '' && this.bk2_choice !== undefined; },
get bk2_target() { return [50, 60, 70][this.bk2_choice] || 50; }
```

Getters are standard ES6 and Alpine parses `x-data` as a JavaScript object
literal, so the pattern works as-is.

### T-38.1-06-01 CLAMP implementation

Confirmed: the controller uses **plain assignment (`=`)**, not `||=`, for
BOTH BK2 entry points (detail-form branch and quick-button else-branch):

```ruby
p[:discipline_a] = "BK2-Kombi"
p[:discipline_b] = "BK2-Kombi"
```

Task 3 test `c` (`start_game with free_game_form=bk2_kombi CLAMPS
discipline_a/b to BK2-Kombi even if attacker sends arbitrary values`) proves
the clamp by submitting `discipline_a="Freie Partie klein"` and
`discipline_b="something else"` and asserting both are overwritten.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Strong-params slice dropped set_target_points**

- **Found during:** Task 3 (first test run after adding BK2 branch).
- **Issue:** `params.permit(params.keys).slice(...)` at `table_monitors_controller.rb:67-76` is a strict whitelist. `set_target_points` was NOT in the list, so the controller's BK2 branch always received `nil` for `p[:set_target_points]`, which `.to_i → 0`, which failed the `[50,60,70].include?` check and defaulted to 50 — regardless of user input. Test `b` caught this immediately (expected 60, actual 50).
- **Fix:** Added `:set_target_points` to the slice whitelist with a comment explaining intent. The BK2 branch then receives the raw user value and packs it into `p[:bk2_options]`.
- **Files modified:** `app/controllers/table_monitors_controller.rb` (slice whitelist only).
- **Commit:** 5b680946 (bundled with Task 3).

**2. [Rule 2 - Correctness] Quick-button branch needed BK2 handling too**

- **Found during:** Task 3 test `d` failure.
- **Issue:** The plan placed the BK2 branch only inside the `unless p[:quick_game_form].present?` block. But `quick_game_form=bk2_kombi` skips the unless-block entirely, so no BK2-specific handling fired. The test `d` asserted `bk2_options.set_target_points == 50` even for the quick-button path (the path that `_quick_game_buttons.html.erb` Task 2 emits), which requires the controller to pack `bk2_options` in BOTH branches — otherwise the quick-button UI flow surfaces `bk2_options=nil` to `AdvanceMatchState`, which silently defaults to 50 but the whitelist in `GameSetup` drops the key entirely.
- **Fix:** Added a parallel BK2 handler in the quick_game_form else-branch that mirrors the unless-block logic: CLAMP discipline_a/b to "BK2-Kombi" + pack `p[:bk2_options] = { "set_target_points" => normalized }`. Both branches now produce identical downstream data.
- **Files modified:** `app/controllers/table_monitors_controller.rb` (quick-game else-branch only).
- **Commit:** 5b680946 (bundled with Task 3).
