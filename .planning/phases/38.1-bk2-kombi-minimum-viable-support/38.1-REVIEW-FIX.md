## Fixed Issues

### WR-01: `four_ball` override can bulldoze BK2-Kombi settings silently

**Files modified:** `app/controllers/table_monitors_controller.rb`
**Commit:** `17fa4931`
**Applied fix:** Added the BK2-Kombi short-circuit to BOTH `four_ball` branches
(lines 136-147 for the quick-game path, lines 253-265 for the post-processing
path). Each branch now gates on
`p[:quick_game_form] != "bk2_kombi" && p[:free_game_form] != "bk2_kombi"` in
addition to the existing `p[:four_ball].present?` check. This honours the
CLAMP posture (T-38.1-06-01): a tampered POST carrying both
`quick_game_form=bk2_kombi` AND `four_ball=1` can no longer silently drop
BK2 state.

Both occurrences are guarded (not merely one) because removing the second
duplicate — while tempting — risks changing other behaviour paths (e.g. the
Shootout block between them can mutate `p[:discipline_a/b]`, and the second
block also sets `color_remains_with_set`, `allow_overflow`, and clears the
timeout). A comment cross-references the first occurrence for maintenance.

### WR-02: System-test `find_or_create_by!` + teardown `.destroy` can delete a pre-existing Game

**Files modified:** `test/system/bk2_kombi_scoreboard_test.rb`
**Commit:** `c935f4cd`
**Applied fix:** Captured `@game_created_by_test = !Game.exists?(id: 50_000_200)`
BEFORE the `find_or_create_by!` call in setup, and changed teardown's
`@game&.destroy` to `@game.destroy if @game_created_by_test && @game&.persisted?`.
This preserves the existing id stability (useful for debugging) while making
teardown idempotent: only records that setup actually created get destroyed,
eliminating the test-to-test leakage risk on aborted runs.

### WR-03: `set_target_points` leaks into non-BK2 branches as stray parameter

**Files modified:** `app/controllers/table_monitors_controller.rb`
**Commit:** `b25ebf2c`
**Applied fix:** Added a centralised cleanup block (lines 237-247) after the
`allow_follow_up` normalisation and before the Shootout / second-`four_ball`
blocks:

```ruby
unless p[:free_game_form] == "bk2_kombi" || p[:quick_game_form] == "bk2_kombi"
  p.delete(:set_target_points)
end
```

This keeps `set_target_points` in the `.slice(...)` whitelist (still needed
as the raw BK2 input) but guarantees it does NOT leak into
`GameSetup#build_result_hash` / `@options` / debug logs for pool, snooker,
karambol, or 4-Ball submissions. BK2 branches still consume it via
`p.delete(:set_target_points)` earlier and pack it into `p[:bk2_options]`;
by the time the post-processing cleanup runs, BK2 paths have already
removed the key (the `delete` here becomes a harmless no-op for them), so
the extra guard on `free_game_form`/`quick_game_form` is defence in depth.
