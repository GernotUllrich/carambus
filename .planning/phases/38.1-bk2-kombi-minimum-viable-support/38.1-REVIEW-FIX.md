### IN-05: Redundant `category_group.present?` guard in quick-buttons partial

**Files modified:** `app/views/locations/_quick_game_buttons.html.erb`
**Commit:** `4329fcc4`
**Applied fix:** Replaced the inline `category_group.present? &&
category_group['category'] == 'BK2-Kombi'` check with a proper structural
guard at the top of the loop: `<% next unless category_group.is_a?(Hash) &&
category_group['category'] %>`. The `is_bk2_kombi` expression then drops the
redundant presence check. This makes malformed preset entries (nil, string,
missing `category` key) skip cleanly rather than crash downstream at
`category_group['buttons'].each`.

### IN-06: `YAML.load_file` on every partial render

**Files modified:** `app/views/locations/_quick_game_buttons.html.erb`
**Commit:** `6f9e09cb`
**Applied fix:** Replaced the two-line `YAML.load_file(...)` + manual
`config.dig(Rails.env, ...) || config.dig('default', ...)` with a single
`presets = Carambus.config.quick_game_presets&.dig(table_kind_key) || []`.
This aligns with the `Carambus.config` convention documented in
`CLAUDE.md` (and implemented in `config/application.rb:10-17` as a memoised
OpenStruct that merges `default` + Rails.env-specific settings). Disk I/O
per partial render drops from one YAML parse to zero (after first-call
memoisation). Env-specific overrides of `quick_game_presets` are still
honoured because `Carambus.config` does the same default+env merge that
the inline code was doing by hand.

## Skipped Issues

### IN-01: `Discipline::BK2_DISCIPLINE_MAP` is declared but never referenced

**File:** `app/models/discipline.rb:154`
**Reason:** skipped per reviewer's own guidance — the review itself says
"Acceptable to defer — the current scope is minimal viable support." The
fix requires choosing between two non-trivial strategies (replace four
string-literal occurrences with `Discipline::BK2_DISCIPLINE_MAP.first`, OR
introduce a new `Discipline::BK2_KOMBI_NAME` constant) and then touching
four files across controllers/views to maintain consistency. A lint/test
cross-check would be a third option. All three are out of plan 06 scope
and better addressed as a dedicated follow-up that covers
`KARAMBOL_DISCIPLINE_MAP` indices at the same time.
**Original issue:** The constant is declared in plan 06 with a rationale
comment but no production code references it. Both the controller and
views use the string literal `"BK2-Kombi"` directly. A typo in any of
those string literals would bypass the constant entirely without a test
failure.

### IN-03: `derive_free_game_form` does not validate values from `discipline.data`

**File:** `app/services/table_monitor/game_setup.rb:240-242`
**Reason:** skipped per orchestrator's explicit guidance
("for findings requiring judgment or behavioral change... if the fix is
non-trivial, mark as skipped"). Adding a `KNOWN_FORMS` whitelist and a
`Rails.logger.warn` introduces new behaviour (a warn log line that did not
previously exist) and requires deciding the authoritative source-of-truth
list (pool / snooker / karambol / bk2_kombi — but also possibly "4 Ball"
derived forms). The reviewer notes this is "not a security issue —
`discipline.data` is authoritative by design (D-10)" and the fix would
"surface reconciliation bugs early rather than silently"; this is a
defence-in-depth polish worth a follow-up ticket where KNOWN_FORMS can be
defined alongside a test that enumerates acceptable values from all
dispatchers.
**Original issue:** `parsed_data["free_game_form"]` is returned directly
without a whitelist check. Typos or stale migration remnants in
`discipline.data` JSON would flow through to
`TableMonitor#data["free_game_form"]` and downstream dispatchers would
silently skip to their default branches.

### IN-04: BK2-Kombi selection not preserved on form reload

**File:** `app/views/locations/scoreboard_free_game_karambol_new.html.erb:219-221, 224`
**Reason:** skipped per orchestrator's explicit guidance
("IN-04 form reload semantics" flagged as a behavioural change). Initialising
`bk2_choice` from the persisted `options[:player_a][:discipline]` changes
form hydration semantics: currently the karambol radio re-selects index 0
("Dreiband klein") whenever the persisted discipline is `"BK2-Kombi"`, and
`bk2_choice` defaults to `null`. The fix would instead hydrate `bk2_choice`
from `[50, 60, 70].index(options.dig(:bk2_options, :set_target_points).to_i)`
and leave the karambol `discipline` uninitialised (or pinned to a neutral
value). That requires coordination with the Alpine x-data block (two
`discipline` / `discipline_a` / `discipline_b` computed values) and a UX
decision about what the karambol radio should display when a BK2 game is
being edited. The reviewer flags this as "Acceptable to defer"; it belongs
in a follow-up plan that also addresses session persistence scope (plan 06
explicitly excluded this per its CONTEXT.md).
**Original issue:** If a user starts a BK2 game, navigates away, and
returns to the detail form, their BK2 selection is lost — the Alpine
initializer reads `Discipline::KARAMBOL_DISCIPLINE_MAP.index("BK2-Kombi")`
which is `nil`, `.to_i` becomes `0`, so "Dreiband klein" silently
replaces the BK2 choice on reload.
