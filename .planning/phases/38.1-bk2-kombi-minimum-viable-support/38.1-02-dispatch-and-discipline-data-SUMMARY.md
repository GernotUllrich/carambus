## GameSetup Assignment Sites

Derivation landed in **two locations** in `app/services/table_monitor/game_setup.rb`:

1. **`initialize_game` (class method, line ~67)** — calls `derive_free_game_form` for the non-PartyMonitor branch. This fires when a game is initialized from scratch.

2. **`build_result_hash` (instance method, line ~388)** — calls `self.class.derive_free_game_form` as a fallback when `@options["free_game_form"]` is nil. This fires during `perform_start_game` → `build_result_hash` → `deep_merge_data!`.

Both sites are needed because `initialize_game` runs first (setting the value) but `build_result_hash` result overwrites it via `deep_merge_data!`. The fallback in `build_result_hash` ensures the derived value survives.

### derive_free_game_form logic

```ruby
def self.derive_free_game_form(tournament)
  discipline = tournament&.discipline
  return nil if discipline.nil?

  # Priority 1: discipline.data["free_game_form"] (JSON text column)
  parsed_data = begin
    JSON.parse(discipline.data) if discipline.data.present?
  rescue JSON::ParserError
    nil
  end
  return parsed_data["free_game_form"] if parsed_data.is_a?(Hash) && parsed_data["free_game_form"].present?

  # Priority 2: strict name match == "BK2-Kombi", logs reconciliation warning
  if discipline.name == "BK2-Kombi"
    Rails.logger.warn("[GameSetup] BK2-Kombi-Ableitung via Name-Match ...")
    return "bk2_kombi"
  end

  nil
end
```

**Key discovery:** `Discipline#data` is a `text` column with NO `serialize` declaration. It stores raw JSON string (or nil). `JSON.parse` is required — the plan's template checked `discipline.data.is_a?(Hash)` which would always be `false`. Fixed to parse the JSON text.
