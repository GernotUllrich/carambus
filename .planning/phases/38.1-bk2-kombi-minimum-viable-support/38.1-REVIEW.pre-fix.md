## Warnings

### WR-01: `four_ball` override can bulldoze BK2-Kombi settings silently

**File:** `app/controllers/table_monitors_controller.rb:118-124` and `:136-142`

**Issue:** In the quick-game branch (line 118), the BK2-Kombi block sets
`p[:discipline_a] = p[:discipline_b] = "BK2-Kombi"` and packs
`p[:bk2_options]`. But immediately after, line 136 unconditionally checks
`if p[:four_ball].present?` and, if true, overwrites `discipline_a/b` to
`"4 Ball"`, `balls_goal_a/b` to 120, and drops all BK2-specific state.
The `four_ball` branch runs AGAIN at line 237-247, re-asserting these values.

For a BK2 quick-button submission, `four_ball` is not emitted by the view,
so in practice this cannot misfire from trusted UI. However, a form-submitter
that sends both `quick_game_form=bk2_kombi` AND `four_ball=1` (either
malicious or via a bug in a future partial) would silently drop BK2 state
and start a 4-Ball game instead. This is inconsistent with the CLAMP
posture (T-38.1-06-01) that the plan explicitly adopts for `discipline_a/b`.

**Fix:** Short-circuit the `four_ball` branch when `quick_game_form` or
`free_game_form` is `bk2_kombi`:

```ruby
if p[:four_ball].present? && p[:quick_game_form] != "bk2_kombi" && p[:free_game_form] != "bk2_kombi"
  p[:discipline_a] = p[:discipline_b] = "4 Ball"
  # ...
end
```

Alternatively, restructure so the BK2 branch runs AFTER `four_ball` (reversing
order), or delete the duplicated `four_ball` block at 237-247 which is
already executed at 136-142.

### WR-02: System-test `find_or_create_by!` + `teardown .destroy` can delete a pre-existing Game

**File:** `test/system/bk2_kombi_scoreboard_test.rb:28` and `:55`

**Issue:** The setup uses `Game.find_or_create_by!(id: 50_000_200)` and the
teardown runs `@game&.destroy`. If a prior test run (or a fixture) left a
`Game` at id `50_000_200`, `find_or_create_by!` will return the existing
record and teardown will destroy it — leaking test-to-test state and
potentially wiping a fixture the next test depends on. Because system tests
do not wrap in transactions, this risk is real.

The file's own comment acknowledges the no-transaction constraint but does
not guard against the found-vs-created distinction.

**Fix:** Track whether the test actually created the record, so teardown only
destroys what setup created:

```ruby
setup do
  @tm = table_monitors(:one)
  @game_created_by_test = !Game.exists?(id: 50_000_200)
  @game = Game.find_or_create_by!(id: 50_000_200)
  # ...
end

teardown do
  @tm.update_columns(game_id: nil, state: "new")
  @tm.update!(data: {})
  @game.destroy if @game_created_by_test && @game&.persisted?
end
```

Or use a non-deterministic id (`id: 50_000_000 + rand(100_000)`) and always
destroy. Id 50_000_200 has no semantic meaning here — it just needs to be
above `MIN_ID`.

### WR-03: `set_target_points` leaks into non-BK2 branches as stray parameter

**File:** `app/controllers/table_monitors_controller.rb:80, 121, 208`

**Issue:** `set_target_points` is whitelisted in the `.slice(...)` list
(line 80) and deleted from `p` only inside the two BK2 branches (lines 121
and 208). For every other form submission (pool, snooker, karambol,
4-Ball), `set_target_points` — if present — remains in `p` and flows into
`@table_monitor.start_game(p)` → `GameSetup#build_result_hash`. It won't
be read by anything, but it pollutes `@options` and can appear in debug
logs with misleading values.

More importantly, the detail-form view ALWAYS emits
`<input name='set_target_points' :value="bk2_selected ? bk2_target : ''">`
(`scoreboard_free_game_karambol_new.html.erb:182`). When BK2 is not
selected, the submitted value is an empty string — harmless but noisy.

**Fix:** Drop `set_target_points` unconditionally at the end of param
processing, or restrict it to BK2 branches by removing it from the `.slice`
list and setting it only inside the BK2 branches by reading from raw
`params` directly:

```ruby
# After all branches, clean up stray BK2 params
p.delete(:set_target_points) unless p[:free_game_form] == "bk2_kombi" || p[:quick_game_form] == "bk2_kombi"
```

## Info

### IN-01: `Discipline::BK2_DISCIPLINE_MAP` is declared but never referenced

**File:** `app/models/discipline.rb:154`

**Issue:** `BK2_DISCIPLINE_MAP = ["BK2-Kombi"].freeze` is introduced in plan
06 with a detailed rationale comment, but no production code references it.
Both the controller (`table_monitors_controller.rb:119, 206`) and view
(`_quick_game_buttons.html.erb:124`, `scoreboard_free_game_karambol_new.html.erb:183`)
use the string literal `"BK2-Kombi"` directly. The constant is only asserted
by `test/models/discipline_test.rb`.

The comment explains the intent (future expansion, parallel to
`KARAMBOL_DISCIPLINE_MAP`), so this is not strictly dead code — but today it
is a write-once constant whose value is duplicated as a string literal in at
least four places. A typo in any of those string literals would bypass the
constant entirely without a test failure.

**Fix:** Either replace the string literals with `Discipline::BK2_DISCIPLINE_MAP.first`
(or a dedicated `Discipline::BK2_KOMBI_NAME = "BK2-Kombi"` constant), or add
a lint/test that cross-checks the constant against all string literal
occurrences. Acceptable to defer — the current scope is minimal viable
support.

### IN-03: `derive_free_game_form` does not validate values from `discipline.data`

**File:** `app/services/table_monitor/game_setup.rb:240-242`

**Issue:** `parsed_data["free_game_form"]` is returned directly without any
whitelist check. If `discipline.data` JSON contains an unexpected value
(e.g. typo, stale migration remnant, accidentally broad value), it flows
through to `TableMonitor#data["free_game_form"]` and downstream dispatchers
will silently skip to their default/else branches.

Not a security issue — `discipline.data` is authoritative by design (D-10) —
but a whitelist check would surface reconciliation bugs early rather than
silently.

**Fix:** Add a whitelist:

```ruby
KNOWN_FORMS = %w[pool snooker karambol bk2_kombi].freeze

if parsed_data.is_a?(Hash) && KNOWN_FORMS.include?(parsed_data["free_game_form"])
  return parsed_data["free_game_form"]
elsif parsed_data.is_a?(Hash) && parsed_data["free_game_form"].present?
  Rails.logger.warn("[GameSetup] Unknown free_game_form '#{parsed_data["free_game_form"]}' in discipline.data (id=#{discipline.id}); falling through to name-match")
end
```

### IN-04: BK2-Kombi selection not preserved on form reload

**File:** `app/views/locations/scoreboard_free_game_karambol_new.html.erb:219-221, 224`

**Issue:** The Alpine state initializer reads
`Discipline::KARAMBOL_DISCIPLINE_MAP.index(options[:player_a][:discipline].presence || "Freie Partie klein").to_i`.
If the currently persisted discipline is `"BK2-Kombi"`, `.index` returns
`nil`, `.to_i` becomes `0` — "Dreiband klein" is preselected. Meanwhile
`bk2_choice: null` initializes the BK2 radio to unselected, even though the
user previously selected BK2.

Effect: if a user starts a BK2 game, navigates away, and returns to the
detail form, their BK2 selection is lost and the karambol discipline
silently replaces it. Plan 06 does not list session persistence as in
scope, but this is a UX regression worth noting.

**Fix:** Initialize `bk2_choice` from the persisted discipline name:

```ruby
bk2_choice: <%= options[:player_a][:discipline] == "BK2-Kombi" ?
  [50, 60, 70].index(options.dig(:bk2_options, :set_target_points).to_i) || 0 :
  "null" %>,
```

Acceptable to defer — flag for follow-up.

### IN-05: Redundant `category_group.present?` guard in quick-buttons partial

**File:** `app/views/locations/_quick_game_buttons.html.erb:43`

**Issue:** `presets.each do |category_group|` already skips `nil` entries
in normal arrays. The inner check
`category_group.present? && category_group['category'] == 'BK2-Kombi'` is
defensive but adds noise. If `presets` contains a nil entry, the
`category_group['buttons'].each` at line 50 would crash anyway before
reaching the branch logic.

**Fix:** Drop the `category_group.present?` guard, or add a genuine guard at
the top of the loop:

```erb
<% presets.each do |category_group| %>
  <% next unless category_group.is_a?(Hash) && category_group['category'] %>
  ...
<% end %>
```

### IN-06: `YAML.load_file` on every partial render

**File:** `app/views/locations/_quick_game_buttons.html.erb:19-22`

**Issue:** `config = YAML.load_file(Rails.root.join('config', 'carambus.yml'))`
parses the YAML file from disk on every render of the partial (and this
partial renders on every visit to the quick-game entry screen). `Carambus.config`
is the established accessor for this file in the rest of the app (per
`CLAUDE.md`: "Custom config in `config/carambus.yml` accessed via
`Carambus.config`").

Performance is out of v1 review scope, but the inconsistency with project
convention is worth flagging.

**Fix:** Use `Carambus.config` and reference the preset key directly:

```erb
<% presets = Carambus.config.quick_game_presets&.dig(table_kind_key) || [] %>
```

(May require adjusting `Carambus.config` to return a deep-navigable OpenStruct
or Hash — check the wrapper in `config/initializers/carambus.rb` or similar.)
