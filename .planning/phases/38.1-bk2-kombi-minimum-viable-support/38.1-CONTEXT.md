# Phase 38.1: BK2-Kombi Minimum Viable Support - Context

**Gathered:** 2026-04-23
**Status:** Ready for planning
**Inserted:** After Phase 38 (urgent) — 2026-05-02 tournament deadline

<domain>
## Phase Boundary

Ship live-scoring scoreboard support for a BK2-Kombi tournament at the BCW club
on Saturday 2026-05-02. Must cover the BK2-Kombi format end-to-end on the table
monitor: shot-by-shot scoring, the two playing phases (Direkter Zweikampf /
Serienspiel), bonus-shot rule, foul handling (shot points become negative / go
to opponent), best-of-3 sets to configurable target (50 / 60 / 70), and match
winner = first to 2 sets.

**Fallback contract:** If a stretch item slips (e.g., full foul UX or phase
auto-transitions), the karambol scoreboard with negative-score support enabled
must remain a working fallback for this discipline on 2026-05-02 — the scorer
can operate it manually.

**Out of scope for 38.1:** Discipline duplicate cleanup (records 57/59/95 vs
107) — deferred to carambus_api scenario follow-up. Config-table-backed
rule_sets — deferred to a post-deadline phase. Generalisation to other
Kegel disciplines (Billard-Kegeln, 5-Kegel-Billard, Eurokegel) — may
piggy-back structurally but NOT part of the 38.1 verification.

</domain>

<decisions>
## Implementation Decisions

### Scope target (Area 1)
- **D-01:** Ship **full BK2-Kombi live scoring** by 2026-05-02, not just
  negative-score support. User is working full-time on this phase (~72 hours
  available through the deadline). Scope includes: engine gating for negatives,
  discipline wiring, two game modes (Direkter Zweikampf + Serienspiel),
  best-of-3 sets, pin+carom hybrid scoring with foul semantics, bonus-shot
  rule, dedicated scoreboard ERB partial, new scoring services.
- **D-02:** **Pre-tournament dry run on a real club table** is the UAT gate
  before 2026-05-02. Runs a full match end-to-end on real hardware with a real
  scorer. Blocks "release" if the dry run fails. Characterization tests +
  developer smoke test are not sufficient substitutes.
- **D-03:** If any stretch item slips during execution, the karambol fallback
  (D-07 with negative-score gate enabled for this discipline) must still be
  operational on 2026-05-02. Fallback is invoked by the user switching the
  tournament's discipline setting if necessary.

### Engine routing + negative-score gating (Area 2)
- **D-04:** Dispatch uses the existing `data["free_game_form"]` string pattern
  on `TableMonitor`. BK2-Kombi introduces a **new value** `"bk2_kombi"`. No
  migration, no new boolean column, no rename of the existing key. This aligns
  with how `"snooker"`, `"pool"`, etc. are dispatched today (see
  `table_monitor.rb`, `score_engine.rb`, `result_recorder.rb`,
  `game_protocol_reflex.rb`, `table_monitor_reflex.rb`, `game_setup.rb`,
  `options_presenter.rb`).
- **D-05:** `data["free_game_form"] = "bk2_kombi"` is **derived from the
  discipline** during tournament setup (in `GameSetup` / `OptionsPresenter`).
  Derivation reads `discipline.data["free_game_form"]` first; if absent, falls
  back to a name match `discipline.name == "BK2-Kombi"`. Explicit —
  name-matching allows the Saturday scenario to work even if the
  `discipline.data` write in carambus_api hasn't synced down yet.
- **D-06:** Negative-score gating at the **three known clamp/guard sites** in
  `app/models/table_monitor/score_engine.rb`:
  - `:84` — guard `(current_inning_value + n_balls) >= 0` must be bypassed when
    `data["free_game_form"] == "bk2_kombi"`
  - `:135` — clamp `[(current + add), 0].max` must not clamp at zero for BK2
  - `:690-692` — protocol modal rejection `"Negative Punktzahlen sind nicht
    erlaubt"` must not fire for BK2
  All three gated behind a helper like `allow_negative_scores?` that returns
  `data["free_game_form"] == "bk2_kombi"`. No separate boolean flag.
- **D-07:** Karambol fallback means: BK2-Kombi tournament configured against
  the BK2-Kombi discipline (id 107) with `free_game_form=bk2_kombi` uses the
  new scoreboard partial (D-12); if the new partial / services aren't ready,
  the admin can manually switch the tournament to a karambol discipline that
  also has `free_game_form="karambol"` AND the negative-score gate applies
  broadly when BK2 is active. Planner ensures the gate bypass is discipline-
  scoped (D-06) so karambol proper is unaffected.

### BK2-Kombi Discipline record (Area 3)
- **D-08:** Route all 2026-05-02 tournament traffic through **Discipline id 107**
  (`BK2-Kombi`, `table_kind="Small Billard"`). Of the four duplicate records
  (57 `BK 2kombi`, 59 `BK2-Kombi`, 95 `BK-2 Kombi`, 107 `BK2-Kombi`), id 107 is
  the only one with a `table_kind`, making it the natural canonical.
- **D-09:** **Do NOT merge** the duplicate records 57 / 59 / 95 during Phase
  38.1. Merging is a separate cleanup, tracked as a **deferred idea for the
  carambus_api scenario** (authoritative scenario for global records; id 107
  is a global record at id < 50M, so the merge must originate there). See
  `<deferred>` section.
- **D-10:** Set `discipline.data = { "free_game_form" => "bk2_kombi" }` on
  id 107 as the **authoritative** routing source.
  - **Primary path:** Make this edit in the `carambus_api` scenario checkout
    (authoritative for global records), commit, sync to BCW production.
  - **Fallback path (if sync timing is tight):** A local `unprotected=true`
    write from carambus_bcw directly on id 107's `data` column, tracked in
    Phase 38.1 execution log and reconciled in carambus_api afterwards.
  - **Either way** the `GameSetup` / `OptionsPresenter` derivation (D-05)
    reads the value and sets `data["free_game_form"]` on the TableMonitor.

### Templates + services (Area 4)
- **D-11:** Scoring services live in a **new namespace**
  `app/services/bk2_kombi/`:
  - `app/services/bk2_kombi/score_shot.rb` — pure shot evaluation (rohpunkte,
    foul logic, scoring assignment). Returns `{scoring, transitions}` hash.
    No DB writes.
  - `app/services/bk2_kombi/advance_match_state.rb` — applies scoring +
    transitions to `TableMonitor.data`, handles set close / match close.
  Matches the existing `app/services/table_monitor/` pattern (ResultRecorder,
  GameSetup, OptionsPresenter). Planner adapts the exact method signatures
  from `38.1-BK2-KOMBI-RULES.md` to fit Carambus conventions.
- **D-12:** Scoreboard UI is a **new dedicated ERB partial**
  `app/views/table_monitors/_scoreboard_bk2_kombi.html.erb`. Reuses karambol
  CSS/chrome (player cards, timer, layout grid) but has its own:
  - Shot-input form: fallen pins dropdown (0..5), middle-pin-only checkbox,
    true-carom / false-carom / passages checkboxes, foul dropdown (codes:
    `wrong_ball`, `no_object_ball_hit`, `premature_shot`, `direct_play_on_pins`,
    other), band-hit checkbox (for `no_object_ball_hit` refinement)
  - Phase indicator: "Direkter Zweikampf" / "Serienspiel"
  - Shots-left-in-turn badge (2 / 1 / bonus)
  - 3-set score display with target line
  - Running match score (sets_won)
  Scoreboard layout chooses partial based on `data["free_game_form"]`; karambol
  partial remains untouched.
- **D-13:** Shot payload shape (input to `Bk2Kombi::ScoreShot`) — rich hash:
  ```ruby
  {
    observations: {
      fallen_pins: Integer (0..5),
      middle_pin_only: Boolean,
      true_carom: Boolean,
      false_carom: Boolean,
      passages: Integer,
      foul: Boolean,
      foul_code: Symbol (:wrong_ball | :no_object_ball_hit | :premature_shot | :direct_play_on_pins | other),
      band_hit: Boolean
    },
    table_snapshot: { full_pin_image: Boolean }
  }
  ```
  Exact keys from the Perplexity reference; planner may rename for Ruby idioms.
- **D-14:** State on `TableMonitor.data` — new key `data["bk2_state"]` hash
  (pattern matches existing `data["snooker_state"]`):
  ```
  bk2_state: {
    current_set_number: 1..3,
    current_phase: "direkter_zweikampf" | "serienspiel",
    player_at_table: "playera" | "playerb",
    shots_left_in_turn: 0..2,
    set_scores: { "1" => { playera:, playerb: }, "2" => ..., "3" => ... },
    sets_won: { playera:, playerb: },
    set_target_points: 50 | 60 | 70
  }
  ```
  Per-turn credited points flow into the existing `innings_list` for protocol
  compatibility; per-shot detail optional in a new `shots_list` at planner's
  discretion.
- **D-15:** Rule config (pin=1, middle_pin_only_from_full_image=2, carom=1,
  false_carom_with_pin_or_passage=1, passage=1, foul values) — stored as
  a **module-level constant** `Bk2Kombi::ScoreShot::DEFAULT_RULES` for Phase
  38.1. A config-table migration is explicitly out of scope.
- **D-16:** Foul values per `38.1-BK2-KOMBI-RULES.md` (wrong_ball=-6,
  no_object_ball_hit with band=-1 / without=-2, premature_shot=-1,
  direct_play_on_pins=max(1, fallen_pins), other=-1). Planner confirms against
  the billardregel.de source and the DBU PDF linked in the rules doc.

### Claude's Discretion

<specifics>
## Specific Ideas

- **"Full live scoring"** to the user means: the volunteer scorer enters one shot at a time through the scoreboard (pin count + carom/foul checkboxes + optional foul dropdown), and the scoreboard automatically handles the two playing phases, bonus-shot detection, player-at-table transitions, set score updates, set close, and match close. No paper protocol necessary for scoring (though the paper copy remains a legal fallback).
- **Pre-tournament dry run** specifically means: before 2026-05-02, run a complete 3-set dry-run match on an actual club table with a real scorer (not just developer-machine testing). The dry run catches touchscreen latency, input-form ergonomics, and workflow issues that unit tests cannot.
- **Perplexity transcript** (saved at `38.1-BK2-KOMBI-RULES.md`): the user brought this in during discuss-phase as a pre-digested rules + architecture reference. Replaces the phase-researcher's would-be billardregel.de web-scrape round trip. Architectural proposals in the transcript (`Bk2Kombi::ScoreShot` / `AdvanceMatchState`) have been adopted literally in D-11.
- User quote on adaptation: *"of course that has to be adapted to carambus MVC structure"* — meaning Perplexity's greenfield Rails API shape (with a new `Match` / `RuleSet` ActiveRecord model) is NOT the target; the target is the existing TableMonitor / `data`-hash state model with new services alongside.
