## Background

Discipline id 107 (`BK2-Kombi`, `table_kind = "Small Billard"`) is a global record
(id < 50,000,000). Its `data` column is **plain text** (no Rails serializer) with
the project convention that structured content is stored as a **JSON-encoded
string**. `GameSetup.derive_free_game_form` calls `JSON.parse(discipline.data)`
to read it, and uses the parsed hash as the authoritative path (Priority 1).
Otherwise it falls back to name-match (Priority 2, which logs a reconciliation
warning).

**Write shape:** `d.data = { "free_game_form" => "bk2_kombi" }.to_json`
(Do NOT use `(d.data || {}).merge(...)` — `d.data` is a String `""` or a
JSON-encoded string, never a Hash, so `.merge` raises NoMethodError.)

**The name-match fallback WILL work** on tournament day even without this write —
`discipline.name == "BK2-Kombi"` is the safety net. But completing this write
makes the derivation authoritative, removes the warning from the logs, and
satisfies T-38.1-05 (Tamper mitigation).

**Sync attempt → bug discovered:**
Syncing Discipline 107 to carambus_bcw triggered `can't cast Hash` at
`app/models/version.rb:333` in `update_from_carambus_api`. Root cause: the
sync code's fallback branch (lines 329-331) calls `YAML.load(args["data"])`;
valid JSON is also valid YAML flow syntax, so YAML.load returns a Hash, and
`update_columns` then rejects the Hash for the `text` column. This is a
**pre-existing bug** exposed by the first structured write to `disciplines.data`.
See backlog item `sync-version-yaml-load-json-collision` for the fix path.
