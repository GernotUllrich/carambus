<objective>
Wire the tiebreak-detection branch into `TableMonitor::ResultRecorder`:

1. **Trigger detection (D-03):** in `perform_evaluate_result` (around line 332-341),
   when end_of_set? has fired and the panel_state is being set to "protocol_final",
   check if a tiebreak is required AND scores are tied. If so, switch
   `current_element` from `"confirm_result"` to `"tiebreak_winner_choice"` so the
   modal (Plan 06) renders the radio fieldset and blocks Submit.

2. **Training-rematch guard (D-13):** in the training rematch branch (lines 370-377
   and 386-393), block the auto-rematch when a tiebreak is pending. Otherwise
   `revert_players` + `do_play` would discard the tied result before the operator
   gets to pick.

3. **ba_results derivation (D-08):** in `update_ba_results_with_set_result!`
   (lines 111-143), read `game.data['tiebreak_winner']` and write
   `ba_results['TiebreakWinner'] = 1` (for "playera") or `2` (for "playerb"). This is
   what Plan 07's PDF view consumes.

4. **acknowledge_result guard (D-08, defense-in-depth):** add a server-side check
   that refuses to clear the protocol modal if the tiebreak is required, scores are
   tied, and no winner pick has landed in `game.data['tiebreak_winner']`. This
   prevents a forged client-side state-bypass.

Output: ~40 LOC across 3 methods in `result_recorder.rb` + 8 unit tests.
</objective>

      return false if @tm.game.data["tiebreak_winner"].present?

    end
