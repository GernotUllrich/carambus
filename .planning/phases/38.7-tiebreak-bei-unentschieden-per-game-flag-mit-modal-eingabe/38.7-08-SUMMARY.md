## Decisions Made

- **Service-level dispatch over full `start_game` (key-decision 1).** The plan's prescribed code calls `@tm.start_game(...)` with a complete options hash. The project's fixture set does not have a Tournament + TournamentMonitor + TournamentPlan + game_plan + Karambol-Liga AASM state wired up to satisfy the full `GameSetup#perform_start_game` invariants. The plan's `<action>` block explicitly anticipates this: 'the 4 tests above MAY need to be merged into fewer if some scenarios cannot be set up with available fixtures — preserve the INTENT'. Direct data setup is the established convention (bk2_scoreboard_test.rb 35-test suite uses the same approach for Plan 38.5 D-11 / D-12 end-to-end tests at lines 1366-1536; table_monitor_test.rb T9 AASM-guard tests from Plan 05 use `@tm.update!(data: ...) + @tm.update_columns(game_id: ...)`). Same code paths exercised, no fixture-graph cost.
- **Reflex via .allocate stubs over Capybara click_button (key-decision 2).** StimulusReflex WebSocket roundtrip is async-only; in headless Selenium, the click would issue an ActionCable subscribe + reflex invoke, but the test would have to poll for the side effect — non-deterministic. The established convention (game_protocol_reflex_test.rb T1-T5 from Plan 06 Task 4) uses `.allocate` + `define_singleton_method` stubs to skip the StimulusReflex base initializer and drive the reflex method directly. The system test still asserts the form + radio + submit-button selectors via `assert_selector` (proving the modal renders correctly); only the click→reflex roundtrip is replaced.
- **TR-Sec covers BOTH 'playerc' AND empty params (key-decision 3).** T-38.7-06-01 covers any bypass of the radio fieldset. The reflex's allowlist guard `%w[playera playerb].include?(tw)` rejects both 'playerc' AND nil/empty (because `nil.to_s == ''` and `%w[playera playerb].include?('')` is false). Two probe variants in one test method lock the contract from both bypass angles.
- **TR-D (Snooker/Pool simple-set parity) deferred to Plan 05 unit tests (key-decision 4).** Plan 05 result_recorder_test.rb T1-T9 cover the simple-set branch directly (`@tm.simple_set_game?` true → tied detection on `Array(@tm.data['sets']).last`). Adding a 5th system test for Snooker/Pool would duplicate that coverage at a more expensive layer (browser launch + page render). System test scope here is the inning-based path + forged-submit probe.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking syntax] `assert_selector` keyword-vs-message-string ambiguity**

- **Found during:** First GREEN-phase run after Task 1 file creation.
- **Issue:** Plan-prescribed code wrote multiple `assert_selector "...selector...", visible: :all, "message string"`. Ruby parses the trailing string as a positional after the kwargs hash, but Capybara's `assert_selector` does not accept a message argument in that position — the parser interprets the trailing kwargs + string as expecting a `=>` and raises `syntax error, unexpected '\n', expecting =>`.
- **Fix:** Removed the trailing message strings from `assert_selector` / `assert_no_selector` calls. Moved the intent into a `# TR-Xxx: ...` comment immediately above the assertion. The actual selector + kwargs preserved verbatim. (Capybara's assertion failure messages already include the selector string and the page state, so the loss is purely cosmetic.)
- **Files modified:** `test/system/tiebreak_test.rb` (committed in `a9741f85`).

## Issues Encountered

None blocking. Two observations worth recording for future work:

1. **System tests do NOT exercise the WebSocket reflex roundtrip end-to-end.** The reflex's confirm_result method is invoked directly via `.allocate` + stubs. If a future regression breaks the StimulusReflex transport layer (`data-reflex="submit->GameProtocolReflex#confirm_result"` attribute, ActionCable subscription, channel routing), THIS test suite would not catch it. The 5 functional reflex tests in game_protocol_reflex_test.rb have the same coverage gap. Mitigation: existing `test/system/table_monitor_isolation_test.rb` exercises the WebSocket transport layer for the broadcast direction; covering the reflex-invocation direction at the system-test level would require a separate, expensive infrastructure investment (Selenium + ActionCable + Sidekiq integration). Not in scope for this plan.

2. **TR-B's `end_of_set?` assertion exercises Plan 02's D-02 branch but NOT the legacy branch.** The test asserts `@tm.end_of_set?` returns true for the Anstoss-at-goal + Nachstoss-finished-followup case. The legacy branch (innings parity) would NOT fire here because innings 5 vs 6 is asymmetric. If Plan 02's branch were accidentally removed in a future refactor, this assertion would catch it. If Plan 02's branch were over-broadened (e.g., fires for innings 5 vs 7), the test would not detect that — a Plan 02 unit test would be needed. Plan 02 has its own test/services/table_monitor/end_of_set_test.rb-equivalent coverage (4 tests in the existing test_monitor characterization suite per Plan 02 SUMMARY). This is the right separation of concerns.
