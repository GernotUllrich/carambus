## Decisions Made

- **Rule 1 — direct mutation `@tm.game.data['k']=v` does NOT dirty-track on Game.** Plan-prescribed code in Task 2 used the direct-mutation pattern (same shape as Plan 04 originally tried). Tests T3 + T4 both showed `Expected: "playera|b"  Actual: nil` after `@game.reload`. Root cause: `Game#data` (game.rb:114-131) returns a freshly-decoded Hash on every call, so `obj.data["k"] = v` mutates a temporary that's discarded on save. Switched to `@table_monitor.game.deep_merge_data!("tiebreak_winner" => tw)` which calls `data_will_change!` and reassigns `self.data`. All 5 tests turned GREEN. Inline comment above the call documents the constraint.
- **Tests use `Game.create!(data: ..., group_no:, seqno:, table_no:)` rather than fixtures.** Project has no `games.yml` fixture file; existing TableMonitor T9 tests in `test/models/table_monitor_test.rb` (Plan 05) use the same `Game.create!` pattern. Keeps the tests self-contained and DB-state-free.
- **`update_columns` for `panel_state`/`current_element`/`game_id` setup** to bypass AASM callbacks during fixture priming. The TableMonitor's AASM transitions can fire side effects when `panel_state` changes via `update!`; `update_columns` is a raw SQL update and side-steps callbacks safely. T9 tests in Plan 05 use the same approach.
- **`Object.stub` (Minitest stdlib) for `TableMonitorJob.perform_later`** — the project has no mocha-style mocking helpers wired in; stdlib `stub` is sufficient and deterministic. Lambda body `->(_a, _b) {}` is empty (returns nil implicitly) to satisfy standardrb's `Style/NilLambda` rule.
- **i18n placement after `bk2_kombi:` block, before `table_id:`.** This is the natural slot for a new `table_monitor.*` sub-namespace and matches the existing tree shape (`status:`, `bk2_kombi:` are the existing sub-blocks; `tiebreak:` joins as a sibling).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Plan-prescribed direct mutation of `@table_monitor.game.data` does NOT dirty-track**

- **Found during:** Task 4 GREEN phase, after the 5 tests landed and were run for the first time.
- **Issue:** Plan-prescribed code for Task 2 wrote:
  ```ruby
  @table_monitor.game.data ||= {}
  @table_monitor.game.data["tiebreak_winner"] = tw
  @table_monitor.game.save!
  ```
  Tests T3 and T4 both failed with `Expected: "playera"|"playerb"  Actual: nil` after `@game.reload`. Root cause: `Game` has a custom `def data` getter (game.rb:113-131) that calls `read_attribute_before_type_cast(:data)` and returns a freshly-decoded Hash via `safe_decode_data`. Mutating the returned Hash does NOT dirty-track because each `.data` call creates a new Hash, and `save!` writes the un-mutated original. **This is the exact same Rule 1 deviation Plan 04 documented** for `@tm.game.data['tiebreak_required']` writes, surfaced here in a different file.
- **Fix:** Replaced direct mutation with `@table_monitor.game.deep_merge_data!("tiebreak_winner" => tw)`. `deep_merge_data!` (game.rb:266-277) calls `data_will_change!` and reassigns `self.data = JSON.parse(h.to_json)`, which dirty-tracks correctly. Result: all 5 tests GREEN.
- **Files modified:** `app/reflexes/game_protocol_reflex.rb` (committed in `73cab1aa`, the Task 4 commit, since the discovery only became visible when the tests ran).
- **Verification:** All 5 reflex tests GREEN (`bin/rails test test/reflexes/game_protocol_reflex_test.rb` → 5 runs / 12 assertions / 0 failures). 47-test nearby suite (table_monitor_test + result_recorder_test + game_test) still GREEN. Inline comment above the `deep_merge_data!` call documents the dirty-tracking constraint for future readers.
- **Committed in:** `73cab1aa` (Task 4 + Rule 1 fix). The plan's prescribed reflex code was wrong about persistence semantics; the fix is documented for the verifier.
