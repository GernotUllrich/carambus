# Phase 38.7 (Plans 09-12): Code Review Report

**Reviewed:** 2026-04-30T20:30:00Z
**Depth:** standard
**Files Reviewed:** 12
**Status:** issues_found

## Summary

Gap-closure review for phase 38.7 plans 09-12 (training-mode tiebreak sources):
preset-level (Plan 09), free-game detail-form toggle (Plan 10), BK-2kombi BK-2-phase
auto-detect (Plan 11), TournamentMonitor startup-form override (Plan 12).

Overall the implementation is sound and correctly threads the new tiebreak source
through controller → GameSetup → Game.data['tiebreak_required']. The four-level
hierarchy (Tournament.data → TournamentPlan.executor_params → preset/detail-form
override → BK-2kombi auto-detect override) respects the project rule that the
flag must NEVER come from Discipline rows or local DB seeding. Tests cover the
core RED→GREEN paths for each plan.

Key concerns surfaced:

1. **Plan 09 sparse-override gate is fragile** — the controller's normalization
   to Bool means `@options.key?("tiebreak_on_draw")` correctly preserves the
   "explicit-false-overrides-resolver-true" semantic, but the GameSetup
   re-normalizes from String/Bool in a way that turns explicit `false` into
   `false` and explicit `true` into `true` — that's correct, but the unit test
   for tournament-mode interaction (Plan 04 resolver-true vs preset-explicit-false)
   is missing — only the training-mode path is exercised. **Warning WR-01.**
2. **Plan 11 auto-detect mutation has a subtle dependency** —
   `bk2_kombi_tiebreak_auto_detect!` calls `@tm.game.save!` while the parent
   `tiebreak_pick_pending?` is invoked from inside `perform_evaluate_result`,
   which runs in a save-heavy state-transition path. A save inside a predicate
   is non-obvious and reachable from multiple call sites. **Warning WR-02.**
3. **Plan 12 `update_columns` correctly bypasses LocalProtector by design**, but
   this is a load-bearing behavioral choice that depends on auth gates upstream;
   any future controller change that reaches `persist_tournament_tiebreak_override`
   from a non-director-gated action would silently bypass LocalProtector.
   **Warning WR-03.**
4. **Tournament_monitors_controller#derive_tiebreak_default uses inline `rescue;{};end`** —
   bare rescue swallows any error class, including LoadError or
   SystemExit. The Game model's `parse_data_hash` is the canonical helper and
   should be reused. **Warning WR-04.**

No critical security issues. No data-loss risks. Test quality is generally good
(RED→GREEN structure with explicit assertion messages). One missing test gap
documented under IN-04.

## Warnings

### WR-01: Plan 09 lacks tournament-mode override regression test

**File:** `test/services/table_monitor/game_setup_test.rb:467-532`
**Issue:** The Plan 09 tests (G1/G2/G3) all set `@tm.tournament_monitor_id = nil`
to exercise training-mode. None of the three tests cover the project-critical
case "tournament-mode resolver returns true via Tournament.data, preset arrives
with explicit false → preset wins (sparse-override consistent with D-06)." The
phase 38.7 scope_note in the review request explicitly calls this out as a
critical semantic. The Plan 04 resolver order plus the GameSetup override block
DO implement this correctly (lines 379-395 of game_setup.rb), but no test locks
the contract. A refactor that moves the override block before the resolver call
would silently regress Plan 04's tournament-mode path.

**Fix:** Add a fourth test that:
1. Builds a mock TournamentMonitor + Tournament with `data: { "tiebreak_on_draw" => true }`
2. Calls `GameSetup.call` with `options: { ..., "tiebreak_on_draw" => false }`
3. Asserts `@tm.game.data["tiebreak_required"] == false`

```ruby
test "G4 (Gap-01): explicit preset tiebreak_on_draw=false overrides tournament-mode resolver-true" do
  mock_tm_monitor, _ = build_mock_tm_with_plan(
    tournament_data: {"tiebreak_on_draw" => true},

    table_monitor: @tm,

      "discipline_a" => "BK-2", "free_game_form" => "bk_2",
      "tiebreak_on_draw" => false
    )
  )

  @tm.reload
  assert_equal false, @tm.game.data["tiebreak_required"],
    "Sparse-override: preset-explicit-false must beat resolver-true (Plan 04 + Plan 09 contract)"
end
```

### WR-02: Side-effecting save inside predicate `tiebreak_pick_pending?`

**File:** `app/services/table_monitor/result_recorder.rb:309-326`
**Issue:** `tiebreak_pick_pending?` is named as a predicate (Ruby convention:
predicates should be pure reads), but its first action is to call
`bk2_kombi_tiebreak_auto_detect!` which mutates `@tm.game.data['tiebreak_required']`
and calls `@tm.game.save!` (line 365). The predicate is invoked from at least
4 call sites in `perform_evaluate_result` (lines 412, 429, 467, 488), each
during AASM transitions that already save state. A future caller introduced via
a different code path (e.g., a view helper or reflex predicate) could trigger
unexpected DB writes via what looks like a read-only check.

The auto-detect logic IS correct in this location (Plan 11 needs the mutation
visible to all subsequent reads in the same call chain — see the docstring
referencing `D-03 wiring + Gap-03`), but the predicate naming hides a side
effect.

**Fix:** Either rename or extract the mutation to a separate explicit call:

```ruby
# Option A: rename to make side effect visible
def tiebreak_pending_after_auto_detect!
  bk2_kombi_tiebreak_auto_detect!
  return false unless @tm.game&.data&.[]("tiebreak_required") == true
  # ...
end

# Option B: lift the auto-detect call to perform_evaluate_result (one place)
# and keep tiebreak_pick_pending? as a pure read.
def perform_evaluate_result
  bk2_kombi_tiebreak_auto_detect! if @tm&.data["free_game_form"] == "bk2_kombi"
  # ... existing logic ...
end
```

Option B is preferred — the auto-detect is a state-transition concern, not a
predicate concern. It also avoids redundant calls (currently auto-detect runs
4× per evaluate_result invocation due to the multiple `tiebreak_pick_pending?`
call sites; idempotence makes this correct but wasteful, and the
`@tm.game.deep_merge_data!` + `save!` round-trip is not free).

### WR-03: Plan 12 `update_columns` silently bypasses LocalProtector

**File:** `app/controllers/tournament_monitors_controller.rb:281`
**Issue:** `t.update_columns(data: current, updated_at: Time.current)` is
deliberately chosen to skip Tournament's `before_save` (organizer-presence
validation) and PaperTrail. This is documented in the helper docstring
(lines 264-280). However, `update_columns` ALSO bypasses LocalProtector's
`disallow_saving_global_records` callback (`after_save` hook —
`app/models/local_protector.rb:22`).

For this controller path the bypass is safe because:
1. `:update` is gated by `ensure_local_server` (line 4) — only fires on local
   servers
2. `:update` is gated by `ensure_tournament_director` (line 3) — only club_admins
3. `tournaments(:local)` fixture has `id: 50_000_001` (local id, MIN_ID range)
   so LocalProtector wouldn't have blocked anyway
4. In test env, `disallow_saving_global_records` early-returns

But: a future contributor who reaches `persist_tournament_tiebreak_override`
from a non-director-gated action (e.g., re-uses the helper from
`update_games`) would silently bypass LocalProtector for global Tournament
records. The current `id < 50_000_000 && local_server?` guard in LocalProtector
exists precisely to prevent local servers from corrupting global records that
will later sync from carambus_api.

**Fix:** Add an explicit defensive check before `update_columns` to make the
intent visible and survive future refactors:

```ruby
def persist_tournament_tiebreak_override(tournament_monitor)
  return unless params.key?(:tournament_tiebreak_on_draw)
  t = tournament_monitor&.tournament
  return if t.nil?

  # Defense-in-depth: even though :update is director-gated + local-server-gated,
  # writing tiebreak_on_draw onto a global Tournament (id < MIN_ID) on a local
  # server should be blocked unless explicitly authorized. The auth gates above
  # are the primary protection; this matches LocalProtector intent and survives
  # future refactors that might call this helper from a non-gated action.
  if t.id.present? && t.id < ApplicationRecord::MIN_ID && ApplicationRecord.local_server?
    Rails.logger.warn "[TournamentMonitorsController] Refusing to write Tournament.data " \
      "tiebreak_on_draw on global Tournament[#{t.id}] from local server"
    return
  end
  # ... rest unchanged ...
end
```

Alternatively, document in the docstring (already partially done) that this
helper assumes director+local-server gating and that any new caller MUST
preserve those gates. The current docstring covers `:create` vs `:update` but
does not call out the LocalProtector bypass.

### WR-04: Bare-rescue in `derive_tiebreak_default` swallows all errors

**File:** `app/controllers/tournament_monitors_controller.rb:225,230`
**Issue:** Two inline `begin; JSON.parse(...); rescue; {}; end` rescues catch
**StandardError-and-below** (Ruby's bare `rescue` defaults to `StandardError`).
While safer than catching `Exception`, this still swallows
`ActiveRecord::ConnectionNotEstablished`, `Encoding::UndefinedConversionError`,
and any other `StandardError` subclass — turning a real infrastructure problem
into a silent default-false return. It also duplicates parsing logic that
already exists as `Game.send(:parse_data_hash, ...)` (private method on Game
model, line 100-110).

**Fix:** Reuse `Game.parse_data_hash` (extract from private to public, or
expose via a dedicated `Carambus::DataParser` utility — already a TODO per
phase context):

```ruby
def derive_tiebreak_default(tournament_monitor)
  t = tournament_monitor&.tournament
  return false if t.nil?

  t_data = Game.send(:parse_data_hash, t.data)
  return !!t_data["tiebreak_on_draw"] if t_data.key?("tiebreak_on_draw")

  plan = t.tournament_plan
  if plan&.respond_to?(:executor_params)
    plan_params = Game.send(:parse_data_hash, plan.executor_params)
    g1 = plan_params["g1"]
    return !!g1["tiebreak_on_draw"] if g1.is_a?(Hash) && g1.key?("tiebreak_on_draw")
  end

  false
end
```

If `Game.parse_data_hash` cannot be made public, narrow the rescue to
`JSON::ParserError` and `TypeError` only, and add a Rails.logger.warn:

```ruby
t_data = if t.data.is_a?(Hash)
  t.data
else
  begin
    JSON.parse(t.data.to_s)
  rescue JSON::ParserError, TypeError => e
    Rails.logger.warn "[derive_tiebreak_default] could not parse Tournament[#{t.id}].data: #{e.class} #{e.message}"
    {}
  end
end
```

### IN-02: `carambus.yml.erb` does NOT contain the Plan 09 tiebreak_on_draw flags

**File:** `config/carambus.yml.erb`
**Issue:** This file is the ERB template that the project's deployment pipeline
renders into `config/carambus.yml`. The Plan 09 flags `tiebreak_on_draw: true`
appear in `config/carambus.yml` (lines 46-49, 61-63) but NOT in
`carambus.yml.erb`. Either:
1. The presets are managed differently (per-scenario, manually edited in
   `carambus.yml`), and the `.erb` is only for env-specific overrides; OR
2. The `carambus.yml` was edited directly and will be overwritten on the next
   deploy.

Looking at the diff, `carambus.yml.erb` only contains environment header and
five scalar fields (`carambus_api_url`, `location_id`, etc.) — it does NOT
contain `quick_game_presets` at all. The presets in `carambus.yml` come from
the `default:` section which lives entirely in the rendered file. This is
likely intentional (presets are static config, not env-specific), but it means
**the Plan 09 changes will be lost on the next `cap deploy` if the pipeline
re-renders `carambus.yml` from `carambus.yml.erb`**.

**Fix:** Verify with `bundle exec cap -T | grep yml` whether the deployment
pipeline overwrites `carambus.yml` from the `.erb` template. If yes, the Plan 09
tiebreak_on_draw flags must be moved into `carambus.yml.erb`'s `default:`
section. If no (i.e., `carambus.yml` is committed and deploy uses the file
as-is), this finding is informational only. Per the project's
[scenario-management skill](.agents/skills/scenario-management/SKILL.md),
config templates often have surprising rendering rules — confirm before
shipping.

### IN-03: `_form.html.erb` checkbox always renders even when create is auth-bypass

**File:** `app/views/tournament_monitors/_form.html.erb:49-53`
**Issue:** The tiebreak checkbox renders unconditionally on both `new` (create
form) and `edit` (update form). The Plan 12 docstring explicitly states the
checkbox is intentionally ignored on `:create` for auth reasons (see
controller line 168-175). This is documented but creates a subtle UX trap:
operators who tick the box during creation will see no error, no warning, and
will assume their pick was saved — only to discover later that it was silently
dropped.

**Fix:** Either:
1. Hide the checkbox on the `new` form via `<% if form.object.persisted? %>`
   guard, OR
2. Add a visible `<small>` hint when on the new form: "(Tiebreak-Einstellung
   kann nach dem Erstellen über 'Bearbeiten' gesetzt werden)"

Option 2 is friendlier and preserves UX continuity. The Plan 12 docstring
already explains the reason (line 44-47 of the form), but the UI itself is
silent.

### IN-04: No integration test exercises the four-level resolver chain end-to-end

**File:** `test/services/table_monitor/game_setup_test.rb` and
`test/services/table_monitor/result_recorder_test.rb`
**Issue:** Each plan has its own unit-test cluster, but no test exercises the
full chain in priority order: Tournament.data > TournamentPlan > preset > BK-2kombi
auto-detect. The chain is correct in the code, but the priority semantic is
load-bearing:

- Plan 04 resolver: Tournament.data > TournamentPlan
- Plan 09 override: preset overrides resolver default-false (training mode)
- Plan 09 sparse-override: explicit-false-in-options must beat resolver-true
- Plan 11: auto-detect overrides ALL prior values (rule, not configurable)

Plans 04 + 09 are tested separately; the interaction (preset + tournament both
present) is not.

**Fix:** Add an integration test in `result_recorder_test.rb` that:
1. Sets up Tournament.data with `tiebreak_on_draw: true`
2. Calls GameSetup with options including `tiebreak_on_draw: false`
3. Asserts `game.data["tiebreak_required"] == false` (preset explicit false wins)
4. Then triggers the BK-2kombi auto-detect path
5. Asserts auto-detect override fires correctly

This locks the priority semantic against future refactors.

### IN-05: Logger.info noise in production for every start_game

**File:** `app/services/table_monitor/game_setup.rb:402-404`
**Issue:** New `Rails.logger.info` line emitted unconditionally on every
`start_game` call:
```ruby
Rails.logger.info "[GameSetup] tiebreak_required=#{tiebreak_required} " \
  "game=#{@tm.game.id} tournament=#{tournament&.id.inspect} " \
  "group_no=#{group_no.inspect} preset_override=#{@options.key?("tiebreak_on_draw")}"
```
Phase 38.1 IN-02 documented a project-wide downgrade-info-to-debug pattern for
hot paths (see line 162, 210 in same file). This new log line follows the
older pattern. Per the Phase 38.1 IN-02 convention, this should be a
`Rails.logger.debug { "..." }` block-form to avoid string interpolation in
production.

**Fix:**
```ruby
Rails.logger.debug do
  "[GameSetup] tiebreak_required=#{tiebreak_required} " \
  "game=#{@tm.game.id} tournament=#{tournament&.id.inspect} " \
  "group_no=#{group_no.inspect} preset_override=#{@options.key?("tiebreak_on_draw")}"
end
```

Same pattern applies to:
- `result_recorder.rb:362-363` (Plan 11 auto-detect log) — keep at info, this
  is a true rule-fire event worth recording.
- `tournament_monitors_controller.rb:282-283` (Plan 12 persist log) — keep at
  info, this is an operator-action audit trail.

### IN-06: `derive_tiebreak_default` parsing duplicates `Game.parse_data_hash`

**File:** `app/controllers/tournament_monitors_controller.rb:225-232`
**Issue:** The inline JSON-parse-or-Hash logic on lines 225 and 230 reimplements
`Game.parse_data_hash` (private method on Game model, file `game.rb:100`).
Triplication risk: every consumer of `Tournament.data` and
`TournamentPlan.executor_params` needs the same tolerant parsing logic, and
the rules for "what's a corrupt level → fall through" are non-obvious.

**Fix:** Same as WR-04. Extract `parse_data_hash` to a public utility module
(e.g., `Carambus::JsonField`) so all consumers share one implementation.
Until then, accept the duplication as documented technical debt.
