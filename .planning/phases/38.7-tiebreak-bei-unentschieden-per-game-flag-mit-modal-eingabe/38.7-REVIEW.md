## Summary

The wiring fix itself is correct and minimal. Moving `data-reflex="submit->GameProtocolReflex#confirm_result"` onto the `<form>` is the idiomatic StimulusReflex pattern for submit-event observation, and the `action="javascript:void(0)"` defense-in-depth is well-reasoned (it neutralizes the smoking-gun GET fall-through documented in the dev log). The reflex code (`GameProtocolReflex#confirm_result`) reads `params[:tiebreak_winner]` and `element.dataset['id']` — both still work because the form serializes the radio input and `data-id` was relocated alongside `data-reflex`. The submit `<button form="tiebreak-form-<id>">` association preserves HTML5 `required` validation.

The integration test correctly locks the wiring contract at two levels (Nokogiri attribute assertions G1-G3, request-level invariant G4) and chooses Nokogiri-on-render over Capybara-on-submit deliberately for determinism.

The issues found are all in the test file — primarily quality concerns about teardown safety, fixture fragility, and a hidden assumption that makes G4's no-op claim weaker than its message advertises. None block merge; all are addressable in a follow-up.

## Warnings

### WR-01: G4's no-op assertion silently relies on `remote_control_detected` being false

**File:** `test/integration/tiebreak_modal_form_wiring_test.rb:208-229`
**Issue:** G4 asserts that `GET /table_monitors/<id>?tiebreak_winner=playera` does not advance state. But the actual reason the controller does not advance state is that `TableMonitorsController#show` calls `@table_monitor.evaluate_panel_and_current`, and that method early-returns `unless remote_control_detected` (`app/models/table_monitor.rb:996`). The fixture `table_monitors(:one)` happens to have no remote control attached, so the call is a no-op. If a future change attaches a remote control to that fixture (or `evaluate_panel_and_current` later grows a code path that runs unconditionally), G4 will start passing or failing for the wrong reason — its message claims it locks the *controller* against state-advance from a GET, but it actually locks the *fixture state*.

Note also that `show` already mutates state on the response path: it can `redirect_to locations_path` (when `@table_monitor` blank) or `redirect_to location_url` (when `game_id` blank). The setup explicitly assigns `@game.id` to avoid the second branch — but this is implicit, not asserted as a precondition.

**Fix:** Either make the no-op invariant explicit, or remove the misleading "controller MUST NOT advance state" framing.

```ruby
# Add an explicit precondition assertion so the test fails loudly if
# the fixture changes in a way that makes the no-op claim vacuous.
assert_not @tm.remote_control_detected,
  "Precondition: fixture must NOT have remote_control_detected, " \
  "otherwise evaluate_panel_and_current can mutate state and G4's " \
  "no-op claim becomes about the fixture, not the controller. " \
  "If this fails, port G4 into a controller-level test that stubs " \
  "evaluate_panel_and_current to verify the GET path itself never " \
  "writes tiebreak_winner."
```

Alternatively, weaken G4's docstring to describe what it actually proves: "no GET-path code reads `params[:tiebreak_winner]` to advance state under the conditions of the current fixture set."

### WR-02: `teardown` overwrites fixture-state with empty `data: {}` — pollutes other tests in the same suite run

**File:** `test/integration/tiebreak_modal_form_wiring_test.rb:67-77`
**Issue:** The teardown block writes `@tm.update!(data: {})` and `@tm.update_columns(game_id: nil, state: "new", panel_state: "pointer_mode", current_element: "pointer_mode")`. `table_monitors(:one)` is a shared fixture used by other tests; Rails fixtures are loaded *once per test class*, not per-test, but persistent writes here happen against the test database within the transaction. In normal Rails Minitest runs each test is wrapped in a transaction that rolls back, which would make this teardown unnecessary — and if rollback works, the teardown is dead code. If rollback does *not* work (e.g., `self.use_transactional_tests = false` somewhere, or the fixture is loaded outside the transaction), the teardown leaks the *empty* `data: {}` state to subsequent tests, which is worse than leaking the test's own state.

`@game.reload.destroy` likewise destroys a record that was created inside this test — if transactional tests are on, that destroy fires inside the rolled-back transaction (harmless but noisy); if transactional tests are off, the destroy-after-test-fail path can leave orphan rows when an assertion raises before teardown runs.

**Fix:** Remove the teardown entirely if `self.use_transactional_tests` is `true` (the Rails default for `ActionDispatch::IntegrationTest`). If transactional fixtures are intentionally off in this project, document why in a comment and use `ensure` or wrap in `begin/rescue/ensure` so a teardown failure does not mask the original assertion failure.

```ruby
# Verify Rails handles cleanup automatically:
# self.use_transactional_tests is true by default; the entire test
# (including the @game.create! and @tm.update!) is rolled back at the
# end of each test. The teardown block is dead code under normal
# operation. Delete it.
```

If teardown is genuinely needed, at minimum guard against double-destroy and clarify intent:

```ruby
teardown do
  next unless @game_created_by_test
  @game&.reload&.destroy if @game&.persisted?
  # Don't reset @tm — let transaction rollback handle it.
end
```

## Info

### IN-01: G3's button selector is brittle — text match instead of structural

**File:** `test/integration/tiebreak_modal_form_wiring_test.rb:163`
**Issue:** `legacy_button = doc.css("button").find { |b| b.text.include?("Ergebnis bestätigen") }` selects by visible German text. If the label is ever I18n-extracted (the tiebreak path already uses `t("table_monitor.tiebreak.submit_label")` — see line 232 of the view), this assertion silently fails to find the button and the subsequent `assert legacy_button` triggers without diagnostic context. The G2 selector in contrast uses `button[type='submit'][form='tiebreak-form-#{@tm.id}']` — structural, stable.

**Fix:** Select the legacy button by `data-reflex` attribute (which is itself the assertion target) and then assert the attribute value matches:

```ruby
legacy_button = doc.at_css("button[data-reflex='click->GameProtocolReflex#confirm_result'][data-id='#{@tm.id}']")
assert legacy_button,
  "Legacy 'Ergebnis bestätigen' button must render with click-bound reflex"
```

This is structural, locale-independent, and avoids a tautological re-assert of the attribute on the next line.

### IN-02: G4's `sign_in users(:club_admin) if users(:club_admin).present?` is dead-code defense

**File:** `test/integration/tiebreak_modal_form_wiring_test.rb:195`
**Issue:** `users(:club_admin)` is a fixture accessor — it raises `ActiveRecord::FixtureClassNotFound` or returns the record; it never returns `nil` or a falsy value when the fixture exists. The `.present?` guard is dead code. The comment says "If TableMonitor#show requires auth, sign in" — but the code already calls `sign_in` unconditionally (modulo the dead guard).

**Fix:** Either drop the conditional, or use a real existence check:

```ruby
sign_in users(:club_admin)
```

If the intent was "skip if `club_admin` fixture is removed in the future," add a hard assertion instead of silently skipping:

```ruby
assert users(:club_admin), "Test depends on :club_admin fixture"
sign_in users(:club_admin)
```

### IN-03: Setup creates a Game with a hard-coded primary-key collision risk

**File:** `test/integration/tiebreak_modal_form_wiring_test.rb:40-45`
**Issue:** `Game.create!(data: {"tiebreak_required" => true}, group_no: 1, seqno: 1, table_no: 1)` uses literal small integers for `group_no/seqno/table_no`. If the Game model has a uniqueness constraint or index on `[group_no, seqno, table_no]` (common pattern for tournament games), this `create!` will collide with whatever fixture data already exists in `test/fixtures/games.yml` for the same triple. The setup will then raise `ActiveRecord::RecordNotUnique` instead of arriving at the assertions. Currently this works because the fixture set probably has no such collision, but it's a lurking flake.

**Fix:** Use a deterministic-but-isolated factory or the existing `games(:fixture_name)` if a suitable fixture exists. If `Game.create!` is required, set the fields to values guaranteed not to collide (e.g., `seqno: -1` or use the `MIN_ID`-floor pattern from `LocalProtector`):

```ruby
@game = Game.create!(
  data: {"tiebreak_required" => true},
  group_no: 9999, seqno: 9999, table_no: 9999
)
```

Better: define a minimal fixture in `test/fixtures/games.yml` and use `games(:tiebreak_required)`.

### IN-04: Comment block in view (lines 173-181) cites file/git history — risks staleness

**File:** `app/views/table_monitors/_game_protocol_modal.html.erb:173-181`
**Issue:** The ERB comment block is excellent context for the next reader, but it references "Plan 06 — see git history of this file" as the failure exemplar. Once the relevant commits are squashed, rebased, or the repo is mirrored without history, the citation becomes ungrounded. The reader still gets the *what* (form-not-button) and *why* (browsers fire submit on forms), but loses the *evidence anchor*.

```erb
<%# Phase 38.7 Plan 13 (Gap-05): submit-event wiring lives on the FORM
    element where browsers actually fire submit events — NOT on the
    <button> below. Putting data-reflex='submit->...' on a button is a

```

This is purely a documentation hygiene improvement; no functional change.

## Items explicitly checked and OK

- **XSS in ERB (view file):** All interpolations in the new tiebreak block use Rails' default ERB escaping. `history[:player_a][:shortname]`, `history[:player_b][:shortname]`, and `t(...)` are all auto-escaped. No `raw`, `html_safe`, or `<%==`. No `onclick=` interpolation.
- **CSRF:** StimulusReflex uses Action Cable, which authenticates via the existing Devise session — not via form-submission CSRF tokens. The `action="javascript:void(0)"` neutralizes form-submission entirely, so CSRF is moot for this form. The reflex itself enforces server-side validation in `GameProtocolReflex#confirm_result` (whitelist on `playera|playerb`, ref `app/reflexes/game_protocol_reflex.rb:122-127`).
- **Server-side input validation:** `confirm_result` validates `tw = params[:tiebreak_winner].to_s` against `%w[playera playerb]` and refuses to advance state on invalid/missing input — defense-in-depth against the smoking-gun GET. Verified in source.
- **Accessibility:** `<fieldset>` + `<legend>` is correct for grouped radios; `required` on both radios + `cursor-pointer` on the wrapping `<label>` keeps tap-target affordance; the `form="..."` association allows the submit button to live outside the form without losing semantic submission. The submit button retains `type="submit"` so HTML5 `required` validation runs. No accessibility regression.
- **StimulusReflex idiom (`data-reflex` on form):** Confirmed via the project's other reflex bindings — submit-event reflexes belong on form elements, click reflexes on buttons. `data-reflex-permanent` is preserved (it is a CableReady morph hint, not a submit-wiring attribute, so it does not interact with this fix).
- **`element.dataset['id']` resolution in reflex:** `GameProtocolReflex#load_table_monitor` (line 350-352) reads `element.dataset['id']`. The `element` in a submit-bound reflex is the form, and `data-id="<%= table_monitor.id %>"` was relocated onto the form alongside `data-reflex`. Verified contract holds.
- **Integration test choice (Nokogiri on `ApplicationController.render` output):** Sound. Render-based assertions are deterministic, fast, and catch the exact attribute-on-wrong-element class of bug that bypassed both the reflex unit tests (`.allocate + define_singleton_method` skips the DOM) and the system tests (Capybara submit triggers the reflex regardless of where `data-reflex` lives because Selenium fires the event on the form anyway). Plan 13's testing decision is well-justified in the SUMMARY.
- **G2 regression guard:** Correctly asserts the *absence* of `data-reflex` on the button. This is the single most valuable test in the suite — it directly inverts the bug.
