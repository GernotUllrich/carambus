## Decisions Made

- **`tiebreak_pick_pending?` as a private helper, not delegating to `TableMonitor#tiebreak_pending_block?`.** The two helpers implement the same logic but live in different scopes. ResultRecorder uses its private helper for the marker-switch decision (no save); TableMonitor uses its public predicate inside the AASM guard (which fires inside the persisted-state lifecycle). Keeping them separate keeps the public/private split aligned with caller surfaces. Future refactor can extract into a shared module without changing semantics.
- **Test T7-T9 do NOT use `@tm.reload`.** `update_ba_results_with_set_result!` does NOT call `save!` — caller owns persistence (`perform_save_current_set` saves AFTER calling). Assertion against in-memory `@tm.data` is the correct contract test; `@tm.reload` would discard the not-yet-persisted change. Discovered during GREEN phase: initial test version reloaded and got `Actual: nil` even though the in-method computation was correct.
- **Tests T5/T6 (training rematch) stub `end_of_set?`, `acknowledge_result!`, AND `final_set_score?`.** This drives `perform_evaluate_result`'s control flow into Branch C's training-rematch arm without needing a fully-realistic AASM flow + tournament_monitor=nil + sets_to_win=1 + state transitions. The stubs isolate the D-13 decision point — what we want to test — while keeping the test deterministic.
- **AASM guard predicate naming convention.** AASM convention is positive (`guard: :foo?` allows when `foo?` returns true). The semantic blocker is `tiebreak_pending_block?` (true = block). Exposed `tiebreak_not_pending?` as the AASM-facing wrapper to match the convention. Both are public so tests + controllers can introspect either.
- **Marker name `tiebreak_winner_choice`** chosen per CONTEXT.md CD-02 over alternatives like `confirm_result_with_tiebreak` — shorter, noun-form, cleaner Plan 06 view branch.
