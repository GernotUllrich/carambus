# Phase 38.7 Plan 12: TournamentMonitor Startup-Form Tiebreak Override (Gap-04) Summary

**A 'Stechen bei Unentschieden' / 'Tiebreak on draw' checkbox now rides in the TournamentMonitor `_form.html.erb` and persists onto `Tournament.data['tiebreak_on_draw']` (resolver Level 1) on UPDATE only — operators no longer need Administrate or the Rails console to enable tiebreak per-tournament, closing UAT Gap-04. The auth surface is unchanged: :create deliberately ignores the form param because that action is not director-gated.**

## Accomplishments

- **Per-Tournament operator override via UI** for the TournamentMonitor edit form — the form's last form-group before the submit row now carries the new toggle.
- **Sparse-override semantic preserved end-to-end**: hidden `'0'` + checked `'1'` cleanly thread through the controller's persist helper → Tournament.data['tiebreak_on_draw'] → Plan 04's resolver Level 1 read at every Game's start_game.
- **Auth surface preserved**: the existing before_action stack (`ensure_tournament_director` + `ensure_local_server`) is registered with `only: %i[show edit update destroy ...]` — `:create` and `:new` are NOT in that list. ## Decisions Made

- **`update_columns` over `update!` for the data write**: chosen because (a) Tournament's `before_save` mutates `organizer` + extracts keys out of `data` (irrelevant to this single-key write and would cause diff noise), and (b) Tournament's `belongs_to :organizer` presence validation can transiently fail because TournamentMonitor's `after_enter :do_reset_tournament_monitor` callback nils `Tournament.organizer_id` during the request lifecycle. The PaperTrail-skip tradeoff is documented inline; audit trail comes from `Rails.logger.info` and the TournamentMonitor's own PaperTrail entry on the same request.
- **Pass Hash to `update_columns`, NOT pre-stringified JSON**: Tournament has `serialize :data, coder: JSON, type: Hash`. AR JSON-encodes on write. Passing a String raises `ActiveRecord::SerializationTypeMismatch`. The plan-prescribed code suggested `t.update!(data: current)` directly — when I switched to `update_columns` to bypass validations, my first attempt JSON-encoded the Hash defensively and got the SerializationTypeMismatch. Reverted to passing the Hash itself.
- **Auth surface preserved by gating to `:update` only**: persisted via the existing before_action stack on `:update` (which IS director-gated + local-server-gated). `:create` is NOT in the before_action's `only:` list, so persisting from `:create` would let any signed-in user mutate Tournament.data. Operator workflow: create-then-edit.
- **i18n namespace placement**: under existing `tournament_monitors.form:` (sibling of `prohibited_this_tournament_monitor_from_being_saved`) rather than top-level. Disjoint from Plan 10's `locations.scoreboard_free_game.tiebreak_on_draw` — no collision risk, both keys verified to resolve in DE + EN.
- **Plan 04 resolver UNCHANGED**: this plan only writes Tournament.data['tiebreak_on_draw'] which is already Level 1 of Plan 04's existing 3-level resolver. `Game.derive_tiebreak_required` at app/models/game.rb:81-98 is byte-identical to its post-Plan-04 state.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Switched from `update!(data: hash)` to `update_columns(data: hash, updated_at: Time.current)` in `persist_tournament_tiebreak_override`**
- **Found during:** Task 1 GREEN, after running RED tests + initial GREEN failed with `ActiveRecord::RecordInvalid: Organisiert von muss ausgefüllt werden`
- **Issue:** Plan-prescribed `t.update!(data: current)` triggered Tournament's `belongs_to :organizer` presence validation. Investigation revealed TournamentMonitor's `after_enter :do_reset_tournament_monitor` callback (via `TournamentMonitor::TablePopulator#do_reset_tournament_monitor`) had nilled `Tournament.organizer_id` during the request lifecycle. The validation blocked the targeted single-key data write.
- **Fix:** Use `update_columns(data: current, updated_at: Time.current)` — single-column SQL write that skips callbacks AND validations. Acceptable here because (a) we only need to flip one JSON key, and (b) we don't want Tournament's `before_save` (which extracts certain data keys out into columns and re-asserts organizer) to fire on this write. PaperTrail tradeoff documented inline; alternative audit signal is `Rails.logger.info`.
- **Files modified:** `app/controllers/tournament_monitors_controller.rb`
- **Verification:** All 4 Gap-04 tests turn GREEN; G3's idempotent contract still holds; G4's auth gate still holds.
- **Committed in:** `321ff8bf` (folded into GREEN commit)

**2. [Rule 1 - Bug] Pass Hash (not JSON-stringified) to `update_columns`**
- **Found during:** Task 1 GREEN, after Deviation 1 fix; second test run failed with `ActiveRecord::SerializationTypeMismatch (can't dump 'data': was supposed to be a Hash, but was a String — "{\"tiebreak_on_draw\":true}")`
- **Issue:** My first `update_columns` attempt defensively pre-serialized the Hash via `current.to_json` and passed the String. Tournament has `serialize :data, coder: JSON, type: Hash` so AR rejects pre-stringified values — the serializer expects a Hash and JSON-encodes it on write.
- **Fix:** Pass the Hash directly: `t.update_columns(data: current, updated_at: Time.current)`. AR's JSON coder handles the encoding.
- **Files modified:** `app/controllers/tournament_monitors_controller.rb`
- **Verification:** All 4 Gap-04 tests GREEN.
- **Committed in:** `321ff8bf` (folded into GREEN commit)

**3. [Rule 1 - Bug] Lint: replaced `plan && plan.respond_to?(:executor_params)` with `plan&.respond_to?(:executor_params)` (Style/SafeNavigation)**
- **Found during:** Task 1 GREEN, after running standardrb on the changed controller
- **Issue:** standardrb flagged Style/SafeNavigation on the and-respond_to? chain that I introduced. Confirmed via baseline check (`git stash && standardrb`) that this is a NEW violation introduced by my edit.
- **Fix:** Use safe-navigation operator `&.` per standardrb convention. Behavior preserved.
- **Files modified:** `app/controllers/tournament_monitors_controller.rb`
- **Verification:** `bundle exec standardrb` baseline 24 → post-edit 22 (net -2 issues from this plan; no new violations).

- **Found during:** Task 1 GREEN, after running standardrb
