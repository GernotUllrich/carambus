<objective>
Bake the per-game `tiebreak_required` flag at game-start by resolving the 4-level
hierarchy (Tournament.data → TournamentPlan.executor_params['gN'] → Discipline.data
→ default false) and writing the result into `game.data['tiebreak_required']`.

Per CD-01: ship a focused `Game.derive_tiebreak_required` class method (NOT extend
the BkParamResolver). Rationale: 38.7 has 1 flag at 4 levels (different hierarchy
from 38.5's 7-level / multiset). Coupling to BkParamResolver would force the
resolver to know about Tournament/TournamentPlan layers that 38.5 explicitly punted
on (38.5 D-14: tournament path "vorbereitet, leer", no UI). A separate, narrow
helper avoids over-coupling and keeps each resolver simple.

Per D-05: bake happens ONCE at start_game. No runtime re-resolution. If the modal
opens (Plan 06) and `game.data['tiebreak_required']` is missing, that's a sign the
match started before this plan landed — fall back to false (= legacy karambol
behaviour, no modal radio).

Output: `Game.derive_tiebreak_required` class method + a 4-line bake call in
`GameSetup#perform_start_game` + 10+ unit tests + 2 integration tests.

Per CONTEXT.md D-05 specifics §"Flag-Bake bei Game-Start": "wenn das Modal aufgeht
und `game.data['tiebreak_required']` nicht gesetzt ist, gibt es keinen sinnvollen
Run-Time-Resolve. Bake einmal bei start_game."

Per CD-01 (Claude's Discretion): we choose `Game#derive_tiebreak_required` as a
**class method** (not instance — bake happens BEFORE the new game.data is finalized,
and we don't always have a fully-saved Game record at that point). The signature:

```ruby
Game.derive_tiebreak_required(tournament: nil, tournament_plan: nil, group_no: nil, discipline: nil)
```

Returns Bool (true | false). Walks the 4-level hierarchy and stops at the first
level whose `data` (or executor_params) has the `tiebreak_on_draw` key explicitly set.

Per D-13: training matches (tm.tournament_monitor.blank?) have NO Tournament,
NO TournamentPlan. Resolver receives nil for both → falls through to Discipline-only.
This is exactly the BK-2 / BK-2kombi case where Plan 01 set the Discipline default
to `true`, so training matches of those disciplines correctly get tiebreak_required=true.
</problem_framing>

    ```ruby
    # Phase 38.7 Plan 04 — D-04, D-05, CD-01.
    # Resolves the per-game `tiebreak_required` flag from the 4-level hierarchy:
    #   1. Tournament.data['tiebreak_on_draw']                    (highest)
    #   2. TournamentPlan.executor_params['g{group_no}']['tiebreak_on_draw']
    #   3. Discipline.data['tiebreak_on_draw']
    #   4. false                                                   (default)
    #
    # Sparse-override semantics (D-05): each level's data is queried via `data.key?`,
    # NOT `data[].present?`. An explicit `false` at any level overrides `true` at lower
    # levels. A missing key falls through to the next level.
    #
    # Returns Bool. Pure function — no DB writes. Safe to call at start_game (D-05).
    # Training-mode (D-13): tournament=nil, tournament_plan=nil → falls through to
    # discipline default. Plan 01 ensures BK-2 + BK-2kombi disciplines have key=true,
    # so training matches of those disciplines correctly get tiebreak_required=true.
    def self.derive_tiebreak_required(tournament: nil, tournament_plan: nil, group_no: nil, discipline: nil)
      # Level 1: Tournament.data
      t_data = parse_data_hash(tournament&.data)
      return !!t_data["tiebreak_on_draw"] if t_data.key?("tiebreak_on_draw")

      # Level 2: TournamentPlan.executor_params['g{group_no}']
      if tournament_plan && group_no.present?
        plan_params = parse_data_hash(tournament_plan.executor_params)
        group_key = "g#{group_no}"
        if plan_params.is_a?(Hash) && plan_params[group_key].is_a?(Hash) &&
           plan_params[group_key].key?("tiebreak_on_draw")
          return !!plan_params[group_key]["tiebreak_on_draw"]
        end
      end

      # Level 3: Discipline.data
      d_data = parse_data_hash(discipline&.data)
      return !!d_data["tiebreak_on_draw"] if d_data.key?("tiebreak_on_draw")

      # Level 4: default
      false
    end

    # Phase 38.7 Plan 04 helper: tolerant JSON-text-or-Hash parser.
    # Discipline.data, Tournament.data, TournamentPlan.executor_params can be either
    # a Ruby Hash (already-parsed) or a JSON-encoded text column. Guard against both
    # plus corruption (returns empty Hash on parse error — D-05 sparse semantics).
    def self.parse_data_hash(raw)
      return {} if raw.blank?
      return raw if raw.is_a?(Hash)
      JSON.parse(raw.to_s)
    rescue JSON::ParserError
      Rails.logger.warn "[Game.parse_data_hash] JSON.parse failed for: #{raw.to_s.truncate(80)}"
      {}
    end
    ```
