## Decisions Made

- **`tw.to_i` (plan-prescribed) over direct integer match** — the plan's prescribed code uses `case tw.to_i when 1 | 2`. This is mildly more permissive than `case tw when 1 | 2` (numeric string `"1"` would also match), but defensively still rejects `99`, `"X"`, `nil`. T5 exercises both `99` and `"X"` cases to lock the contract; both return empty string.
- **PartyGame.new (no DB persistence) for tests** — the plan suggested `games(:game_one)` but the project has no `games.yml` fixture file, AND the helper actually receives a `PartyGame` (not a `Game`) at the actual call site (`row[:game]` is from `party.party_games.index_by(&:seqno)` in `parties_helper.rb` line 7). PartyGame.new + fixture-loaded Player records is the correct unit-test boundary because the helper does not call `save!` on the game.
- **i18n keys under NEW `parties: → spielbericht:` sub-namespace** — `parties:` in both DE and EN already exists (with `index:`, `parties:`, `party:` keys); we added `spielbericht:` as a sibling key. This matches Plan 06's pattern of nesting new sub-namespaces in the existing tree (`table_monitor:` already had `bk2_kombi:`, then Plan 06 added `tiebreak:`).
- **Helper returns plain Ruby String, NOT html_safe** — Rails's `t` helper auto-escapes `%{role}` interpolations by default. Player#shortname is user-editable from the admin surface, so we MUST NOT bypass auto-escaping. The view's `<%= %>` form preserves the escape contract end-to-end.
- **No partial extraction for the indicator** — the helper output is a single-line String suffix; a Rails partial would add file-system overhead without abstraction value. CONTEXT.md D-12 explicitly granted planner discretion on the helper-vs-partial choice; helper is the simpler path.
