<objective>
Add the tiebreak-priority branch (D-10) to `PartyMonitor::ResultProcessor#update_game_participations`
so that — when `game.data['tiebreak_winner']` is set AND the rank-based comparison
produces a tie — the operator's pick decides which side gets `game_points['win']`
vs `game_points['lost']` instead of both sides receiving `game_points['draw']`.

Purpose: Without this branch, even after the modal lands in Plan 06 and writes
`game.data['tiebreak_winner']='playera'`, league standings would still record the
match as a draw — the tiebreak pick would be displayed in the PDF (Plan 07) but
have NO effect on points/standings. This plan closes the loop on D-10/D-11.

Output: ~10-line branch in `result_processor.rb` + 5 new unit tests in
`result_processor_test.rb`. Independent of Plans 01/02/04/06 — fixture games can
synthesize `game.data['tiebreak_winner']` directly.
</objective>

<problem_framing>
Per CONTEXT.md D-10: when an operator picked a tiebreak winner via the modal (Plan 06),
that pick is persisted in `game.data['tiebreak_winner']`. The points-awarding step in
`update_game_participations` must read that key BEFORE applying the legacy "draw" branch.

The branch must be DEFENSIVE: invalid values (forged form posts, future regressions)
must fall through silently to legacy logic, NOT crash. Tiebreak only applies when the
rank-comparison would have produced a draw — if one side legitimately outranked the
other (impossible after a tiebreak but possible if game.data was hand-edited),
legacy comparison wins.

This plan ships independently in Wave 1 because the unit tests synthesize
`game.data['tiebreak_winner']` directly into a fixture game — no dependency on the
bake (Plan 04) or modal (Plan 06) actually setting that key.

Per `.agents/skills/extend-before-build/SKILL.md`: small guard on existing service,
no new state. Branch is ~10 lines.
</problem_framing>
