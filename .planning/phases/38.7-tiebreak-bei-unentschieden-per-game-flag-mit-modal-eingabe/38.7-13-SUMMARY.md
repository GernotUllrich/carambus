## Decisions Made

- **Render-based test pattern over Capybara DOM submit.** StimulusReflex WebSocket roundtrip is non-deterministic in headless Selenium, and the existing reflex unit tests + system tests both bypass the DOM submit-event entirely (via `.allocate + define_singleton_method`). A Nokogiri parse of `ApplicationController.render` output catches the exact bug (submit-event observability on the form vs. the button) and runs in <1s. This matches the existing render-test pattern used by `render_protocol_modal` in `game_protocol_reflex.rb:360-371`.
- **G4 GET-fallback no-op test as forward invariant.** The action='javascript:void(0)' defense in Task 1 prevents the browser from doing a default GET-form-submission, but a future change could accidentally remove it. G4 locks the controller-side invariant: even if a stray GET reaches the show action with `?tiebreak_winner=playera`, no state advance happens. This closes the smoking-gun audit trail (dev log line 5159819) at the request level.
- **`data-reflex-permanent` attribute preserved.** It's a CableReady morph hint (preserves form state across morphs), unrelated to submit-event wiring. Removing it could regress an unrelated concern. The plan explicitly preserves it; verified via grep in the post-edit view.
- **Submit semantics preserved.** Did NOT swap `submit->` for `click->` on the button per plan constraint — HTML5 `required` radio validation only runs on submit, not on click. Empty radio pick is still blocked client-side via the `required` attribute on each `<input type="radio">`.
- **No reflex / service / controller / Plans 09-12 changes.** The downstream Ruby chain is correct end-to-end (verified in debug session): predicate, auto-detect, reflex handler, deep_merge_data!, AASM guard. Gap-05 is a pure view-wiring bug, and Plan 13 ships a pure view-only fix.
