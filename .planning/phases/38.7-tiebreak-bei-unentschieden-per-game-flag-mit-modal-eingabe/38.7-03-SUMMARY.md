## Decisions Made

- **Test data isolation via stubs over fixture-chain expansion.** The plan suggested using `party_monitors(:pm_one)` / `table_monitors(:tm_one)` / `games(:game_one)` fixture keys. Those keys don't exist in this codebase's fixtures (verified via `grep`). Plan's fallback guidance says "ADAPT to actual fixture keys; if no usable PartyMonitor fixture, mock the chain via OpenStruct or a tiny fixture-free setup". Followed that guidance: existing `create_party_monitor_with_party` helper produces a PartyMonitor without a populated `league.game_plan.data["rows"]` array, which makes the real `get_attribute_by_gname` / `get_game_plan_attribute_by_gname` lookups fail. Stubbing both at the `@pm` instance level is the cleanest path and matches existing patterns in `test/services/region_cc/`.
- **Helper signature `(rank_a:, rank_b:, tiebreak_winner: nil, balls_goal: 100)` instead of full 5-param plan signature.** Plan's helper signature included `sets_to_play: 1` but the helper itself didn't use it (the value lives in the stub). Removed dead parameter. Plan's `ba_results` payload extended with `Ergebnis1/2`, `Aufnahmen1/2`, `Höchstserie1/2` fields because the participation-write loop reads them in the multiset path; even though all 5 tests run with `sets_to_play=1`, defensive coverage avoids a hidden NPE if a follow-up plan flips the stub.
- **TableMonitor#game_id wiring via update_columns + reload.** TableMonitor's `belongs_to :game` association requires a saved game with a valid id before assignment. `tm.update_columns(game_id: game.id) ; tm.reload` is the cleanest path that survives ApiProtector and the AASM initial-state callback (which fires before `before_validation` on `create!`).
- **Audit log severity = info, not warn.** The override is normal-flow when the modal is used; not a fault. Operations can `grep tiebreak-override` to count override events as a KPI; warn-level would imply something went wrong.
- **Defensive guard short-circuit ordering.** `tiebreak_winner.is_a?(String) && %w[playera playerb].include?(tiebreak_winner) && rank["playera"] == rank["playerb"]` evaluates clause 1 first so non-String values bail without ever reaching `Array#include?`. Tests A/B/C/D/E exercise all 3 clauses across the matrix.
