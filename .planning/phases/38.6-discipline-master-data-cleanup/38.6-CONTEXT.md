# Phase 38.6: Discipline Master-Data Cleanup — BK-* Duplikate auf carambus_api - Context

**Gathered:** 2026-04-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Auf der **carambus_api** (Source of Truth für globale Records, `id < 50_000_000`) duplikate `Discipline`-Rows der BK-Familie zusammenführen. Ein **One-shot Ruby-Skript** (`script/merge_bk_disciplines.rb`) führt 6 Loser-Rows in 5 kanonische Winner-Rows zusammen, transferiert dabei alle 17 ActiveRecord-Reflections, und nutzt **ausschließlich PaperTrail-aware AR-Calls** (`update!`/`destroy`), damit `Version`-Records geschrieben werden und die Bereinigung via `Version#update_from_carambus_api` an alle Local-Server (carambus_bcw, carambus_phat, etc.) propagiert.

**Kanonische Namen:** BK-2, BK-2plus, BK50, BK100, BK-2kombi.

**Out-of-scope (für andere Phases / spätere Arbeit):**
- Recompute der `PlayerRanking`-Aggregate (Skript markiert nur betroffene Disciplines, eigentliche Recompute ist eigener Task)
- Andere Discipline-Familien (Karambol, Snooker, Pool, Kegel, Biathlon) bleiben unangetastet
- UI-Änderungen (data-only Cleanup)
- Local-Server-Cleanup (passiert automatisch über Sync-Pfad)

</domain>

<decisions>
## Implementation Decisions

### Merge-Map (Winner-IDs absichtlich gewählt, damit 38.5-Seed-Anker erhalten bleibt)

- **D-01:** Winner-IDs = 107 (BK-2kombi), 108 (BK50), 109 (BK100), 110 (BK-2), 111 (BK-2plus). Diese matchen die Anker, die Phase 38.5-Seed (`script/seed_bk2_disciplines.rb`) bereits gegen referenziert hat — kein erneutes Seed-Rollout nötig.
- **D-02:** id 60 "BK 2-100" ist semantisch identisch mit BK100 (Spezialfall von BK-2 mit `balls_goal=100`) → wird Loser für Winner 109.
- **D-12:** Winner 107 wird beim Merge umbenannt von "BK2-Kombi" → "BK-2kombi" (kanonischer Name) via `update!` (erzeugt Version, propagiert).

**Loser → Winner:**
| Winner-ID | Kanonisch | Loser-IDs |
|-----------|-----------|-----------|
| 107 | BK-2kombi | 57, 59, 95 |
| 108 | BK50 | 61 |
| 109 | BK100 | 60, 62 |
| 110 | BK-2 | — |
| 111 | BK-2plus | — |

### Sync- und PaperTrail-Disziplin

- **D-04:** Skript verwendet ausschließlich PaperTrail-aware AR-Calls: `update!`, `update`, `save!`, `destroy`. **Verboten:** `update_column`, `update_columns`, `update_all`, `delete`, `delete_all`, raw SQL `UPDATE`/`DELETE`. Grund: nur Version-erzeugende Operationen propagieren via `Version#update_from_carambus_api` an Local-Server.
- **D-10:** PaperTrail-`versions`-Rows der Loser bleiben unangetastet. Annahme: alle Local-Server sind sync-aktuell, Audit-History bleibt korrekt am ursprünglichen `item_id`. (Reassign der Versions würde raw SQL erfordern → verletzt D-04.)

### PlayerRanking — FK-Update mit Konflikt-Recompute

- **D-03:** PlayerRanking-Strategie pro Loser-Row:
  1. FK-Update versuchen: `loser_ranking.update!(discipline_id: winner.id)`
  2. Bei `ActiveRecord::RecordNotUnique` (Unique-Constraint auf `(player_id, discipline_id)` verletzt): Loser-Ranking destroyen + Tupel `(player_id, winner_discipline_id)` in eine Recompute-Liste sammeln.
  3. Recompute-Liste am Ende des Skripts ins Markdown-Protokoll schreiben (eigentliche Recompute = nachfolgender separater Task, nicht Teil dieses Skripts).

### `*_cc` Konflikte — manuelle Entscheidung

- **D-08:** `discipline_cc`, `competition_cc`, `branch_cc` sind `has_one` mit `dependent: :destroy`. Wenn sowohl Winner als auch Loser eine CC-Row haben:
  - Skript **HÄLT AN**, druckt Konflikt-Details (winner-CC vs. loser-CC mit allen Feldern), und fragt interaktiv pro Fall: keep-winner / move-loser / merge-fields / abort.
  - Kein Auto-Merge bei singulären CC-Rows, da regional governance metadata kontextabhängig.
- Wenn nur Loser eine CC-Row hat (Winner ist `nil`): direkter FK-Reassign `loser_cc.update!(discipline_id: winner.id)` (vor `loser.destroy`, sonst cascade-delete).

### `super_discipline` self-ref — defensiv im Skript

- **D-09:** Loser können `super_discipline` für andere Rows sein (oder via `super_discipline_id` auf andere Loser zeigen). Strategie:
  - **Pass 1:** Für jeden Loser: alle `sub_disciplines` (children mit `super_discipline_id = loser.id`) auf Winner umhängen via `child.update!(super_discipline_id: winner.id)`.
  - **Pass 2:** Falls Loser selbst `super_discipline_id` auf einen anderen (mittlerweile gemergten) Loser hatte → wird beim Loser-Destroy obsolet.
  - Topologische Reihenfolge nicht zwingend nötig, weil Loser am Ende sowieso destroyed werden.

### Stats-First-Design

- **D-07:** Skript Step-0 = **Stats-Phase** (kein Side-Effect). Für jeden Loser × jede der 17 Reflections wird der Row-Count ermittelt und ausgegeben:
  ```
  Loser 57 (BK 2kombi):
    versions: 23
    discipline_tournament_plans: 0
    sub_disciplines: 0
    tournaments: 4
    player_classes: 0
    player_rankings: 117
    discipline_cc: 0  (nil)
    leagues: 1
    game_plan_ccs: 0
    game_plan_row_ccs: 0
    seeding_plays: 0
    competition_cc: 0  (nil)
    branch_cc: 0  (nil)
    training_concept_disciplines: 0
    [...]
  ```
  Erwartung des Users: **die meisten Reflections haben 0 Refs auf Loser**. Stats-Output informiert (a) ob Konflikt-Handler überhaupt nötig (z.B. `*_cc` immer nil → D-08 nie getriggert), (b) Before/After-Protokoll, (c) Größenordnung der `player_rankings`-Migration.

### Skript-Format und Verification

- **D-05:** Format = One-shot Ruby-Skript `script/merge_bk_disciplines.rb` (matched bestehendes Pattern `script/seed_bk2_disciplines.rb`). Wird via `bin/rails runner script/merge_bk_disciplines.rb` ausgeführt. Erst in `development` (mit api-Snapshot oder dev-DB), dann in `production` auf carambus_api.
- **D-06:** Verification = Before/After-Association-Counts-Protokoll als Markdown nach `tmp/merge-bk-disciplines-{timestamp}.md` (im dev-Run). Format pro Loser: stats-vorher / aktion / stats-nachher / Winner-Counts inkl. Delta. Kein separater Dry-Run-Modus — die Stats-Phase (D-07) IST die Vorschau.
- **D-11:** Idempotenz: Re-Run = sicherer No-Op. Skript prüft initial `Discipline.where(id: [57, 59, 60, 61, 62, 95]).exists?` — wenn keine Loser mehr existieren: log "Cleanup bereits durchgeführt" und exit 0. Kein Error.

### Skript-Strukturierung

- **D-13:** Skript-Phasen (sequentiell, im Code klar getrennt):
  1. **Pre-flight:** Loser-Existenz-Check (D-11), Winner-Existenz-Check (alle 5 müssen vorhanden sein).
  2. **Stats:** D-07 Usage-Matrix erfassen + ins Protokoll schreiben.
  3. **CC-Konflikt-Scan:** Pro Loser × `*_cc`-Reflection prüfen ob Konflikt → falls ja: D-08 interaktive Klärung *vor* dem Merge.
  4. **Merge:** Pro Loser: alle `has_many`-Reflections (außer `versions`) FK-update → `*_cc` reassign falls Winner-leer → `loser.destroy`.
  5. **Rename:** `Discipline.find(107).update!(name: "BK-2kombi")` (D-12).
  6. **Post-flight:** Winner-Counts erfassen + Diff zu Pre-flight ins Protokoll, PlayerRanking-Recompute-Liste serialisieren.

### Claude's Discretion

- Genaue Logging-Verbosity (Rails.logger vs puts vs Markdown-Append-Reihenfolge)
- Exakte Spaltenformatierung der Markdown-Tabellen
- Exception-Handling für unerwartete Reflection-Konflikte (z.B. neue Reflection seit Schema-Änderung)
- Ob Skript Transaction-wrapped (pro Loser oder gesamt) — Empfehlung: pro Loser eine Transaction, um Teil-Erfolge zu isolieren

<code_context>
## Existing Code Insights

### Reusable Assets

- **`script/seed_bk2_disciplines.rb`** — Vorlage für Skript-Struktur (`bin/rails runner`, Klassen-Methoden, klare Phasen). Neues Skript folgt selbem Pattern.
- **PaperTrail-aware AR-API** — `update!`, `destroy`, `save!` sind alle bereits PaperTrail-tracked auf `Discipline` und allen 17 referenzierten Modellen (ApplicationRecord-Default).
- **`Version#update_from_carambus_api`** — bereits vorhandener Sync-Pfad, der Versions auf Local-Server appliziert. Skript muss NUR sicherstellen, dass es ausschließlich Versions-erzeugende Calls verwendet.

### Established Patterns

- **One-shot Ruby-Skript via `bin/rails runner`** — Project-Convention für DB-Cleanups (vgl. `seed_bk2_disciplines.rb`). Keine Rails-Migration nötig (D-05).
- **Stats-vor-Mutation** (D-07) — passt zum bestehenden defensive-style des Projekts; Phase 38.4-08 nutzte ähnliche Pattern (Pre-flight-Checks).
- **`belongs_to ... optional: true`** auf `table_kind` und `super_discipline` — bedeutet: kein NOT-NULL-Constraint, Loser-Destroy lässt diese Felder am Winner unberührt.
- **`dependent: :destroy`** auf 4 Reflections (`discipline_cc`, `competition_cc`, `branch_cc`, `training_concept_disciplines`) — **kritisches Merge-Risiko**: ohne explizite FK-Reassign vor `loser.destroy` werden diese cascade-deleted.

### Integration Points

- **carambus_api Production-DB** — Ziel-Umgebung. Skript wird via `RAILS_ENV=production bin/rails runner script/merge_bk_disciplines.rb` ausgeführt.
- **carambus_bcw/_phat/etc. Local-Server** — kein direkter Eingriff. Sync-Pfad `Version#update_from_carambus_api` propagiert die rename + destroy + FK-updates automatisch.
- **CableReady-Broadcasts** — `ApplicationRecord` includiert `CableReady::Updatable`; Discipline-Updates broadcasten potentiell. Bei großen `player_rankings`-FK-Updates evtl. Broadcast-Storm. Plan-Phase sollte prüfen, ob `paper_trail.without_versioning` oder Skip-Broadcast nötig — ABER D-04 verbietet Versions-Skip. Möglicherweise Broadcast pausieren via `CableReady.broadcaster.disable!` während Skript-Run (zu klären in Plan-Phase).
- **Test-Suite** — Fixtures `test/fixtures/disciplines.yml` müssen nach Merge angepasst werden (Loser-Einträge entfernen). Eigener Plan-Schritt.

<specifics>
## Specific Ideas

- "at first we should make statistics - most associations are not used with discipline_id at all" — User-Observation, die D-07 motiviert. Skript-Architektur prio Stats-Output, dann erst Mutation.
- "PlayerRankings can be recomputed later" — D-03 reduziert Komplexität: keine Merge-Heuristik für Ranking-Werte (highest/newest/sum), nur FK-Update + Recompute-Markierung.
- "Wichtig ist es bei einem Merge die associations mit zu übertragen" — User-Anchor; das Skript-Design hat Reflection-Transfer als ersten-Klassen-Concern (D-13 Schritt 4).
- "assume that all local servers are synchronized" (D-10 Annahme) — vereinfacht das Versions-Handling massiv; keine Audit-History-Reassign-Logik nötig.

</specifics>
