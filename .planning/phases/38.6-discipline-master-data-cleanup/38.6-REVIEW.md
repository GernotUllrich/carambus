# Phase 38.6: Code Review Report

**Reviewed:** 2026-04-29
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found (0 blockers, 4 warnings, 3 info)

## Summary

Phase 38.6 introduces a production data-migration script (`merge_bk_disciplines.rb`) that merges 6 duplicate BK-family `Discipline` rows into 5 canonical winners, plus a dry-run wrapper, fixture/constant renames to the new canonical name `"BK-2kombi"`, and a loser-synonyms preservation block.

The overall design is sound: all DB writes go through PaperTrail-aware AR methods (`update!`, `save!`, `destroy`), FK reassignment precedes `loser.destroy`, super_discipline children are reassigned first, and idempotency via pre-flight exit is correct. No hardcoded credentials, no raw SQL, no cross-checkout path leakage.

Four warnings merit attention before production execution:

1. **`BK_NAME_TO_FORM` stale entry** — `GameSetup::BK_NAME_TO_FORM` still maps `"BK2-Kombi"` (old name) to `"bk2_kombi"`. After the merge renames the winner to `"BK-2kombi"`, the name-based fallback path in `derive_free_game_form` can never match a DB row. The impact is mitigated (priority-1 `data["free_game_form"]` makes priority-2 a dead path for any properly seeded row), but the entry is silently wrong and could confuse future maintainers.

2. **`BkParamResolver.lookup_discipline("bk2_kombi")` returns nil after merge** — `BK_NAME_TO_FORM.invert["bk2_kombi"] == "BK2-Kombi"`, so `Discipline.where(name: "BK2-Kombi")` finds nothing after the rename. For BK-2kombi games, `compute_effective_discipline` returns `"bk_2plus"` / `"bk_2"` (the component forms), so this code path is never hit at runtime — but the lookup silently fails if it ever is.

3. **`rescue ActiveRecord::Rollback` outside the transaction** — In `merge_loser_into_winner`, the `rescue ActiveRecord::Rollback` clause at the method level (outside the `Discipline.transaction do … end` block) will never fire: `ActiveRecord::Rollback` is swallowed silently by the transaction block itself and never propagates to the enclosing method. The rescue block is dead code and the protocol entry it would write is never produced.

4. **`exit 0` inside the dry-run transaction wrapper skips post-rollback verification** — When pre-flight detects all losers already gone and calls `exit 0`, execution leaves the `ActiveRecord::Base.transaction` block without hitting `raise ActiveRecord::Rollback`. The DB is already empty of losers (no real mutation occurred), so no data risk, but the post-rollback verification block is skipped and no "ROLLBACK VERIFIED" message is printed.

## Warnings

### WR-01: `BK_NAME_TO_FORM` entry for `"BK2-Kombi"` is stale after the rename

**File:** `app/services/table_monitor/game_setup.rb:243`

**Issue:** After the merge script renames winner-107 from `"BK2-Kombi"` to `"BK-2kombi"` (Phase 5), `BK_NAME_TO_FORM["BK2-Kombi"]` is unreachable for any live DB row: `derive_free_game_form` priority-2 checks `BK_NAME_TO_FORM[discipline.name]`, but no Discipline with `name == "BK2-Kombi"` will exist post-merge. The fallback map also becomes wrong for `BkParamResolver.lookup_discipline`:

```ruby
# bk_param_resolver.rb:127
bk_name = TableMonitor::GameSetup::BK_NAME_TO_FORM.invert[free_game_form]
# "bk2_kombi" → "BK2-Kombi"  ← wrong; DB has "BK-2kombi" now
Discipline.where(name: bk_name)  # → empty
```

Runtime impact: mitigated because `derive_free_game_form` priority-1 (`data["free_game_form"]`) fires first for every properly-seeded row, and `BkParamResolver` resolves BK-2kombi via its component disciplines (`"bk_2plus"` / `"bk_2"`), not via `"bk2_kombi"` directly. However, the map entry silently misidentifies the canonical name, and local servers that never had `data` set (legacy gap) would get `nil` from the fallback where they previously got a match.

**Fix:**
```ruby
# app/services/table_monitor/game_setup.rb — BK_NAME_TO_FORM
BK_NAME_TO_FORM = {
  "BK-2kombi" => "bk2_kombi",   # D-12: renamed from "BK2-Kombi" in phase 38.6
  "BK2-Kombi" => "bk2_kombi",   # legacy alias — local servers pre-sync still carry this name
  "BK50"      => "bk50",
  "BK100"     => "bk100",
  "BK-2"      => "bk_2",
  "BK-2plus"  => "bk_2plus"
}.freeze
```

Or replace the stale key in one step:
```ruby
"BK-2kombi" => "bk2_kombi",
```

Keep the old key as an alias until all local servers have synced, then drop it.

### WR-02: `rescue ActiveRecord::Rollback` at method level in `merge_loser_into_winner` is dead code

**File:** `script/merge_bk_disciplines.rb:353-356`

**Issue:** `ActiveRecord::Rollback` is a sentinel exception that `ActiveRecord::Base.transaction` swallows internally and never re-raises. The `rescue` at line 353 is therefore unreachable — it lives outside the `end` that closes the transaction block (line 352). The protocol entry and the `raise` inside the rescue block are never executed.

```ruby
  Discipline.transaction do
    # ... all the work ...
  end                               # line 352 — Rollback swallowed here
rescue ActiveRecord::Rollback => e  # line 353 — NEVER reached
  puts "ROLLBACK on loser #{loser.id}: #{e.message}"
  protocol_append(...)
  raise
end
```

This is dead code, not a data-safety risk (the transaction still rolls back on `Rollback`). But it creates a false confidence that rollbacks are being logged.

**Fix:** Remove the dead rescue, or change it to rescue `StandardError` / `ActiveRecord::RecordInvalid` to catch actual failures that would propagate out of the transaction:

```ruby
  Discipline.transaction do
    # ... all the work ...
  end
rescue ActiveRecord::RecordInvalid, ActiveRecord::RecordNotSaved => e
  puts "ERROR on loser #{loser.id}: #{e.message}"
  protocol_append("ERROR on loser id=#{loser.id}: #{e.class}: #{e.message}")
  raise
end
```

### WR-03: `exit 0` inside the dry-run transaction wrapper bypasses post-rollback verification

**File:** `script/merge_bk_disciplines_dry_run.rb:36-49` combined with `script/merge_bk_disciplines.rb:106`

**Issue:** When all loser IDs are absent (idempotent re-run), `pre_flight!` calls `exit 0` from inside the dry-run's `ActiveRecord::Base.transaction` block. Ruby's `exit` unwinds the stack without triggering `raise ActiveRecord::Rollback` — execution never reaches the post-rollback verification. The console displays no "ROLLBACK VERIFIED" or "WARNING: Rollback verification FAILED" message.

No data risk (the DB was already clean), but the dry-run loses its own invariant assertion.

**Fix:** Guard against this in the dry-run wrapper:

```ruby
begin
  ActiveRecord::Base.transaction do
    load Rails.root.join("script", "merge_bk_disciplines.rb")
    raise ActiveRecord::Rollback
  end
rescue SystemExit
  puts "DRY-RUN: pre_flight exit 0 detected — DB already clean, no rollback needed."
end
```

Or: intercept the `exit 0` in `pre_flight!` with a custom exception instead of `Kernel#exit`:

```ruby
raise "ALREADY_DONE" if existing_losers.empty?
```

### WR-04: Two tests in `discipline_test.rb` exercise the old `"BK2-Kombi"` name and now test a historical artifact, not the current canonical record

**File:** `test/models/discipline_test.rb:201,231`

**Issue:** `T-P5-seed-only-bk2kombi-has-flag` (line 196-218) and `T-P5-bk2kombi-keeps-nachstoss` (line 226-247) both call `Discipline.find_or_initialize_by(name: "BK2-Kombi")`. After the rename, no fixture or DB record has `name == "BK2-Kombi"`, so `find_or_initialize_by` always initializes a new (transient) record. The tests still pass — they write to a fresh record via `save!(validate: false)` — but they are no longer exercising the seed's backfill logic for the canonical record (now at `name: "BK-2kombi"`). A future developer looking at these tests will see `"BK2-Kombi"` and think it is still the canonical name.

**Fix:** Update both tests to use `"BK-2kombi"`:

```ruby
# line 201
"BK-2kombi" => {"free_game_form" => "bk2_kombi", "ballziel_choices" => [50, 60, 70], "nachstoss_allowed" => true}
# expected_flag line 208
"BK-2kombi" => true

# line 231
bk2 = Discipline.find_or_initialize_by(name: "BK-2kombi")
```

## Info

### IN-01: `RECOMPUTE_LIST` top-level constant is mutable (not frozen)

**File:** `script/merge_bk_disciplines.rb:252`

**Issue:** `RECOMPUTE_LIST = []` is an unfrozen mutable constant. In a Rails runner context this is harmless (single-threaded, single execution), but it is inconsistent with the project's frozen-constant convention and could cause Ruby to emit a constant-mutation warning on future Ruby versions.

**Fix:**
```ruby
RECOMPUTE_LIST = [].freeze
```
And use `RECOMPUTE_LIST.dup` or a local variable if the array needs to be mutated — or simply convert to a local variable inside the entry point since it is only appended to and read in `post_flight!`.

### IN-02: Dry-run script does not handle the case where CC conflicts arise during the dry-run (interactive `$stdin.gets` would block)

**File:** `script/merge_bk_disciplines_dry_run.rb`

**Issue:** If CC conflicts exist in the dev DB, `scan_cc_conflicts!` will call `$stdin.gets` and wait for interactive input. The dry-run's documentation does not warn about this. In CI or non-interactive terminals this will hang indefinitely.

The prod DB pre-flight analysis (38.6-03 dry-run output) showed zero conflicts, so this is not an active risk for this specific execution, but it is a reliability gap for future re-use of the dry-run wrapper.

**Fix:** Add a comment in the dry-run script header noting that CC conflicts will trigger an interactive prompt, or redirect `$stdin` to `/dev/null` in the wrapper (which would cause `choice` to be `nil` and raise "Invalid choice nil").

### IN-03: 26 occurrences of `"BK2-Kombi"` remain as POST-param values in `table_monitors_controller_test.rb` (tests g–o)

**File:** `test/controllers/table_monitors_controller_test.rb:230,250,269,293,348,372,404,429,454,479,521,615` (and others)

**Issue:** Tests g through the end of the file still POST `discipline_a: "BK2-Kombi"`. These tests pass because the controller's CLAMP replaces any non-canonical value with `BK2_DISCIPLINE_MAP.first` (`"BK-2kombi"`). The tests are not broken, but they document pre-rename behavior. If the CLAMP is ever removed or the fallback logic changes, the tests would silently start testing the wrong discipline name.

This was documented as a "Rule 1 deviation" — the executor chose to leave these unchanged because the CLAMP makes them exercise valid server-side defense. That reasoning is sound. No test reliability issue exists today. Flagging as info-level for future maintainability.

**Fix (optional, low priority):** Update POST params in tests g–o to use `"BK-2kombi"` to match the canonical DB name, so tests exercise the happy path rather than relying on the CLAMP.
