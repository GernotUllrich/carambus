## Gaps Closed

- **O2** — BK50 and BK100 (and all 5 BK-* disciplines with `nachstoss_allowed:true`) now defer set-close when the leader reaches `balls_goal`. State transitions to `nachstoss_pending=true`, `nachstoss_for=trailing`, `player_at_table=trailing`. The trailing player's CommitInning call resolves the set via the Nachstoß resolution branch.
- **O4** — Off-by-one root cause identified and fixed. The "kommt nur bis 49" observation was entirely caused by O2: the set closed immediately when the leader reached 50, making any subsequent CommitInning for the trailing player a no-op (the bk2_state was already marked finished). With the O2 deferral in place, the trailing player's inning is processed with no cap — they can reach exactly `balls_goal`.

## O4 Root Cause Trace

The off-by-one cap was NOT a separate increment guard. There is no `< balls_goal` cap anywhere in `apply_additive_rule` or `apply_opponent_credit_rule`. The root cause was:

1. CommitInning(playera, 50) → `apply_additive_rule` adds 50 → `close_set_if_reached!` fires → set closes immediately → `set_finished_1=true`, `sets_won.playera=1`, set advances to set 2.
2. CommitInning(playerb, N) → `apply_inning!` adds N to set 1 score (no guard) → `close_set_if_reached!` fires for set 1 again → `a >= target` is still true (playera has 50) → closes set 1 AGAIN or is in wrong set state → trailing player's score change was committed to set 1 but the set had already advanced to set 2.

After O2 fix: step 1 defers close (nachstoss_pending=true). Step 2 runs `apply_inning!` correctly, then `close_set_if_reached!` sees `nachstoss_pending=true` → enters resolution branch → correctly credits trailing's score and closes set with appropriate winner.

## Nachstoß State Machine

```
Leader reaches balls_goal (nachstoss_pending not set):
  IF discipline_nachstoss_allowed?:
    state["nachstoss_pending"] = true
    state["nachstoss_for"] = trailing
    state["player_at_table"] = trailing
    RETURN  (do NOT close set)
  ELSE:
    Close set immediately (legacy behavior)

Trailing commits equalizing inning (nachstoss_pending = true):
  apply_inning! adds trailing's score
  close_set_if_reached! called:
    trailing_score >= target → trailing wins (provisional rule)
    trailing_score < target  → original leader wins
  state["nachstoss_pending"] = false
  state["nachstoss_resolved"] = true
  advance_to_next_set!()
```

## Provisional Rule for Equal Case

When the Nachstossende reaches exactly `balls_goal`, this plan implements: **trailing player wins the set**. Rationale: the equalizing-inning rule's fairness intent is that the trailing player is given a chance to match — if they succeed, they demonstrated equal performance with the advantage of going second (knowing the target).

This differs from "Verlängerungssatz" (extension set). The plan notes this as a provisional interpretation. **Landessportwart sign-off required** before the 2026-05-02 tournament. If the correct rule is Verlängerungssatz on equal, the resolution logic is isolated in `close_set_if_reached!`'s Nachstoß branch (`nachstoss_score >= target` condition) and can be changed in a single line without re-wiring the state machine.

## Existing T-BK50 / T-BK100 Tests — No Breaking Change

The existing `T-BK50` and `T-BK100` tests in `bk2_scoreboard_test.rb` use `create_bk_family_table_monitor` which creates a `TableMonitor` **without a discipline association** (`@tm.discipline` returns nil or a String). Since `discipline_nachstoss_allowed?` checks `@tm.discipline.respond_to?(:nachstoss_allowed?)` first, it returns `false` for these tests — they continue to exercise the immediate-close (legacy/no-flag) path. **No changes needed to T-BK50 / T-BK100.**

This is correct behavior: those tests are regression guards for disciplines that don't have `nachstoss_allowed:true` in their data.

## define_singleton_method Shortcut

`build_table_monitor_with_discipline` in `advance_match_state_test.rb` uses `define_singleton_method(:discipline) { discipline }` to wire a Discipline to a TableMonitor without going through the full `game → discipline` association chain. This is an acceptable unit-test shortcut (also used in existing 38.4-06 tests in the same file, see `define_singleton_method(:tournament_monitor)`). For long-term maintainability, a fixture-based approach (creating a Game with discipline association) would be more robust, but adds significant fixture complexity. The shortcut is safe because `LocalProtector` is disabled in tests and the singleton method is scoped to the test instance.
