**Note on dz_max=2 / sp_max=5 / nachstoss flag in the button payload:**
Quick-game buttons in `_quick_game_buttons.html.erb` (Plan 38.4-04) emit
`quick_game_form='bk2_kombi'` + balls_goal + sets_to_win/sets_to_play, but they do
NOT emit nested `bk2_options[direkter_zweikampf_max_shots_per_turn]` or
`bk2_options[serienspiel_max_innings_per_set]` hidden fields. The defaults are
applied in `clamp_bk_family_params!` (controller line 432-433 — `dz_max = 2 unless
(1..99).cover?(dz_max)` and `sp_max = 5 unless (1..99).cover?(sp_max)`). When
quick-game submits with no bk2_options, both defaults fire, giving the user's
requested 2/5 config. Plan 10 reinforces dz_max=2 unconditionally for BK-2plus +
BK2-Kombi. Therefore the button payload does NOT need explicit dz_max/sp_max fields —
the server-side clamps handle it.

The "nachstoss" flag in the user's request ("Nachstoß enabled") corresponds to
Plan 11's `discipline.data['nachstoss_allowed']=true` on the central BK2-Kombi
record (id 107). After Plan 11 ships, every BK-2kombi game (regardless of which
button started it) honors Nachstoß. So the button payload doesn't need to encode
the Nachstoß flag either — it's a discipline-level setting, not a per-game override.
