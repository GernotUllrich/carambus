## Summary

1. **I8 fallback partial is an inline block inside `app/views/table_monitors/_show.html.erb` lines 101-106** — not a separate partial. The `bk2_state_uninitialized?` div renders a warning banner with title + body text but zero action buttons. The idiomatic Rails 7.2 delete pattern is `button_to @table_monitor, method: :delete, form: { data: { turbo_confirm: t("are_you_sure") } }` — this pattern is confirmed at `app/views/club_locations/edit.html.erb:10`. The `destroy` action exists at line 364-365, is protected only by `block_tournament_manipulation` (blocks if `tournament_monitor_id.present?`; free-play TableMonitors pass through). `TableMonitor` uses `ApiProtector` (not `LocalProtector`), and the `ApiProtector#disallow_saving_local_records` check only blocks on the API server side, so BCW (local server) can freely destroy its own TableMonitors.

2. **Sync bug (D-05) is exactly as documented**: `version.rb:328-331` and `337-339` both call `YAML.load(args["data"])`, converting a JSON string like `{"free_game_form":"bk2_kombi"}` into a Ruby Hash. The subsequent `update_columns(args)` or `save!` then rejects the Hash for the `text` column. `test/models/version_test.rb` does not exist — no existing test coverage. Minimum viable fix: detect `args["data"].start_with?("{", "[")` and use `JSON.parse` instead of `YAML.load` on those two lines (lines 330/331 and 338/339 in the update and create paths respectively).

3. **`Bk2Kombi::` namespace rename touches 5 production files + 5 test files** across 4 categories: reflex (1), services (3), test/services (3), test/integration (1), test/system (1). The total `Bk2Kombi::` grep count is ~80 occurrences. The only module-level constant referenced externally is `Bk2Kombi::ScoreShot::DEFAULT_RULES` — only referenced in `test/services/bk2_kombi/score_shot_test.rb` (production code never references it externally). Hard rename with no aliases is safe.

4. **`bk2_options[:set_target_points]` is deeply embedded** across the controller (2 CLAMPs), `AdvanceMatchState#derive_set_target` (line 102), `AdvanceMatchState#init_state_if_missing!` (populates `state["set_target_points"]`), `_player_score_panel.html.erb` (reads `bk2_state["set_target_points"]`), and the detail view form (hidden input `name='set_target_points'`). The I9 bug is that this entire path yields `0` when Alpine emits an empty string → `to_i`. D-06 retires this and wires `balls_goal` (from `tournament_monitor.balls_goal`) as the per-set target. Migration strategy matters: for in-flight BK2-Kombi tournaments on BCW, `bk2_state["set_target_points"]` will already be set at 50/60/70 from the prior `initialize_bk2_state!` call — so in-flight games are fine. The backfill concern is about the *next* `start_game` call on a TableMonitor that has `bk2_options["set_target_points"]` set but `game.balls_goal = 0` — that transition must be handled in `GameSetup` by reading `bk2_options["set_target_points"]` and writing it to `tournament_monitor.balls_goal` once.

5. **The detail-view Alpine x-data scope is a single `<div x-data="{...}>` wrapper at line 184** that contains `bk2_choice`, `bk2_dz_max_shots`, `bk2_sp_max_innings`, and all hidden inputs. This is the GAP-02 scope-lift fix from 38.2-02 that MUST NOT regress. The 4 new BK-* discipline branches must be added inside this same x-data scope or as additional Alpine state slots within it.

### Focus Area 2: version.rb sync bug (D-05)

**File:** `app/models/version.rb`, lines 328-339

Two affected code paths:

**Path A — update existing record (obj.present?, obj.valid? false):**
```ruby
# lines 328-332
else
  args = YAML.load(h["object"])
  args["data"] = YAML.load(args["data"]) if args["data"].present?      # line 330
  args["remarks"] = YAML.load(args["remarks"]) if args["remarks"].present?  # line 331
end
obj.update_columns(args)  # line 333
```

**Path B — create new record (obj.blank?):**
```ruby
# lines 337-339
args = YAML.load(h["object"])
args["data"] = YAML.load(args["data"]) if args["data"].present?         # line 338
args["remarks"] = YAML.load(args["remarks"]) if args["remarks"].present?    # line 339
```

**Root cause:** `YAML.load('{"free_game_form":"bk2_kombi"}')` returns the Ruby Hash `{"free_game_form"=>"bk2_kombi"}` (valid JSON = valid YAML flow syntax). The `disciplines.data` column is a plain `text` column (schema line 228 confirms: `t.text "data"`). `update_columns` rejects a Hash value for a text column with `can't cast Hash`. The error is swallowed at line 348 (`rescue StandardError`).

**Fix surface:** Lines 330-331 and 338-339 (4 lines across 2 paths). The `remarks` column on Discipline doesn't exist but the code defensively handles it anyway.

**Minimum viable fix (2 approaches):**

Option A — JSON-sniff before YAML.load:
```ruby
def self.safe_parse(raw)
  return raw if raw.blank?
  raw.start_with?("{", "[") ? JSON.parse(raw) : YAML.load(raw)
rescue JSON::ParserError, Psych::Exception
  raw
end
# Then use: args["data"] = safe_parse(args["data"]) if args["data"].present?
```

Option B — Check if the attribute is a text column (not a serialized column):
Query `classz.columns_hash["data"]&.type == :text` and skip re-parse if so.

Option A is simpler and safer. The question of "why does `obj.valid?` return false for Discipline 107" is explicitly out of scope per the todo file — don't chase it.

**Test coverage:** `test/models/version_test.rb` does NOT exist. Wave 1 Plan for this fix MUST include a regression test: write a Discipline with JSON text in `data`, invoke `Version#update_from_carambus_api`, assert `data` lands as the original JSON string (not a Hash cast failure).

## Open Questions / Risks

1. **`free_game_form` values for BK50/BK100/BK-2/BK-2plus**: The CONTEXT.md says `bk50`, `bk100`, `bk_2`, `bk_2plus`. These become the canonical string values used throughout. The underscore in `bk_2` is important (not `bk-2`) to avoid Ruby/YAML issues with hyphens in hash keys. Planner must lock these as frozen constants in the new `Bk2::` module.

2. **`result_recorder.rb` dispatch (line 60-67):** Currently calls `Bk2Kombi::AdvanceMatchState.call` for the `bk2_kombi` free_game_form. With I7/D-11, the primary scoring path is `Bk2::CommitInning` (called from the reflex). The `result_recorder.rb` path is the *tournament-game scoring path* (different from the live scoreboard path). This may need to dispatch to `Bk2::CommitInning` for all 5 BK-* forms, or it may need to remain as `Bk2::AdvanceMatchState` for the event-based tournament path. Planner must clarify which path `result_recorder.rb` serves and whether D-11 applies to it.

3. **`close_set_if_reached!` in `AdvanceMatchState`** reads `state["set_target_points"].to_i` (line 157). After D-06, this in-state key must be renamed to `balls_goal` OR `init_state_if_missing!` must populate it from `game.balls_goal` / `tournament_monitor.balls_goal` instead of from `bk2_options["set_target_points"]`. Both paths work; the planner must pick one. Recommendation: populate `state["balls_goal"]` in `init_state_if_missing!` from the tournament_monitor field; rename the key throughout `bk2_state` structure.

4. **New Discipline records in carambus_api:** Planner must specify which carambus_api admin action creates the 4 records (e.g., `bin/rails console` write with `unprotected=true`). The CONTEXT.md says "central Discipline records via carambus_api" but doesn't specify the creation mechanism. This is likely a seed script or console one-liner — not a migration.

5. **`is_bk2` vs `bk2_discipline?` predicate proliferation:** Currently `is_bk2 = options[:free_game_form] == "bk2_kombi"` appears in 3 view files and the reflex. After I1, this needs to be a predicate over 5 values. A Ruby helper method `bk2_discipline?(free_game_form)` and a view helper `is_bk_family` would centralize this. The planner should add this to `TableMonitorsHelper` or as a model method.

6. **Discipline `type` and `table_kind_id` for new records:** The `disciplines` schema has `type: string` (STI) and `table_kind_id: integer`. The new BK-* records must match Discipline.find(107)'s values for these columns.
