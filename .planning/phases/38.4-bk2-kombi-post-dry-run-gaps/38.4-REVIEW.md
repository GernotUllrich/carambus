## Critical Issues

### CR-01: NameError in `Version.local_from_api` — calls undefined method `local_server` (UNCHANGED)

**File:** `app/models/version.rb:434`
**Issue:** `Version.local_from_api` calls `local_server` (no `?` suffix). The
defined class method is `local_server?` at line 429. Any caller of
`Version.local_from_api` will crash with `NameError: undefined local variable
or method 'local_server'`. Pre-existing; not addressed by 38.4-08/09.

**Fix:**
```ruby
def self.local_from_api
  Version.sequence_reset if local_server?
end
```

### WR-02: `ResultRecorder#perform_save_result` writes stale `result` field as `Ergebnis1`/`Ergebnis2` (UNCHANGED)

**File:** `app/services/table_monitor/result_recorder.rb:57-77`
**Issue:** For all 5 BK-* free_game_forms the dispatcher delegates set-state
mutation to `Bk2::AdvanceMatchState` (line 67-70), but the
`ergebnis1`/`ergebnis2` values written into `data["sets"]` / `ba_results`
are still read from `@tm.data["playera"]["result"]` /
`@tm.data["playerb"]["result"]` (lines 57-58), NOT from the freshly-mutated
`bk2_state["set_scores"][current_set]`. For BK-* games the `result` field is
never updated by the BK2 commit/advance pipeline, so the saved
tournament-protocol `Ergebnis` will be `0` or stale.

**Fix:**
```ruby
if Discipline::BK2_FREE_GAME_FORMS.include?(@tm.data["free_game_form"])
  Bk2::AdvanceMatchState.call(
    table_monitor: @tm,
    shot_payload: @tm.data.fetch("current_bk2_shot_payload", {})
  )
  set_no = @tm.data.dig("bk2_state", "current_set_number").to_s
  set_scores = @tm.data.dig("bk2_state", "set_scores", set_no) || {}
  ergebnis1 = set_scores["playera"].to_i
  ergebnis2 = set_scores["playerb"].to_i
elsif @tm.data["free_game_form"] == "snooker"
  # ... existing snooker code
end
```

### WR-NEW-01: `clamp_bk_family_params!` reads `params[:bk2_options]` directly while caller works on a `.to_h`-converted copy

**File:** `app/controllers/table_monitors_controller.rb:429`
**Issue:** After 38.4-08, `start_game` builds a plain `Hash` (`p`) from
`params.permit(params.keys).to_h` (line 72). The clamp helper reads
`p[:discipline_a]` / `p[:balls_goal]` from that Hash, but for the nested
`bk2_options` it reaches back to the *raw* `params` object:

```ruby
submitted_bk2_opts = params[:bk2_options] || {}
```

This works correctly today (CLAMP intent: validate the nested values directly
from the raw params before they enter `p[:bk2_options]`), but it creates
two distinct sources of truth in one method:

- **Mutated/converted state:** `p` (passed in as argument, plain Hash)
- **Raw input:** `params` (closed over from controller scope)

A future refactor that moves `clamp_bk_family_params!` to a service object
(see IN-03) will likely break unless the raw `params[:bk2_options]` is
explicitly forwarded. The defensive intent (don't trust the slice list) is
sound; the implicit closure over controller `params` is the smell.

**Fix:** Pass the bk2_options hash explicitly so the helper has a single
source of truth and can be moved to a service:
```ruby
clamp_bk_family_params!(p, bk2_options: params[:bk2_options].to_h)
# inside:
def clamp_bk_family_params!(p, bk2_options: {})
  ...
  dz_max = bk2_options[:direkter_zweikampf_max_shots_per_turn].to_i
  sp_max = bk2_options[:serienspiel_max_innings_per_set].to_i
  ...
end
```

**File:** `app/controllers/table_monitors_controller.rb:66-284`
