# Phase 38.4: Code Review Report

**Reviewed:** 2026-04-25T11:00:00Z
**Depth:** standard
**Files Reviewed:** 30

## Summary

Reviewed 30 files implementing the Phase 38.4 BK2-Kombi post-dry-run gap closure
(I1–I9): expansion of the BK-* discipline family from one (BK2-Kombi) to five
(BK2-Kombi, BK50, BK100, BK-2, BK-2plus), the per-set `balls_goal` migration
(D-06), and the unified `Bk2::CommitInning` dispatcher.

Overall the change-set is well-tested (advance_match_state_test, commit_inning_test,
score_shot_test, bk2_scoreboard_test all exercise the new dispatch paths) and
applies the project's `LocalProtector` / clamp-and-default security posture
consistently. CLAMP logic in `TableMonitorsController#clamp_bk_family_params!`
is implemented correctly (whitelist + range checks), and the i18n keys are
present in both DE and EN.

The most serious finding is a **NameError-level bug in `Version.local_from_api`**
(missing `?` suffix — calls `local_server` instead of `local_server?`). It is
pre-existing, but reachable code, so it should be fixed in this phase since the
file was touched. Several BK-2 family migration loose ends (player_score_panel
still reads the old `set_target_points` key, ResultRecorder writes Ergebnis from
the stale `result` field for BK-* games, BK family list duplicated in a view)
are flagged as warnings.

## Critical Issues

### CR-01: NameError in `Version.local_from_api` — calls undefined method `local_server`

**File:** `app/models/version.rb:434`
**Issue:** `Version.local_from_api` calls `local_server` (no `?` suffix). The
defined class method is `local_server?` at line 429. Ruby will resolve
`local_server` first as a local variable (unbound, raises `NameError`) or fall
through to `method_missing`. There is no local variable in scope, so any caller
of `Version.local_from_api` will crash with `NameError: undefined local variable
or method 'local_server'`.

This is a latent bug, but `local_from_api` is invoked from external scripts /
console, and the file is in scope for this phase's review.

**Fix:**
```ruby
def self.local_from_api
  Version.sequence_reset if local_server?
end
```

## Warnings

### WR-01: `_player_score_panel.html.erb` still reads legacy `set_target_points` for BK-*, missing D-06 `balls_goal` migration

**File:** `app/views/table_monitors/_player_score_panel.html.erb:67`
**Issue:** Phase 38.4 D-06 migrated all server-side state and tests from
`set_target_points` to `balls_goal`. `Bk2::AdvanceMatchState#derive_balls_goal`
reads `tournament_monitor.balls_goal` first, with `bk2_options[set_target_points]`
as transitional fallback, and writes BOTH keys to `bk2_state`. All the new
service / commit_inning code paths read `state["balls_goal"]` first, then fall
back to `state["set_target_points"]`. The view is the only reader still using
the legacy-only key:

```erb
bk2_set_target = bk2_state["set_target_points"].to_i
```

For freshly-started games this still works (init writes both keys), but as soon
as the planned cleanup drops `set_target_points` from `init_state_if_missing!`,
the player panel goal display will silently regress to `0` and render
`Ziel: ?` (see line 128 / 147 fallback).

**Fix:**
```erb
bk2_set_target = bk2_state["balls_goal"].to_i
bk2_set_target = bk2_state["set_target_points"].to_i if bk2_set_target.zero?
```

### WR-02: `ResultRecorder#perform_save_result` writes stale `result` field as `Ergebnis1`/`Ergebnis2` for BK-* games

**File:** `app/services/table_monitor/result_recorder.rb:57-77`
**Issue:** For all 5 BK-* free_game_forms, the dispatcher delegates set-state
mutation to `Bk2::AdvanceMatchState` (line 67-70), but the
`ergebnis1`/`ergebnis2` values that get written into `data["sets"]` /
`ba_results` are read from `@tm.data["playera"]["result"]` /
`@tm.data["playerb"]["result"]` (lines 57-58), NOT from the freshly-mutated
`bk2_state["set_scores"][current_set]`. For BK-* games the `result` field is
never updated by the BK2 commit/advance pipeline (only by Karambol
`update_innings_history`), so the saved tournament-protocol `Ergebnis` will
likely be `0` or whatever stale value was last written.

The Snooker branch correctly recomputes from `innings_list`+`innings_redo_list`
(lines 74-75); the BK-* branch needs an analogous override.

**Fix:**
```ruby
if Discipline::BK2_FREE_GAME_FORMS.include?(@tm.data["free_game_form"])
  Bk2::AdvanceMatchState.call(
    table_monitor: @tm,
    shot_payload: @tm.data.fetch("current_bk2_shot_payload", {})
  )
  # Read the just-committed set scores from bk2_state
  set_no = @tm.data.dig("bk2_state", "current_set_number").to_s
  set_scores = @tm.data.dig("bk2_state", "set_scores", set_no) || {}
  ergebnis1 = set_scores["playera"].to_i
  ergebnis2 = set_scores["playerb"].to_i
elsif @tm.data["free_game_form"] == "snooker"
  # ... existing snooker code
end
```

### WR-03: `Version.update_from_carambus_api` uses `eval` on remote-API-supplied data (RCE risk)

**File:** `app/models/version.rb:291,358`
**Issue:** Both create and update branches do
`args["data"] = eval(args["data"])` for `PartyCc` records when `args["data"]` is
a String. The string comes from the JSON payload returned by the
`/versions/get_updates` endpoint of the remote `carambus_api`. If the upstream
API is compromised (or replaced via DNS/MITM in a non-HTTPS deployment), this
gives the attacker arbitrary Ruby code execution on every local sync.

This is pre-existing behaviour (not introduced by Phase 38.4), but the file is
in scope. Note the surrounding `# TODO: what's going on here?` comment — this
has been flagged before. The `safe_parse_for_text_column` introduced by Phase
38.4 (lines 208-229) is exactly the safe replacement for this code path.

**Fix:** Replace both `eval` calls with `Version.safe_parse(args["data"])` or
the JSON-only equivalent (`JSON.parse(args["data"]) rescue args["data"]`).
PartyCc payloads in newer carambus_api versions are JSON; older Hash-string
payloads should be migrated server-side rather than `eval`'d on every read.

### WR-04: `_quick_game_buttons.html.erb` duplicates BK-family discipline list

**File:** `app/views/locations/_quick_game_buttons.html.erb:51`
**Issue:** The view hardcodes `bk_family_names = %w[BK2-Kombi BK50 BK100 BK-2
BK-2plus]` instead of using `Discipline::BK2_DISCIPLINE_MAP`. This is the same
list, so the constant exists for exactly this purpose. If Phase 38.5+ adds a
new BK-* discipline, the constant gets updated but the view will silently fall
through to the karambol branch and emit the wrong hidden fields, producing a
karambol-quick-start with a BK-* discipline name (which then fails in
`derive_free_game_form`).

**Fix:**
```erb
<% is_bk_family = !is_pool && !is_snooker && Discipline::BK2_DISCIPLINE_MAP.include?(category_group['category']) %>
```
(Drop the now-unused local `bk_family_names`.)

### WR-05: `Bk2::ScoreShot#calculate_raw_points` always credits `false_carom_with_pin_or_passage` even when there is no pin or passage

**File:** `app/services/bk2/score_shot.rb:71`
**Issue:** The rule key is named `false_carom_with_pin_or_passage` —
documenting the intent that a false carom only scores when accompanied by a
pin or passage. The implementation, however, unconditionally adds the points
on `obs[:false_carom]`:

```ruby
carom_points += @rules.dig(:scoring, :false_carom_with_pin_or_passage).to_i if obs[:false_carom]
```

Either the rule name is misleading (the gate isn't enforced anywhere, so a
solo "unechter Karambolage" with 0 pins and 0 passages currently scores 1
point), or the gate was forgotten. Cross-check with BK2-KOMBI-RULES.md §Rohpunkte
to confirm intent.

If the rule should be conditional:
```ruby
if obs[:false_carom] && (obs[:fallen_pins].to_i.positive? || obs[:passages].to_i.positive?)
  carom_points += @rules.dig(:scoring, :false_carom_with_pin_or_passage).to_i
end
```

### WR-06: `BackfillBk2BallsGoal` migration scans every TableMonitor without filtering, may be slow on large prod DBs

**File:** `db/migrate/20260425065047_backfill_bk2_balls_goal.rb:14`
**Issue:** `TableMonitor.find_each` scans every TableMonitor row, regardless of
discipline. On BCW (small dataset) this is fine, but the same migration on
larger Carambus deployments (e.g. carambus_api with multiple regions) will
read every TM and parse JSON. The work-set is at most a handful of in-flight
BK2-Kombi tournaments.

**Fix:** Restrict the scan to TMs with bk2_options keys:
```ruby
TableMonitor
  .where("data ILIKE ?", "%bk2_options%")
  .find_each do |tm|
  # ... existing logic
end
```
Also: the migration runs `disable_ddl_transaction!` but does no DDL, so the
flag is just a `say` comment — keep it as-is, it's harmless.

### IN-02: `Discipline#data_free_game_form` and `Discipline#ballziel_choices` parse JSON twice

**File:** `app/models/discipline.rb:391-410`
**Issue:** Both predicates parse the same `data` JSON column. For tight loops
(e.g., listing all 5 BK-* disciplines) this is parsed twice per record. Not
a correctness issue. Could be memoized:

```ruby
def data_parsed
  return {} if data.blank?
  @data_parsed ||= begin
    JSON.parse(data)
  rescue JSON::ParserError
    {}
  end
end

def data_free_game_form
  data_parsed["free_game_form"]
end

def ballziel_choices
  Array(data_parsed["ballziel_choices"])
end
```

### IN-03: `TableMonitorsController#start_game` is 200+ lines with deep branching (high complexity)

**File:** `app/controllers/table_monitors_controller.rb:66-279`
**Issue:** The action handles 5 game families (pool, snooker, karambol,
bk_family quick, bk_family detail) plus shootout / four_ball special-cases.
Branching is now layered (quick_game_form vs free_game_form vs commit) and
two CLAMP guards run in different branches. Cyclomatic complexity exceeds
the soft limit and makes correctness audits hard.

Phase 38.4-04 D-04 documented the choice to keep BK-* params clamped in-place
rather than refactoring; an `app/services/table_monitor/start_game_params.rb`
extraction (one class per game family) would reduce this controller back
to under 30 lines and enable per-family unit tests. Track as a future
clean-up item.

### IN-05: `seed_bk2_disciplines.rb` hardcodes `Discipline.find(107)` as the BK2-Kombi anchor

**File:** `script/seed_bk2_disciplines.rb:15,49`
**Issue:** The script depends on `Discipline.find(107)` being BK2-Kombi on the

### IN-07: `_show.html.erb` opens nested `<div>` blocks across `<%- if %>` / `<%- else %>` branches (HTML structure fragile)

**Issue:** The fallback-banner `<%- if @tm.bk2_state_uninitialized? %>` /

### IN-08: Test fixtures and integration tests use `OpenStruct.new(balls_goal: 60)` to stub `tournament_monitor`

**File:** `test/services/bk2/advance_match_state_test.rb:644,664`
**Issue:** `# rubocop:disable Style/OpenStructUse` is added per project linting
rules. The `define_singleton_method(:tournament_monitor)` pattern works but
is brittle — if `Bk2::AdvanceMatchState` ever calls another method on the
returned object (e.g. `tournament_monitor.tournament`), the OpenStruct will
silently return `nil` and a NoMethodError surfaces only at call time. Consider
a test factory that returns a real `TournamentMonitor` record (or a Minitest
Mock with explicit expectations).
