## Summary

Round-4 gap closure for BK2-Kombi post-dry-run delivers five focused changes:

- **P1 + P3** (`TableMonitorsController`): `BK_FAMILY_BALLZIEL_FALLBACK` constant + a widened
  `:balls_goal` → `:balls_goal_a` → `:balls_goal_b` fallback chain in
  `clamp_bk_family_params!`. Defense-in-depth against the carambus_api → local-server
  Version-sync gap.
- **P4** (`TableMonitor#add_n_balls` + `#set_n_balls`): "Option B" dispatch — when the
  active discipline is BK-family with `nachstoss_allowed`, route `:goal_reached` through
  `Bk2::CommitInning` (engages Plan 11's deferred-close) instead of legacy karambol
  `terminate_current_inning`. Resolution via name-lookup on the AR `Discipline` record
  (preserves the long-standing String contract on `TableMonitor#discipline`). Mirror
  fixes in `Bk2::AdvanceMatchState#discipline_nachstoss_allowed?` and
  `Bk2::CommitInning#free_game_form` accept either AR record or String.
- **P2** (BK-Variante view): `grid-cols-8` → `grid-cols-5`; BK-* Aufnahmebegrenzung
  `cols=5` → `cols=4`; vertical spacer above MEHRSATZ.
- **P5** (`script/seed_bk2_disciplines.rb` + new migration): narrows
  `nachstoss_allowed: true` to BK-2kombi only. Migration uses an EXPLICIT pre-check
  (`if !ApplicationRecord.local_server? || rec.id >= MIN_LOCAL_ID`) that correctly
  replaces the unreachable `rescue ActiveRecord::RecordInvalid` from earlier iterations.
- **CR-01** (`Version.local_from_api`): `local_server` → `local_server?` typo fix
  (was raising `NameError` on console invocation).

The behaviour-preservation posture is solid — fallback chains, idempotency guards, and
defensive `find_by(name: ...)` lookups dominate the diff. Test coverage scales with
the changes (controller integration tests for both BK_FAMILY constants, dispatcher
unit tests including a no-stub integration variant, ERB-compiled rendered-template
test, migration-shape regression tests, and Version-stub tests).

The findings below are quality/maintainability observations and a few minor
correctness concerns. None block deployment.

## Warnings

### WR-01: Double `@tm.save!` from `route_goal_reached_through_bk2_commit_inning`

**File:** `app/models/table_monitor.rb:1683-1699`
**Issue:** `Bk2::CommitInning.call` already performs `@tm.save!` internally (see
`app/services/bk2/commit_inning.rb:59`). After that call returns,
`route_goal_reached_through_bk2_commit_inning` then mutates the in-memory
`data[active_player]["innings_redo_list"]` and issues a second `save!` on line 1699.

This is correct (both saves operate on the same in-memory instance, so the second one
captures the post-CommitInning state plus the redo-list reset), but:

1. It executes two `UPDATE` statements per `:goal_reached` event (one inside
   `CommitInning`, one in the helper) plus full PaperTrail versioning twice — the
   trailing player's Nachstoß equalizer is one of the busiest broadcast moments of a
   BK match.
2. Future readers may add another mutation between the two saves and miss that the
   first save has already fired.

**Fix:** Either inline the redo-list reset into a helper that
`Bk2::CommitInning` calls before its `save!`, or document the intended two-save
contract on the helper docstring. Lower-impact: rename to
`route_and_reset_inning_buffer` and add a `# NOTE: triggers a second save after
CommitInning's internal save — both targeting the same in-memory @tm.data.`

```ruby
# Cheaper alternative (avoids the double save):
def route_goal_reached_through_bk2_commit_inning(player)
  # ...resolve active_player + inning_total as today...
  Bk2::CommitInning.call(...)  # this saves
  # The redo-list reset can be performed via update_columns to skip a second
  # PaperTrail record and broadcast.
  active_data = data
  active_data[active_player]["innings_redo_list"] ||= []
  active_data[active_player]["innings_redo_list"][-1] = 0
  active_data[active_player]["innings_redo_list"] << 0
  update_columns(data: active_data) # bypass callbacks/PaperTrail — buffer-reset only
rescue ArgumentError => e
  Rails.logger.error("[add_n_balls/set_n_balls bk2-routing] invalid payload: #{e.message}")
  terminate_current_inning(player)
end
```

(If PaperTrail tracking of the buffer reset is desirable, keep `save!` and just add
the `# NOTE` comment.)

### WR-02: Helper rescues only `ArgumentError`; outer `add_n_balls` rescue rewrites unrelated failures into bare `StandardError`

**File:** `app/models/table_monitor.rb:1683-1703` (helper) +
`app/models/table_monitor.rb:884-917` (`add_n_balls`)
**Issue:** The new helper rescues `ArgumentError` and falls back to
`terminate_current_inning`. Other failure modes from `Bk2::CommitInning.call`
(e.g. `ActiveRecord::RecordInvalid` from `@tm.save!`,
`ActiveRecord::StaleObjectError`, or anything raised by the inner
`Bk2::AdvanceMatchState` close-set / close-match logic) propagate up. They are then
caught by `add_n_balls`'s outer `rescue StandardError => e` (line 913), which logs
and `raise StandardError` — discarding the original message and backtrace context
for the caller.

For ProtokollEditor (`set_n_balls`), the situation is slightly different: there's no
outer rescue, so the original error propagates up to the controller. Inconsistent
error semantics between the two call sites is a code smell that will surface as a
debugging-time annoyance during dry-run.

**Fix:** Either widen the helper's rescue to `rescue StandardError => e` with the
same fallback-to-legacy posture (defensible — the docstring already says "fallback
to legacy on payload errors (defense-in-depth)"), or document why
`ActiveRecord::*` errors should propagate. If widening, log at `error` and tag with
`[bk2-routing fallback]` so operators can filter:

```ruby
rescue StandardError => e
  Rails.logger.error(
    "[add_n_balls/set_n_balls bk2-routing] dispatch failed; falling back to " \
    "legacy terminate_current_inning. error=#{e.class}: #{e.message}"
  )
  terminate_current_inning(player)
end
```

### WR-03: Migration's PaperTrail-on-master propagation contract is undocumented and unverified

**File:** `db/migrate/20260425090000_clear_nachstoss_allowed_for_non_bk2_kombi.rb:55-98`
**Issue:** The migration relies on `rec.update!(data: parsed.to_json)` on master to
(a) clear the key in master's DB AND (b) trigger PaperTrail to write a `Version` row
that local servers consume via sync. The header comment lays out this contract for
the seed script (master-first) but does NOT verify the precondition: PaperTrail must
be configured to track `Discipline#data` changes (`has_paper_trail` on the model,
plus a non-disabled `PaperTrail.enabled?` flag in the migration runtime).

If a future change disables versioning on `Discipline` or adds
`PaperTrail.request.enabled = false` to a deploy hook, this migration becomes a
silent no-op for local servers (no Version emitted by master → nothing to sync →
local DBs keep the stale flag, and `Discipline#nachstoss_allowed?` returns true on
BK50/BK100/BK-2/BK-2plus).

**Fix:** Add an explicit check at migration start that PaperTrail is enabled. Fail
loudly if not:

```ruby
def up
  unless defined?(PaperTrail) && PaperTrail.enabled?
    raise "Aborting: PaperTrail must be enabled so the cleanup propagates to local servers via Version sync"
  end
  # ...existing body...
end
```

Lower-impact alternative: add a regression test asserting the seed + migration
produce a `Version` row when run on master, locking in the contract.

### IN-02: `bk_family_with_nachstoss?` issues a DB read on every `:goal_reached`

**File:** `app/models/table_monitor.rb:1668-1675`
**Issue:** Every `add_n_balls` / `set_n_balls` call that returns `:goal_reached`
performs a `Discipline.find_by(name: name)` query before deciding the dispatch
route. For BK-2kombi matches the discipline is invariant for the life of the
game, so this lookup is redundant on every leader-reach / equalizer event. The
same redundant lookup pattern repeats in
`Bk2::AdvanceMatchState#discipline_nachstoss_allowed?` (lines 269-274) and
`Bk2::CommitInning#free_game_form` (lines 114-128).

**Fix (out of scope for v1 per `<review_scope>`, but worth a TODO):** memoize the
resolution either on the instance (`@bk_family_nachstoss_resolved` cache cleared
in `clear_options_cache`) or via `data["_bk_family_with_nachstoss"]` written
once at game-start in `start_game`. Until then, the per-event lookup is a
correctness-safe cost.

### IN-03: `route_goal_reached_through_bk2_commit_inning` mutates `data` in-place before `save!` without a `data_will_change!` call

**File:** `app/models/table_monitor.rb:1696-1699`
**Issue:** The helper writes:

```ruby
data[active_player]["innings_redo_list"] ||= []
data[active_player]["innings_redo_list"][-1] = 0
data[active_player]["innings_redo_list"] << 0
save!
```

Without an explicit `data_will_change!` before the trailing `save!`, ActiveRecord's
JSONB dirty-tracking can fail to detect this kind of nested in-place mutation
(this is a well-documented Rails pitfall — `[]=` / `<<` on a Hash/Array column
needs `*_will_change!` so AR persists it). Both `add_n_balls` (line 895) and
`set_n_balls` (line 1077) explicitly call `data_will_change!` for exactly this
reason. The helper inherits no such guarantee.

In practice this MAY still work because `Bk2::CommitInning#call` already wrote
`@tm.data["bk2_state"] = state` and saved (which marks `data` dirty, then
clean), but between that save and the helper's save, nothing has marked `data`
as changed again — so the redo-list reset risks being silently dropped, and the
phenomenon would be intermittent (only triggered on specific JSONB equality
short-circuits).

**Fix:** Add `data_will_change!` immediately before line 1699's `save!`:

```ruby
data[active_player]["innings_redo_list"] << 0
data_will_change!  # ensure JSONB dirty-tracking picks up the in-place mutation
save!
```

This is cheap insurance and matches the `add_n_balls` / `set_n_balls`
convention.

### IN-04: P2 spacer `<div class="col-span-6 h-4">` is positionally fragile — unconditional vertical space

**File:** `app/views/locations/scoreboard_free_game_karambol_new.html.erb:524-529`
**Issue:** A literal empty `<div>` (with `col-span-6 h-4`) is inserted between
the Aufnahmebegrenzung partials and the MEHRSATZ partial. The spacer is
unconditional (no `x-show`), so it adds 1rem of vertical space even when:

- `is_bk_family` is false (non-BK karambol layout where Aufnahmebegrenzung
  uses `cols=5` and the row above is a fixed-height single row — no crowding
  problem to solve).
- `parameters != 0` (the row group is hidden but the empty div still occupies
  grid space).

The system test `T-P2-mehrsatz-row-spacing` only checks that the spacer
precedes MEHRSATZ; it doesn't gate on layout state. UAT may surface visual
regression on the non-BK path.

**Fix:** Gate the spacer with the same `x-show` predicate as the rows it
separates, or scope it to BK-family only:

```erb
<%# spacer only when BK-* Aufnahmebegrenzung is rendered (variable-height row above) %>
<div x-show="is_bk_family && !is_bk_fixed_goal" class="col-span-6 h-4"></div>
```
