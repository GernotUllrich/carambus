## Warnings

### WR-01: Provisional Nachstoß resolution rule landed without sign-off

**File:** `app/services/bk2/advance_match_state.rb:197-216`
**Issue:** The new Nachstoß resolution branch encodes a provisional rule — when the trailing player equalises (`nachstoss_score >= target`), they win the set; when they fall short, the original leader wins. The inline comment explicitly says "Landessportwart sign-off pending" (line 204) and the test `T-O4-nachstoss-equal-resolves-set` echoes the same caveat ("provisional rule — verify with Landessportwart"). This is the *first* shipped semantic for set-win attribution under Nachstoß; any later correction will require a state-migration plan because in-flight `bk2_state` values written under the provisional rule will already carry `set_winner_<n>` and incremented `sets_won` counters.

In particular, the equal-case rule is non-obvious: in many tournament rule books equal-after-Nachstoß typically means the *original leader* keeps the set (because they reached `balls_goal` first), or triggers an additional shoot-off. Awarding it to the trailing player is unusual and worth a written sign-off before more games are played under this code path.

**Fix:** Either (a) gate the resolution behind a feature flag (e.g. `Discipline.data["nachstoss_resolution_rule"]` with values like `"trailing_wins_on_equal"` / `"leader_wins_on_equal"` / `"shoot_off"`), or (b) explicitly mark the equal-case path with a TODO referencing the open sign-off ticket so the rule is not silently forgotten:

```ruby
winner = if nachstoss_score > target
  nachstoss_for           # strictly above — trailing wins (uncontroversial)
elsif nachstoss_score == target
  # PROVISIONAL: equal = trailing wins. Confirm with Landessportwart before
  # removing this comment. See Plan 38.4-11 Objective.
  nachstoss_for
else
  original_leader
end
```

Option (a) is preferred because it makes the rule data-driven and lets the central API push a corrected rule without a code deploy.

### WR-03: `clamp_bk_family_params!` reads `params[:bk2_options]` outside the permit/slice

**File:** `app/controllers/table_monitors_controller.rb:429-433`
**Issue:** Inside `clamp_bk_family_params!`, the `submitted_bk2_opts` is read directly from `params[:bk2_options]` rather than from the sliced `p` hash. The comment correctly notes "nested hash not in the slice list; CLAMP provides security". That's defensible — `dz_max`/`sp_max` are integer-clamped to 1..99 right after — but it does mean:

1. The unfiltered-parameters protection at `to_h` (line 72) is bypassed for this nested key, so any future code that reads other keys from `submitted_bk2_opts` would not be permit-checked.
2. The hardcode override on line 440 (`dz_max = 2 if %w[BK-2plus BK2-Kombi].include?(p[:discipline_a].to_s)`) makes `submitted_bk2_opts[:direkter_zweikampf_max_shots_per_turn]` effectively dead-input for those two disciplines — the previous read on line 430 + clamp on line 432 are wasted work for them, though harmless.

**Fix:** Permit `:bk2_options` as a nested hash alongside the slice (so `p[:bk2_options]` would carry it), then read from `p` rather than `params`:

```ruby
# In start_game, after slice:
if params[:bk2_options].is_a?(ActionController::Parameters)
  p[:bk2_options_in] = params.require(:bk2_options).permit(
    :direkter_zweikampf_max_shots_per_turn,
    :serienspiel_max_innings_per_set
  ).to_h
end

# In clamp_bk_family_params!, read p[:bk2_options_in] instead of params[:bk2_options].
```

The integer-clamp on line 432-433 is the actual security boundary for this branch, so the urgency is low — but the deviation from the slice pattern is worth aligning.

### WR-04: `config/carambus.yml` and `config/carambus.yml.erb` are both checked in

**File:** `config/carambus.yml.erb` (and its companion `config/carambus.yml`)
**Issue:** `ls -la config/carambus.yml*` shows both files exist on disk:
- `config/carambus.yml`     (4296 bytes, Apr 25 17:48)
- `config/carambus.yml.erb` (7597 bytes, Apr 25 17:46)

The `.erb` file contains the new Plan 38.4-12 O5 canonical preset (`BK-2kombi 2/5/70+NS`) at line 44 and the full quick_game_presets map. The `.yml` companion was written *after* the `.erb` (timestamp difference) and is presumably the rendered output. If `Carambus.config` reads the rendered `.yml` (via Rails' standard YAML loader) then the rendered file is the authoritative source and any direct edit to `.yml.erb` requires a regeneration step that isn't documented in the plan. If it reads `.yml.erb` (via ERB+YAML), then the rendered `.yml` is stale and shouldn't be in the repo.

This is a workflow/configuration-management issue, not a logic bug, but the `T-O5-quick-game-shortcut-renders` test reads the `.erb` via `File.read` (line 957 of the test) and then queries `Carambus.config.quick_game_presets` programmatically (line 963). If those two sources have drifted, the test passes only because both happen to agree today.

**Fix:** Either (a) gitignore `config/carambus.yml` and document that it is generated from the `.erb` template at boot, or (b) remove the `.erb` and edit `.yml` directly (the `default:` section uses no actual ERB tags — only the env-specific overrides at lines 110-145 use ERB interpolation). Confirm with the deployment skill (`scenario-management`) which workflow is canonical for the multi-checkout setup.

### IN-01: `dz_max` clamped twice for BK-2plus/BK2-Kombi (dead-clamp branch)

**File:** `app/controllers/table_monitors_controller.rb:430-440`
**Issue:** For BK-2plus and BK2-Kombi the code computes `dz_max` from submitted params (line 430), clamps it to 1..99 (line 432), and then unconditionally overrides it to 2 on line 440. The intermediate read+clamp is dead computation for those two disciplines. Functionally fine, just slightly wasteful and obscures the simpler "always 2" intent.

**Fix:** Skip the read+clamp entirely when the discipline forces 2:

```ruby
dz_max = if %w[BK-2plus BK2-Kombi].include?(p[:discipline_a].to_s)
  2
else
  raw = submitted_bk2_opts[:direkter_zweikampf_max_shots_per_turn].to_i
  (1..99).cover?(raw) ? raw : 2
end
```

### IN-03: `data` JSON parsed multiple times in BK-family predicates

**File:** `app/models/discipline.rb:386-423`
**Issue:** `bk_family?`, `data_free_game_form`, `ballziel_choices`, and `nachstoss_allowed?` each independently call `JSON.parse(data)` — three of them re-parse on every call. For a hot-path discipline (called once per shot in `Bk2::AdvanceMatchState#discipline_nachstoss_allowed?`) this is a minor avoidable overhead. More importantly, the pattern is repeated three times with the same `rescue JSON::ParserError` boilerplate, which is duplication.

**Fix:** Extract a memoised `parsed_data` helper:

```ruby
def parsed_data
  return {} if data.blank?
  @parsed_data ||= begin
    JSON.parse(data)
  rescue JSON::ParserError
    {}
  end
end

def data_free_game_form;   parsed_data["free_game_form"];                 end
def ballziel_choices;      Array(parsed_data["ballziel_choices"]);        end
def nachstoss_allowed?;    parsed_data["nachstoss_allowed"] == true;      end
```

Note that memoisation must invalidate on `data=` writes (override the writer or hook into `before_save`); for read-only call sites the memoisation is safe.

### IN-04: Nachstoß state slots accumulate across sets without cleanup

**File:** `app/services/bk2/advance_match_state.rb:197-228, 239-258`
**Issue:** When the Nachstoß resolution path fires (lines 211-215), `state["nachstoss_pending"]` is set to `false` and `state["nachstoss_resolved"]` to `true` — but `state["nachstoss_for"]` is **not** cleared. Then `advance_to_next_set!` (lines 239-258) does not reset `nachstoss_resolved` or `nachstoss_for` either. So set 2 starts carrying the set 1 Nachstoß metadata. If a tooling/UI consumer reads `nachstoss_for` to render a "Nachstoß for playerb" banner, it would persist across set boundaries.

The legacy/non-Nachstoß path (line 230-234) doesn't touch any of these slots, so the leak only manifests when set 1 closes via Nachstoß resolution. None of the current tests assert on residual state between sets.

**Fix:** Clear the Nachstoß slots in `advance_to_next_set!`:

```ruby
def advance_to_next_set!(state, set_winner)
  return if state["current_set_number"] >= 3

  # 38.4-11 O2 cleanup: Nachstoß slots are per-set; reset on set advance.
  state.delete("nachstoss_pending")
  state.delete("nachstoss_for")
  state.delete("nachstoss_resolved")

  state["current_set_number"] += 1
  # …rest unchanged
end
```

Also worth adding a regression test that initialises a fresh set 2 after a Nachstoß-closed set 1 and asserts these keys are absent.

### IN-06: Test `T-O4-nachstoss-equal-resolves-set` bakes in the provisional rule

**File:** `test/services/bk2/advance_match_state_test.rb:766-777`
**Issue:** The test asserts `state.dig("sets_won", "playerb") == 1` when the trailing player equalises at `balls_goal=50`. If the Landessportwart sign-off comes back with the opposite rule (leader wins on equal), this test will need to flip and any production data already written under the provisional rule will be inconsistent with the corrected rule. The test name and comments do call out the provisional nature ("provisional rule — verify with Landessportwart"), but a test that bakes in a contested rule is itself an issue worth tracking.

**Fix:** Pair the test with a tracking marker — either a `skip` guarded by a constant, or tag it with a fixture-flag so it can be flipped centrally:

```ruby
NACHSTOSS_EQUAL_WINNER = :trailing # PROVISIONAL — see Plan 38.4-11
…
expected_winner = (NACHSTOSS_EQUAL_WINNER == :trailing) ? "playerb" : "playera"
assert_equal 1, state.dig("sets_won", expected_winner), …
```

Lower priority than WR-01 (which asks for the same thing on the production-code side) — flagged as Info because the test correctness will follow whatever the production rule decides.
