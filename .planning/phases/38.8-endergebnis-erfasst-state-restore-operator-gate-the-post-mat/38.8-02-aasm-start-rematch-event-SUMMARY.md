## Decisions Made

- **After-callbacks Array literal vs %i[...] form** — Plan's prescribed code uses `after: [:revert_players, :do_play]` (Array of symbols literal). Acceptance criterion grep is `after: \[:revert_players, :do_play\]` (regex matches the bracket-form, not the percent-i form). Used the exact prescribed form. Both forms produce identical Ruby objects (`[:revert_players, :do_play]`), so this is purely a stylistic match-to-spec.
- **i18n alphabetical insertion** — Plan literally said `next_game:` should go "immediately after the `name:` key". Strict literal interpretation would put it as the next line after `name: Name`, which happens to be alphabetically correct (`next_game` < `next_game_id`). The two interpretations converge here, so no real choice was needed. Documented for clarity.
- **NOT modifying ResultRecorder's auto-rematch block** — Plan explicitly says "DOES NOT delete the auto-rematch block in ResultRecorder (Plan 03 owns that)". The Plan 01 RED test STILL fails after this plan, which is correct: this plan only ships the new event; Plan 03 ships the deletion that turns the test GREEN.
- **Phase 38.7 tiebreak guard preservation verified explicitly** — Ran `grep -c "tiebreak_not_pending?"` before AND after the edit. Both returned 3. This is a defense against accidental search-and-replace damage to neighboring AASM events.
