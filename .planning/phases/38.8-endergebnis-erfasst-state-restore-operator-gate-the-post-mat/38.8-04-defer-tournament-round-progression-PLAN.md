acceptance the TM enters `:final_match_score` AND the round-progression cascade
(`populate_tables`, `incr_current_round!`, `finalize_round`, `accumulate_results`,
`update_ranking`, `start_playing_groups!`/`start_playing_finals!`,

Today the cascade fires inside `report_result` (lines 84-116) immediately after
`finish_match!` runs under the DB lock — clobbering the "Endergebnis erfasst"
display before the operator sees it.

Approach (extend-before-build SKILL — small refactor of an existing method,
NOT a parallel state machine):

1. EXTRACT the cascade (lines 84-116 of result_processor.rb) into a new public
   method `advance_round_after_match_close(table_monitor)` on
   `TournamentMonitor::ResultProcessor`.
2. REMOVE the cascade-call from `report_result` — keep only the lock-guarded data
   write + `finish_match!` (lines 51-77) + `finalize_game_result` call (lines 81-82).
3. ADD a new instance method on TableMonitor `advance_tournament_round_if_present`
   that delegates to `tournament_monitor && TournamentMonitor::ResultProcessor.new(tournament_monitor).advance_round_after_match_close(self)`.
4. WIRE the AASM `:close_match` event to fire `advance_tournament_round_if_present`
   as an `after:` callback. Existing `close_match` event currently has no callback
   (table_monitor.rb:353-355).

After this plan: tournament TM enters `:final_match_score`, stays there. Operator
clicks "Weiter" / "Continue" (Plan 05 wires the button) -> reflex calls
`close_match!` -> AASM transitions to `:ready_for_new_match` -> `after` callback
