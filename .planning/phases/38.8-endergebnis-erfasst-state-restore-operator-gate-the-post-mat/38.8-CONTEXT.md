# Phase 38.8 — Endergebnis-erfasst State Restore

## Why this phase exists

Discovered during phase 38.7 UAT (post-tiebreak retest). User reported a state-machine integrity bug affecting **all disciplines, both modes**: the AASM state `final_match_score` (display name "Endergebnis erfasst") is being skipped after match completion. In training mode the system auto-starts the next game; in tournament mode the system auto-progresses the round. Both bypass the operator-gate that was working before phase 38.

## User-stated expected behavior (verbatim)

> "There is a state skipped now when the Match is finished. It is The state 'Endergebnis erfasst'. In Training mode: when the final result has been accepted, it must not immediately give way to the next game. There has to be a display of the End-status, signalling also, that the endresult has been reported. Only when going to the next state from 'Endergebnis erfasst', the TableMonitor gives headway to the TournamentMonitor for Tournament Matches or starts a new game in Training Mode."

## Root cause (already diagnosed)

`ResultRecorder#evaluate_result` fuses rematch (training) / round-progression (tournament) with result acknowledgment in the same call. Regression introduced in commit `c3dedb69` (2026-03-24, "Fix: Race conditions, duplicate protocol modals, and training game rematch") — pre-dates phase 38.x numbering but lands in the same dev window. Subsequent refactors (`ea49e65b` extracted ResultRecorder; `7df72943` added the 38.7 tiebreak guard) preserved the broken block verbatim. Survived all of phase 38 because no test asserted the `final_match_score` operator-gate contract.

**Full diagnosis:** `.planning/debug/endergebnis-erfasst-state-skip.md`
