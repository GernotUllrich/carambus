## Decisions Made

- **Reflex method placement: clustered with operator-control surface.** Inserted between force_next_state and stop, keeping admin_ack_result → force_next_state → start_rematch → close_match → stop together. Matches Carambus's organizational convention (related methods adjacent in source).
- **View branch as `elsif` not separate `if`.** The plan literally says replace the existing `<%- if state_display.present? && !final_match_score? %>` block with an expanded if/elsif chain. This is mutex-safe: any TM state hits exactly one branch. A separate `<%- if final_match_score? %>` block would also work, but two adjacent state-conditional blocks at the same DOM level is harder to review.
- **Two content_tag blocks (training + tournament) instead of one parameterized block.** A single content_tag with a conditional `data:` hash would have saved ~5 LOC, but reads less cleanly because the data-reflex string is the load-bearing part of the contract. Each arm having its own visible reflex string aids debugging when StimulusReflex frame logs land in production.
- **touchstart event pairing.** Mirrored from existing admin_ack_result pattern in `_game_row.html.erb:81`. Provides mobile/touchscreen responsiveness without waiting for the synthetic click event from the browser. Both arms include touchstart for symmetry.
- **id="next_game_button" shared between arms.** Safe because if/else mutex guarantees only one arm renders. Allows external CSS / Cypress / system tests to address the button consistently regardless of mode.
- **Phase 38.7 tiebreak guard preservation explicitly verified.** Same defensive grep-before-and-after pattern as Plans 02-04. Zero collateral damage to upstream guards.
- **No model layer touched.** This plan is purely view + reflex. The plan's `out_of_scope` section explicitly forbids model edits ("DOES NOT modify the AASM events themselves — Plan 02 / Plan 04 own those").
