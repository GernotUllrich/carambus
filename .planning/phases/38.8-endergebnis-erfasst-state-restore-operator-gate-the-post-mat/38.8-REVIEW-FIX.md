### CR-02: `:close_match` after-callback re-enters itself via `TournamentMonitorState#finalize_round` — **[RECLASSIFIED: FALSE POSITIVE 2026-05-01]**

**Status:** Reclassified after fix landed. User manually validated BK-2 round_play scenario end-to-end and observed cascade fires exactly once per operator click. The static read of `TournamentMonitorState#finalize_round` underlying CR-02 missed an actual runtime guard preventing recursion. Per CLAUDE.md ("don't add validation for scenarios that can't happen"), the sentinel was dead code.

**Reverted commits:**
- `0dbe45c4` reverts `37796f7d` (the sentinel + 63-line synthetic regression test in `test/services/tournament_monitor/result_processor_test.rb`)
- `61e9e8c8` reverts `9878db16` (the standardrb style follow-up)

After revert: `app/models/table_monitor.rb#advance_tournament_round_if_present` retains only the CR-01 polymorphism guard. No sentinel. The synthetic regression tests are gone (they tested a sentinel mechanism that no longer exists).

Original applied-fix description preserved below for archival reference only.

---

**Files modified (REVERTED):** `app/models/table_monitor.rb`, `test/services/tournament_monitor/result_processor_test.rb`
**Commit (REVERTED):** `37796f7d`
**Applied fix:**
- Added a thread-local re-entry sentinel `Thread.current[:_advancing_round_for_tm]` in `TableMonitor#advance_tournament_round_if_present` (table_monitor.rb:1704-1714). When the cascade calls `@tournament_monitor.finalize_round`, that method itself iterates over its TableMonitors and fires `tabmon.close_match!` on each one — re-entering this same after-callback. The sentinel short-circuits any nested invocation while the outer cascade is still in flight, and is cleared via `ensure` so subsequent unrelated cascades are not blocked even on exception.
- Added two regression tests in `test/services/tournament_monitor/result_processor_test.rb`:
  - `advance_tournament_round_if_present short-circuits when sentinel is set (CR-02)` — verifies `TournamentMonitor::ResultProcessor.new` is NOT called when the sentinel is pre-set
  - `advance_tournament_round_if_present clears sentinel via ensure even on exception (CR-02)` — verifies the sentinel is cleared even when the cascade raises
- All 26 TournamentMonitor::ResultProcessor tests pass.

**Note (logic verification):** The sentinel-based re-entry guard is the lowest-risk option among the three fix alternatives proposed in REVIEW.md. Option (a) (remove `close_match!` calls from `finalize_round` and `PartyMonitor::ResultProcessor#finalize_round`) was deferred — those calls predate Phase 38.8 and removing them would change legacy state-reset behavior beyond this fix's scope. Option (c) (restrict the AASM `from:` set) would have required splitting `:close_match` into two events. The sentinel keeps the AASM contract intact while preventing the recursion. Recommend manual confirmation that the cascade fires exactly once per operator click in a real tournament-mode end-to-end run before merging to production.
