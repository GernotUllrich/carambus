# Phase 38.8: Code Review Report

**Reviewed:** 2026-05-01
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

Phase 38.8 restores the `:final_match_score` operator-gate by:
1. Deleting the auto-rematch AASM bypass in `ResultRecorder` (Plan 03)
2. Adding an `:start_rematch` AASM event with `revert_players` + `do_play` after-callbacks (Plan 02)
3. Deferring the round-progression cascade out of `report_result` into a new public method `advance_round_after_match_close` and wiring it via an `after:` callback on `:close_match` (Plan 04)
4. Adding two new reflex actions (`start_rematch`, `close_match`) and a final_match_score branch in the scoreboard (Plan 05)
5. Locking the contract with model, service, and system tests (Plans 01, 06)

The training-mode operator-gate is correctly restored and the static-source contract tests are well-designed locks.

However, two **Critical** issues regress the `PartyMonitor` (league-match) flow because the new `:close_match` after-callback unconditionally treats the polymorphic `tournament_monitor` association as a `TournamentMonitor`, and because `TournamentMonitorState#finalize_round` itself calls `close_match!` on every TM — re-entering the same after-callback chain. Both pre-existing call sites of `close_match!` (in `PartyMonitor::ResultProcessor` and `TournamentMonitorState#finalize_round` itself) inherited a brand-new and incorrect side-effect from this phase.

Plus four warnings and four info items, mostly around the recursion / idempotency claim, error logging level, and ERB duplication.

## Critical Issues

### CR-01: `advance_tournament_round_if_present` mishandles polymorphic `PartyMonitor` association — crash in PartyMonitor flow

**File:** `app/models/table_monitor.rb:1684-1691`
**Issue:**
`tournament_monitor` is a **polymorphic** belongs_to (`belongs_to :tournament_monitor, polymorphic: true, optional: true`, line 53). The actual associated record can be a `TournamentMonitor` OR a `PartyMonitor` (compare lines 106 `tournament_monitor.is_a?(PartyMonitor)` and 156 `tournament_monitor.is_a?(TournamentMonitor)`).

The new bridge method unconditionally builds a `TournamentMonitor::ResultProcessor` with whatever the association returns:

```ruby
def advance_tournament_round_if_present
  return if tournament_monitor.blank?
  TournamentMonitor::ResultProcessor.new(tournament_monitor).advance_round_after_match_close(self)
  ...
end
```

When `tournament_monitor` is a `PartyMonitor`, `advance_round_after_match_close` will then call:
- `@tournament_monitor.populate_tables` — **does NOT exist on `PartyMonitor`** (verified: `grep "def populate_tables" app/models/party_monitor.rb` returns nothing)
- `@tournament_monitor.start_playing_finals!` — does not exist on `PartyMonitor`
- `@tournament_monitor.start_playing_groups!` — does not exist on `PartyMonitor`
- `@tournament_monitor.group_phase_finished?` — does not exist on `PartyMonitor`
- `@tournament_monitor.finals_finished?` — does not exist on `PartyMonitor`
- `@tournament_monitor.end_of_tournament!` — does not exist on `PartyMonitor`

Because `app/services/party_monitor/result_processor.rb` calls `tabmon.close_match!` at lines 154 (`finalize_round`) and 186 (`finalize_game_result` for manual_assignment), every league-match flow now triggers this path and will raise `NoMethodError` mid-cascade. The rescue in `advance_tournament_round_if_present` re-raises, which then propagates out of the AASM transition.

**Fix:**
Guard on the association class:

```ruby
def advance_tournament_round_if_present
  return if tournament_monitor.blank?
  return unless tournament_monitor.is_a?(TournamentMonitor)
  Rails.logger.info "[advance_tournament_round_if_present] m6[#{id}] delegating to ResultProcessor#advance_round_after_match_close"
  TournamentMonitor::ResultProcessor.new(tournament_monitor).advance_round_after_match_close(self)
rescue StandardError => e
  Rails.logger.error "[advance_tournament_round_if_present] m6[#{id}] ERROR: #{e}, #{e.backtrace&.first(3)&.join(" <- ")}"
  raise
end
```

Add a regression test in `test/services/party_monitor/result_processor_test.rb` (or extend the existing one) to assert `PartyMonitor::ResultProcessor#finalize_round` does not crash with the Phase 38.8 `:close_match` after-callback wired up — preferably with a real `PartyMonitor` fixture so the polymorphic dispatch is exercised end-to-end.

### CR-02: `:close_match` after-callback re-enters itself via `TournamentMonitorState#finalize_round` — non-idempotent recursion

**File:** `app/models/table_monitor.rb:358` (AASM event) + `app/services/tournament_monitor/result_processor.rb:120` + `lib/tournament_monitor_state.rb:54`
**Issue:**
The new `after: :advance_tournament_round_if_present` callback on `:close_match` fires the cascade. Inside that cascade, `advance_round_after_match_close` calls `@tournament_monitor.finalize_round` (line 120), which is `TournamentMonitorState#finalize_round` (mixed into `TournamentMonitor`):

```ruby
# lib/tournament_monitor_state.rb:44-56
table_monitors.joins(:game).each do |tabmon|
  ...
  tabmon.close_match!         # <-- triggers :close_match again
  tabmon.update(game_id: nil)
end
```

Each `tabmon.close_match!` here re-fires the after-callback, which calls `advance_round_after_match_close` again, which calls `finalize_round` again, etc. The AASM `from:` set for `:close_match` includes `ready_for_new_match` (line 359), so the self-transition is always valid. Termination depends on the second-pass `populate_tables` having reset some TMs to `:ready` (which would fail `all_table_monitors_finished?`) — a fragile race-style termination, not the "idempotent" behavior the new comment claims:

```ruby
# table_monitor.rb:1682-1683
# Idempotent — the cascade is internally gated by
# all_table_monitors_finished? + AASM may_? predicates.
```

Actual outcomes:
- Best case: cascade fires twice (once for the operator click, once for `finalize_round`'s loop), with duplicated `accumulate_results`, duplicated `TournamentMonitorUpdateResultsJob`, duplicated `TournamentStatusUpdateJob`, and an unnecessary second `populate_tables` / `incr_current_round!`. Round counter advances by 2 instead of 1.
- Worst case: if `populate_tables` fails to assign new games to all TMs, `all_table_monitors_finished?` stays true and the recursion runs further.

This is the same issue raised separately in CR-01 if the TM has a `PartyMonitor` — but applies to plain `TournamentMonitor` flows too.

**Fix:**
Either:

(a) Remove the `close_match!` calls from `TournamentMonitorState#finalize_round` (lib/tournament_monitor_state.rb:54) and from `PartyMonitor::ResultProcessor#finalize_round` (app/services/party_monitor/result_processor.rb:154) — they were the legacy way of marking TMs done, but now `close_match!` carries new side-effects.

(b) Add a re-entry guard in `advance_tournament_round_if_present`:

```ruby
def advance_tournament_round_if_present
  return if tournament_monitor.blank?
  return unless tournament_monitor.is_a?(TournamentMonitor)
  return if Thread.current[:_advancing_round_for_tm]
  Thread.current[:_advancing_round_for_tm] = true
  TournamentMonitor::ResultProcessor.new(tournament_monitor).advance_round_after_match_close(self)
ensure
  Thread.current[:_advancing_round_for_tm] = nil
end
```

(c) Restrict the `from:` set on `:close_match` so the cascade only fires from `:final_match_score` (the operator-gate state), and use a different event/AASM transition for `finalize_round`'s legacy flow.

Option (a) is cleanest if no behavior depends on `close_match!` resetting state in `finalize_round`'s loop. Add a test that drives the full `report_result` -> `close_match!` -> `finalize_round` cycle on a real fixture and asserts `accumulate_results` runs exactly once per operator click.

## Warnings

### WR-01: `advance_round_after_match_close` is not actually idempotent — comment claim is misleading

**File:** `app/services/tournament_monitor/result_processor.rb:104-153`
**Issue:**
The header comment states:

```
# Idempotent: each branch is gated by AASM may_? predicates and
# all_table_monitors_finished? — re-running on an already-progressed round
# is a no-op.
```

But none of the AASM `may_?` predicates are checked inside this method. The actual calls are unconditional:
- `@tournament_monitor.finalize_round` (no `may_?` predicate exists for this)
- `@tournament_monitor.incr_current_round!` (advances counter every time)
- `@tournament_monitor.populate_tables` (mutates table assignments)
- `@tournament_monitor.start_playing_groups!` / `start_playing_finals!` (these DO have `may_?` predicates but are not guarded here)

A second invocation with `all_table_monitors_finished?` still true (e.g., from the recursion in CR-02, or from a double-click race) will:
- Increment `current_round` again
- Re-populate tables (could clobber operator-set assignments)
- Re-enqueue both jobs
- Re-call AASM events that may raise `AASM::InvalidTransition` from the new state

**Fix:**
Either make it actually idempotent (wrap each call in a `may_?` guard or a `return if @tournament_monitor.last_round_advanced_at == ...`-style sentinel), or rewrite the comment to say "intended to be called exactly once per operator click via the `:close_match` after-callback; not safe to re-invoke". Pair with the recursion fix in CR-02.

### WR-02: System test monkey-patches `TableMonitor.find` at the class level — leaks under parallel execution

**File:** `test/system/final_match_score_operator_gate_test.rb:144-154`
**Issue:**
`TableMonitor.define_singleton_method(:find) { |_id| tm }` overrides the class-level `find` method process-wide. The `ensure` block tries to undo it via `remove_method` but rescues `nil` if it fails. Two concrete risks:

1. If another thread (Minitest parallel runner, ActionCable broadcast inside the test, Sidekiq inline runner) calls `TableMonitor.find(id)` while the patch is active, it returns the wrong object regardless of the requested id.
2. If `remove_method` ever raises something other than what `rescue` catches (e.g., a `TypeError` from a frozen class), the stub persists into subsequent tests.

The standard project pattern (`@tm.stub(:method, ...)` per result_recorder_test.rb) doesn't apply here because we're patching a class method, but `Mocha`/`Minitest::Mock`-based stubs are still safer.

**Fix:**
Replace the manual override with `Minitest::Mock` or `stub_any_instance`:

```ruby
TableMonitor.stub(:find, ->(_id) { tm }) do
  reflex.start_rematch
end
```

This auto-restores after the block exits even on exception, with no manual `ensure` needed.

### WR-03: Two near-identical ERB blocks in `_scoreboard.html.erb` final_match_score branch

**File:** `app/views/table_monitors/_scoreboard.html.erb:169-185`
**Issue:**
The training-arm and tournament-arm both render the same div with the same id, classes, label, and tabindex — only the reflex name differs. ID `next_game_button` is rendered identically in both branches (no Stimulus controller cares about uniqueness here, but having two branches that differ only in one attribute is a code-quality smell and a future bug hazard).

**Fix:**
Compute the reflex name once, then render once:

```erb
<%- elsif table_monitor.final_match_score? %>
  <%- reflex_name = table_monitor.tournament_monitor.blank? ? "start_rematch" : "close_match" %>
  <%= content_tag "div",
        data: { reflex: "click->TableMonitorReflex##{reflex_name} touchstart->TableMonitorReflex##{reflex_name}", id: table_monitor.id },
        id: "next_game_button",
        class: "flex flex-col h-1/6 items-center justify-center font-bold text-green-500 text-#{vw2} text-center cursor-pointer dark:bg-black hover:bg-gray-200 dark:hover:bg-gray-800 rounded",
        tabindex: 1 do %>
    <%= t("table_monitor.next_game") %>
  <%- end %>
<%- end %>
```

This also surfaces the routing decision in one place rather than buried in the structure of two if-branches.

### IN-01: `Rails.logger.info` used to log error conditions in two places

**File:** `app/services/tournament_monitor/result_processor.rb:98, 151`
**Issue:**
Both rescue blocks log at `info` level for `StandardError`:

```ruby
rescue StandardError => e
  Rails.logger.info "StandardError #{e}, #{e.backtrace&.join("\n")}"
  raise ActiveRecord::Rollback
```

Errors should be logged at `error` (or at least `warn`) level so they surface in production log filtering. The pre-existing `report_result` rescue is the same pattern (not introduced by Phase 38.8), but the new `advance_round_after_match_close` rescue copied it.

**Fix:** Change both to `Rails.logger.error`. The bridge in `app/models/table_monitor.rb:1689` already uses `error` correctly.

### IN-03: Static-source regex in `report_result_match` is fragile to method-ordering changes

**File:** `test/services/tournament_monitor/result_processor_test.rb:313, 328`
**Issue:**
```ruby
src.match(/def report_result.*?(?=\n  def )/m)
```

This relies on the next method definition being indented exactly two spaces. If a developer adds a private method, prepends a `module ... end`, or refactors with `concerns`, the regex stops matching the right region. The test would silently start passing for the wrong reason (no body matched, no `populate_tables` found = test "passes").

**Fix:** Use AST parsing (e.g., `Parser::CurrentRuby`), or assert a positive marker first:

```ruby
report_result_match = src.match(/def report_result.*?(?=\n  def )/m)
assert_not_nil report_result_match
body = report_result_match[0]
# Sanity check: the body should still contain at least one expected line.
assert_match(/finalize_game_result\(table_monitor\)/, body, "regex did not capture report_result body")
refute_match(...)
```

In the Rails test environment OpenStruct is currently autoloaded (other tests in this codebase use it the same way without `require`), but for portability and as a safety net it would be cleaner to `require "ostruct"` at the top of the file.
