## Summary

Phase 38.2 re-aligns the BK2-Kombi scoreboard UI to be Karambol-parallel (D-01 structural parity) and augments `bk2_options` with `first_set_mode` + per-phase rule limits (D-14 / D-20). The changes are tightly scoped and well-commented, with consistent clamp/whitelist posture in the controller (`T-38.1-06-01` parity for `first_set_mode`). Test coverage is strong at the unit + integration layers (5 new controller tests for `first_set_mode`, 8 new service tests for bk2_options-driven init and phase-flip, 9 system tests covering all major deltas + UAT-GAP regressions).

Security review clean: XSS surface is unchanged (all user-controlled fields flow through `<%= %>` ERB escaping or `t()`), strong-params whitelist for `bk2_first_set_mode` is narrow (single scalar, controller-side enum clamp), and the Alpine `x-data` scope lift (UAT-GAP-02) widens only over the intended form tree — no sibling elements gained unintended reactivity. The Stimulus dataset-key contract (10 camelCase targets → underscore reflex keys) is preserved verbatim between the 38.2 controller rewrite and `TableMonitorReflex#bk2_kombi_submit_shot`; the reflex code was out of scope (correctly) and consumes the same keys.

One warning flags a cross-boundary inconsistency: `direkter_zweikampf_max_shots_per_turn` from `bk2_options` is respected at `init_state_if_missing!` and at set-boundary reset in `close_set_if_reached!`, but the intra-set turn-end reset inside `ScoreShot#calculate_direct_duel_transitions` (out of scope, untouched) hardcodes a reset value of `2`. Info items flag an unused local variable, duplicate DOM ids copied from Karambol parent, unconditional production logging, a `rescue`-as-nil-guard idiom, and a missing test case for a nil (vs. absent) `first_set_mode` value.

## Warnings

### WR-01: `direkter_zweikampf_max_shots_per_turn` not honored at intra-set turn-end reset

**File:** `app/services/bk2_kombi/advance_match_state.rb:91-93` (plus untouched `app/services/bk2_kombi/score_shot.rb:191`)
**Issue:** Phase 38.2 introduces `derive_dz_max_shots` to seed `state["shots_left_in_turn"]` from `bk2_options["direkter_zweikampf_max_shots_per_turn"]` (default 2) at two places — `init_state_if_missing!` (line 73) and `close_set_if_reached!` (line 162, on set advance). However, WITHIN a set, when a turn ends in DZ phase, `ScoreShot#calculate_direct_duel_transitions` (unchanged in 38.2) hardcodes `next_shots_left_in_turn: turn_ends ? 2 : next_shots`. So an operator who configures `direkter_zweikampf_max_shots_per_turn: 3`:

- Set starts → `shots_left_in_turn = 3` (correct, seeded from options)
- After shot 1 → `shots_left_in_turn = 2` (correct, decremented)
- After shot 2 (turn ends on foul or zero-point) → `shots_left_in_turn = 2` (WRONG — should be 3)
- Subsequent set start → `shots_left_in_turn = 3` (correct again)

This yields an internally inconsistent rule: the first turn of each set respects the configured limit, but all subsequent turns within that set silently fall back to the hardcoded default. CONTEXT explicitly notes ScoreShot is out of scope for 38.2, so this is not a 38.2 regression per se, but the new configurability only partially plumbs through. The existing test `38.2-01 T1` asserts `shots_left_in_turn == 2` after one non-bonus shot (seeded 3, decremented by 1) — it does not exercise the turn-end reset path, so this inconsistency is not test-caught.

**Fix:** Either (a) scope-extend ScoreShot to honor the bk2_option, or (b) have `apply_transitions!` in `AdvanceMatchState` override the hardcoded reset when in DZ and `turn_ends`. Option (b) is local to files in 38.2 scope:

```ruby
# app/services/bk2_kombi/advance_match_state.rb, apply_transitions!
def apply_transitions!(state, transitions)
  state["player_at_table"] = transitions[:next_player_at_table].to_s
  if state["current_phase"] == "direkter_zweikampf"
    # Override ScoreShot's hardcoded reset-to-2 when turn ends in DZ, so
    # bk2_options.direkter_zweikampf_max_shots_per_turn is honored.
    state["shots_left_in_turn"] = transitions[:turn_ends] ? derive_dz_max_shots : transitions[:next_shots_left_in_turn]
  end

  if state["current_phase"] == "serienspiel" && transitions[:turn_ends]
    current = state["innings_left_in_set"].to_i
    state["innings_left_in_set"] = [current - 1, 0].max
  end
end
```

Either way, add a regression test alongside `38.2-01 T1` that sets `direkter_zweikampf_max_shots_per_turn: 3` and drives a turn-end within set 1, asserting `shots_left_in_turn` resets to 3 (not 2).

### IN-02: Duplicate HTML `id="game_state"` in scoreboard partial

**File:** `app/views/table_monitors/_scoreboard_bk2_kombi.html.erb:132, 137, 142`
**Issue:** Three `content_tag "div", id: "game_state", ...` calls produce duplicate DOM ids. Lines 132 and 137 are mutually exclusive (if/else on `state_display.present?`) so only one renders, but line 142 unconditionally renders a THIRD `<div id="game_state">` in the rendered output. `document.getElementById("game_state")` returns only the first match and any CSS/JS selector targeting `#game_state` will bind ambiguously. This is a copy-paste of the same issue from `app/views/table_monitors/_scoreboard.html.erb:63, 69, 75` (Karambol parent) — technically pre-existing technical debt, but replicated into a NEW partial in this phase. Out-of-scope for v1 (`quick`/`standard` review) under strict interpretation, but flagged since the new partial reproduces rather than fixes the pattern.
**Fix:** Rename at least one to a distinct id (e.g., `game_state_body` for line 142) and update any JS/CSS consumers. Consider replicating the fix upstream in `_scoreboard.html.erb` if the patch is small.

### IN-03: Unconditional `Rails.logger.info` on every render in production

**File:** `app/views/table_monitors/_show_bk2_kombi.html.erb:4`
**Issue:** `<%- Rails.logger.info "tam _show_bk2_kombi !!!!!!!!!!!!!!!!!!!!!!!!!!!!! #{tid} !!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!" %>` fires on every partial render in ALL environments including production. This is a verbatim copy of the existing `_show.html.erb:4` pattern so it's technically parity-compliant (D-01 structural 1:1 copy), but the log line is pure diagnostic noise and adds no value at `:info` level. At the frequency the scoreboard re-renders (every shot, every CableReady broadcast), this floods production logs.
**Fix:** Downgrade to `.debug` with a block form, matching the 38.1 IN-02 pattern used elsewhere in `table_monitors_controller.rb`:

```erb
<%- Rails.logger.debug { "tam _show_bk2_kombi [#{tid}]" } %>
```

If the Karambol parent is fixed at the same time (to preserve parity), this is a trivial two-file patch.

### IN-04: `rescue`-as-nil-guard in scoreboard winner lookup

**File:** `app/views/table_monitors/_scoreboard_bk2_kombi.html.erb:86`
**Issue:** `(options[:player_a][:fullname] rescue "Player A")` uses a modifier `rescue` to catch `NoMethodError` when `options[:player_a]` is `nil`. This is a Ruby anti-pattern (rescue catches all `StandardError`, not just the nil-navigation case — any exception in `[:fullname]` would also silently fall back to the fixture string). Since `options` is a Hash built by `TableMonitor#get_options!`, safer to use `dig`:

```erb
winner: match_winner == "playera" ? (options.dig(:player_a, :fullname) || "Player A") : (options.dig(:player_b, :fullname) || "Player B")
```

**Fix:** Replace `rescue` with `Hash#dig || fallback` at line 86.

### IN-05: No explicit test for `bk2_options["first_set_mode"] = nil` (as opposed to missing key)

**File:** `test/services/bk2_kombi/advance_match_state_test.rb` (tests `38.2-01 T7` and `38.2-01 T8`)
**Issue:** T7 covers `bk2_options => {}` (key absent) and T8 covers `bk2_options["first_set_mode"] = "attacker_injected_value"` (non-whitelisted string). Neither covers the case `bk2_options["first_set_mode"] = nil`, which `derive_first_set_mode` handles via `.to_s` → `""` → whitelist miss → default. Today the predicate-plus-default chain is robust, but an explicit test would guard against a future refactor that replaces `.to_s` with stricter type handling (e.g., `.to_s.presence || ...`) and accidentally regresses this path.
**Fix:** Add a one-liner test mirroring T8 with `"first_set_mode" => nil` in `bk2_options`.
