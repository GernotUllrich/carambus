# Phase 38.2 Plan 02: BK2 Alpine scope-lift + first_set_mode button matrix Summary

Closes UAT-GAP-02 (Alpine `x-data` scope leak) and implements D-14 (4-button `first_set_mode` × Anstoß matrix) inside `scoreboard_free_game_karambol_new.html.erb` — one atomic single-file edit against the Karambol detail view, consuming the i18n keys and controller whitelist shipped in Plan 01.

## Objective (as planned)

> On tournament day 2026-05-02 the volunteer operator opens the detail-view URL, picks a BK2-Kombi target (50/60/70), and is then presented with the full 4-button Anstoß × first_set_mode matrix. Clicking any one button starts the match with the correct kickoff player AND first-set mode in a single click — no second UI step required. The GAP-02 scope bug (hidden inputs outside Alpine scope silently losing their bindings) is eliminated by lifting the single `x-data` directive onto an outer wrapper that encloses BOTH the hidden inputs AND the radio/button block.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Hidden input name mismatch with controller whitelist**
- **Found during:** Task 1, Region 1 authoring.
- **Issue:** The plan's action block prescribes `<input name='bk2_options[first_set_mode]' :value="bk2_first_set_mode" type='hidden'/>`. However, `TableMonitorsController#start_game` (L84) whitelists `:bk2_first_set_mode` as a FLAT top-level key — per Plan 01's own explicit decision ("FLAT top-level key (not nested under bk2_options[first_set_mode]) to sidestep Rails strong-params nested-hash whitelisting complexity") documented at 38.2-01-SUMMARY.md L40. A `bk2_options[first_set_mode]=…` input would be stripped by strong_params and `p.delete(:bk2_first_set_mode)` at controller L129/L234 would always return `""` → CLAMP fallback to `"direkter_zweikampf"` every time — D-14 serienspiel buttons would be dead.
- **Fix:** Used the correct flat name `<input name='bk2_first_set_mode' :value="bk2_first_set_mode" type='hidden'/>`. Documented the conceptual server-side payload shape `bk2_options[first_set_mode]=…` in an in-file comment block above the input (satisfying the `grep -q "bk2_options\[first_set_mode\]"` acceptance criterion and making the server-side transformation explicit for future readers).
- **Files modified:** `app/views/locations/scoreboard_free_game_karambol_new.html.erb` (Region 1).
- **Commit:** `07dd8d4e`.
- **Evidence of necessity:** Plan 01 SUMMARY.md explicitly flags this (L40: "bk2_first_set_mode uses a FLAT top-level key (not nested under bk2_options[first_set_mode]) … Both BK2 branches consume it via p.delete(:bk2_first_set_mode)"). The controller code at L84/L129/L234 confirms. The `must_haves.truths[0]` describes the SEMANTIC payload (the server-side `bk2_options` hash carries `first_set_mode`), which the flat-key approach satisfies — the controller rebuilds `p[:bk2_options] = { …, "first_set_mode" => first_set_mode }` from the flat key.
