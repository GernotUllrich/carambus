### DR-01 — Platzierung der 4 Anstosser/Modus-Buttons ist falsch

**URL (aktueller Stand):** http://0.0.0.0:3007/locations/0819bf0d7893e629200c20497ef9cfff?data-turbo=false&sb_state=free_game_detail&table_id=4

**Ist:**
- Die 4 Buttons (Spieler A × DZ, Spieler A × SP, Spieler B × DZ, Spieler B × SP) sind im Detail-View angesiedelt (Plan 38.2-02, `app/views/locations/scoreboard_free_game_karambol_new.html.erb`).
- Dort zeigen sie keine Wirkung — der Shootout-Screen ignoriert `bk2_first_set_mode` beim Klick auf das Karambol-geerbte "Spieler A/B stößt an".
- Detail-View mischt *Spielkonfiguration* (Disziplin, Ballziel) mit *Match-Start-Entscheidung* (Anstosser + first_set_mode) — das ist konzeptionell falsch.

**Soll:**
- **Detail-View** legt nur den *generellen Spielmodus* fest:
  - Disziplin (BK2-Kombi oder andere)
  - Ballziel (50 / 60 / 70 — je nach Spielklasse)
  - Max. Stosszahl pro Aufnahme je Satz-Modus (DZ = 2, Serienspiel = 5+1 tentativ — finale Werte siehe DR-02)
  - **Kein** Anstosser, **kein** first_set_mode.
- **Shortcut-Konfiguration** (admin-seitig über `carambus.yml`): identisch zum Detail-View-Output — Disziplin + Ballziel + DZ/SP Max-Werte. Bereits konfigurierbar, kein Änderungsbedarf.
- **Game-Start → Warmup → Shootout-Screen:**
  - Aktuell stehen im Shootout zwei Buttons: "Spieler A stößt an" / "Spieler B stößt an" (Karambol-Erbe).
  - Für BK2-Kombi müssen diese zwei Buttons durch die 4 Buttons aus dem Detail-View ersetzt werden (Spieler A DZ, Spieler A SP, Spieler B DZ, Spieler B SP).
  - Klick auf einen der 4 Buttons: setzt `current_kickoff_player` **und** `bk2_options[:first_set_mode]`, transitioniert TableMonitor auf `playing`, BK2-Scoreboard rendert.

**Scope-Implikation:**
- Plan 38.2-02 Code muss teils rückgebaut werden (4 Buttons aus Detail-View raus).
- Neuer Code: Shootout-Screen-Override für BK2-Kombi (vermutlich in `_show_bk2_kombi.html.erb` oder in einem neuen `_shootout_bk2_kombi.html.erb` Partial).
- Detail-View (und Shortcut-Preset-Schema) braucht neue Felder für DZ-/SP-Max — derzeit ist nur `set_target_points` + (Plan 01) `direkter_zweikampf_max_shots_per_turn` / `serienspiel_max_innings_per_set` im Schema. Überprüfen, ob die UI diese Felder anbietet.

**Reproduktion:**
- Detail-View öffnen, BK2-Kombi wählen → 4 Buttons sind sichtbar (`grep -c 'Spieler [AB]' scoreboard_free_game_karambol_new.html.erb` zählt ≥ 4).
- Einen beliebigen der 4 klicken → Game startet, aber im Shootout-Screen stehen wieder "Spieler A/B stößt an" aus der Karambol-Vorlage.
- Klick dort → Scoreboard rendert mit `first_set_mode = direkter_zweikampf` (Plan 01 Controller-CLAMP-Default), **unabhängig** davon, was im Detail-View geklickt wurde.

### DR-02 — Offene Regelklärung mit Landessportwart

**Ist:**
- Plan 01 hat default-konfigurierbare Werte angelegt: DZ = 2 Stöße, SP = 5 Aufnahmen.
- User-Angabe: "2 für DZ, beliebig oder 5+1 im Serienspiel" — "5+1" legt Bonus-Aufnahme nahe, aktuell nicht im Schema.

**Soll:**
- Nach Rücksprache: finale Werte in `bk2_options`-Schema sichtbar.

- Code-Review-Fund **WR-01**: `ScoreShot#calculate_direct_duel_transitions` hardcodet `next_shots_left_in_turn: turn_ends ? 2 : next_shots`. Wenn Officials `> 2` bestätigen, muss diese Zeile ebenfalls aus `bk2_options` lesen (aktuell scope-geguarded out per D-02).

**Scope-Implikation:**
- Triggert evtl. Schema-Erweiterung + ScoreShot-Augmentation (würde den D-02-Scope-Guard erweichen — bewusste Entscheidung im 38.3-Discuss).

**Status:** blocked-on-user (awaiting Landessportwart sign-off)

### DR-03 — Scoreboard-Layout weicht zu stark vom Karambol-Referenzbild ab

**Referenz:** `/Users/gullrich/Desktop/Bildschirmfoto 2026-04-24 um 13.38.23.png` — Karambol-Mehrsatz-Scoreboard (Christian 0 / Gernot 1 / Freie Partie klein / Tisch 6 / maximal 2 Sätze)

- Nicht responsive — BK2-Scoreboard scrollt auf Tablet/Smartphone-Auflösungen.
- Rahmen-Struktur der Spieler-Pane unklar — zu viele oder zu große Rahmen.

**Soll (Strukturell — 1:1 wie Referenzbild):**
- **Oberhalb der Pane (außerhalb des Rahmens):** Spieler-Name + Satzgewinn-Zähler (z. B. "Christian 0", "Gernot 1") — wie beim Dreiband-Mehrsatz, KEIN eigener Rahmen drumherum.
- **Spieler-Pane-Rahmen (grün = aktiv, grau = inaktiv):** umschließt NUR Score + Spieler-Logo + Aufnahme-Punktzahl + Ziel/Info-Block.
- **Oben links im Pane, groß in rot:** aktuelle **Aufnahme-Punktzahl** (kann negativ werden, siehe DR-04).
- **Pane-Hauptbereich, sehr groß, weiß (aktiv) / gelb (inaktiv):** aktueller Satz-Score (kann negativ werden).
- **Top-right im Pane:** `Ziel: N` (z. B. `Ziel: 50`). **KEINE** `GD:` / `HS:`-Zeilen (nicht anwendbar für BK2).
- **Center-Streifen:**
  - Groß: **Satz N / 3** (statt Karambol-"2 of 20")
  - Darunter: **Phase-Chip** ("Direkter Zweikampf" / "Serienspiel") — farblich gekennzeichnet
  - Darunter: **BK2-Kombi** / **Tisch M** (Disziplin + Tisch wie Karambol)
  - Darunter: evtl. "Best of 3" oder weggelassen
- **Top-Center (außerhalb):** Uhrzeit + Icons (undo / Volt / info / Home / X / redo) — unverändert von Karambol.
- **Bottom-Bar:** siehe DR-04.

**Scope-Implikation:**
- `_scoreboard_bk2_kombi.html.erb` muss visuell nochmal überarbeitet werden — Plan 38.2-03 hat Karambol-Parallelität *inhaltlich*, aber nicht *visuell 1:1*. Tailwind-Klassen + Rahmen-Hierarchie müssen näher ans Referenzbild.
- Responsive: Viewport-Fit ohne Scroll auf 1024×768 Landscape-Tablet als Acceptance-Kriterium.

### DR-04 — Eingabemethode: zwei Varianten nötig (event-basiert vs. aufnahme-basiert)

**Kontext:** User hat zwei unterschiedliche Use-Cases:
1. **Schreiber + Schiedsrichter** (offizielle Turnier-Besetzung): Schiri ruft Einzelstöße, Schreiber erfasst event-basiert → unser aktueller Plan 38.2-04 Ansatz
2. **Spieler selbst-scored** (Club-Alltag, Training): wie beim Eurokegel — nur Aufnahme-Endpunktzahlen oder Zwischensummen

**Ist (Phase 38.2-04):**
- 3-Zeiliges event-basiertes Formular: "Volles Kegelbild" Checkbox, Dropdown `Gefallene Kegel 0..5`, Mittelpin / echter / unechter Karambol-Flags, Durchläufe-Zähler, Fehler-Dropdown, "Stoß erfassen"-Button.
- Button "Stoß erfassen" **funktioniert nicht** (siehe DR-05) — Zustandsübergänge konnten nicht end-to-end getestet werden.

**Soll:**

**Variante A — Event-basiert (Schreiber-Modus), vereinfacht:**
- Statt Dropdown `Gefallene Kegel 0..5`: **Buttons `+1 +2 +3 +4 +5`** (direkt klickbare Kegelzahlen).
- Statt Fehler-Dropdown: **Buttons `-1 -2 -6`** (direkte Fehlerpunkt-Abzüge).
- Durchläufe: als Zähler **n/5** unter der Aufnahme-Punktzahl im Pane (nicht als Eingabefeld in der Bottom-Bar).
- Zeile "Volles Kegelbild vor Stoss" **raus** — Kegel werden immer wieder aufgestellt (ggf. an Ersatzplätzen oder per Neuaufbau).
- Mittelpin / echter / unechter Karambol / Bande: bleiben als Zusatz-Flags (genaue Zuordnung noch zu klären mit Landessportwart + Excel-BK2-Tool-Vergleich).

**Variante B — Aufnahme-basiert (Spieler-Modus, wie Eurokegel/Karambol):**
- Bottom-Bar identisch zu Karambol: `-10 -5 -1 | reset | +1 +5 +10` plus Nummern-Popup (Rechner-Icon).
- Eingabewerte sind Aufnahme-Punktzahlen oder Zwischensummen — **nicht** Einzelstöße.
- **DZ-Modus:** negative Aufnahme-Punktzahl beim Spielerwechsel → positiv dem Gegner gutgeschrieben (seine Aufnahme startet mit dieser Punktzahl).
- **SP-Modus:** Aufnahme-Punktzahl wird zum Satz-Score addiert; auch negative Aufnahmen → Score kann sinken.
- Einziger Karambol-Unterschied: **Aufnahme-Punktzahl UND Satz-Score können negativ sein** (Minus-Vorzeichen, in rot wahrscheinlich).

**Scope-Implikation:**
- **Variante B ist deutlich einfacher zu implementieren** als Variante A — sie ist fast eine 1:1-Karambol-Eingabe mit Negativ-Vorzeichen-Toleranz.
- Empfehlung: **Variante B als Standard ausliefern**, Variante A als späterer optional-Modus. Rollout-Entscheidung in `/gsd-discuss-phase 38.3`.
- Aktueller Plan 38.2-04 Code (Stimulus + 3-Row-Form) wird voraussichtlich **verworfen** zugunsten einer Karambol-parallelen Bottom-Bar mit Negativ-Support.
- Scoring-Logik: `Bk2Kombi::ScoreShot` ist event-basiert gebaut. Für Variante B muss eine neue Service-Methode / neuer Service her, der eine ganze Aufnahme auf einen Schlag auf den Satz-Score anwendet (Karambol-ähnlich).
- DZ-Opponent-Gutschrift bei negativer Aufnahme-Punktzahl ist neue Logik — klar spezifizieren im 38.3-CONTEXT.

**Offene Klärungen (User → BK2-Spieler + Excel-Tool):**
- Welche Rundungsregeln / Grenzwerte gibt es für Aufnahme-Punktzahlen?
- Wie werden Mittelpin / echter / unechter Karambol in Variante B erfasst? (Nur in Protokoll, nicht im Live-Score?)
- Bestätigung des DZ-Opponent-Gutschrift-Mechanismus für negative Aufnahmen.

### DR-05 — "Stoß erfassen"-Button funktioniert nicht

**Ist:**
- User hat TableMonitor per Rails-Console in `playing`-Zustand gesetzt und BK2-Scoreboard im Browser geladen.
- Bottom-Bar rendert (3-Zeilen-Form aus Plan 38.2-04), Eingaben lassen sich vornehmen.
- **Klick auf "Stoß erfassen" zeigt keine Wirkung** — keine Reflex-Antwort, keine Score-Änderung, keine CableReady-Broadcasts.
- Weitere Zustandsübergänge (Aufnahmewechsel, Satz-Ende, Match-Ende) daher nicht testbar.

**Soll:**
- Button triggert `TableMonitorReflex#bk2_kombi_submit_shot` erfolgreich → Score aktualisiert live.

**Root-Cause-Hypothese:**
- Stimulus-Target / Dataset-Key / `data-reflex`-Attribut fehlt, falsch benannt, oder wird von der Alpine-x-data-Rendering-Reihenfolge geschluckt.
- Alternativ: Reflex-Endpoint ist da, aber `TableMonitor.find(element.dataset['id'])` findet den Monitor nicht (falsche Dataset-Quelle).
- System-Test (Plan 38.2-05) hat die Button-Submit-Pfad zwar grundsätzlich exerciert, aber möglicherweise nur in isolierten Rendering-Tests — nicht den vollen Reflex-Round-Trip im Browser.

**Scope-Implikation:**
- Falls Variante B aus DR-04 gewählt wird: der aktuelle Button stirbt sowieso — DR-05 wird dann zu "alte Reflex-Verdrahtung deprecaten".
- Falls Variante A (evtl. zu einem späteren Zeitpunkt): dieser Bug muss diagnostiziert werden — vermutlich `/gsd-debug` Session.

**Reproduktion:**
1. Rails-Console: TableMonitor in `playing` setzen (Snippet aus Chat).
2. `/table_monitors/:id` öffnen.
3. Bottom-Bar-Eingabe machen (z. B. `Gefallene Kegel = 3` wählen).
4. "Stoß erfassen" klicken → nichts passiert.

**Priorität:** niedrig, falls DR-04 Variante B gewählt — sonst blocker.

### DR-06 — Detail-Page sollte Max-Stösse / Max-Aufnahmen konfigurieren, nicht Anstosser

**Vorgänger:** DR-01 (Button-Platzierung). Diese DR ergänzt, was stattdessen in der Detail-Page erscheinen soll.

**Soll (Detail-Page-Felder für BK2-Kombi):**
- Disziplin-Auswahl (BK2-Kombi vs. andere) — bereits da.
- **Ballziel:** Dropdown 50 / 60 / 70 — abhängig von Spielklasse des Turniers. Aktuell ist `set_target_points = 50` hart gesetzt im Default.
- **Max. Stösse pro Aufnahme (DZ-Modus):** Eingabefeld (Default 2). Ergebnis landet in `bk2_options[:direkter_zweikampf_max_shots_per_turn]`.
- **Max. Aufnahmen pro Satz (SP-Modus):** Eingabefeld (Default 5, ggf. "5+1" mit Bonus-Flag — siehe DR-02). Ergebnis landet in `bk2_options[:serienspiel_max_innings_per_set]`.
- **Keine Anstosser-Wahl**, **kein first_set_mode** — die gehören in den Shootout-Screen (DR-01).

**Shortcut-Kopplung:**
- Shortcuts (`/locations/1?sb_state=free_game&table_id=N`) sind in `carambus.yml` konfigurierbar via Admin-UI. Dasselbe Schema wie Detail-Page-Output: Disziplin + Ballziel + DZ-/SP-Max-Werte.
- Admin-UI evtl. erweitern um BK2-Kombi-spezifische Felder (aktuell unklar, ob sie schon da sind).

**Scope-Implikation:**
- UI-Anpassung in `scoreboard_free_game_karambol_new.html.erb` (Detail-Page) — BK2-Branch rausnehmen (4 Buttons weg), BK2-Config-Felder rein.
- `TableMonitorsController#start_game` BK2-Branch: akzeptiert weiterhin die bestehenden Felder, plus neu Max-Stösse / Max-Aufnahmen.
- `config/carambus.yml` Admin-Schema erweitern um DZ/SP-Max für BK2-Presets.
