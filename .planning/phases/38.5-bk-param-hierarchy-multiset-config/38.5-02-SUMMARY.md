## Module API + REGISTERED_PARAMS Contract

**Public surface** (frozen, callers depend on these):

```ruby
BkParamResolver::REGISTERED_PARAMS
  # => [:allow_negative_score_input, :negative_credits_opponent]
BkParamResolver::MULTISET_DEFAULT_DZ_FIRST # => ["bk_2plus", "bk_2"]
BkParamResolver::MULTISET_DEFAULT_SP_FIRST # => ["bk_2", "bk_2plus"]

BkParamResolver.bake!(table_monitor)
# Side-effect: writes data["effective_discipline"] (String) +
#              data["allow_negative_score_input"] (Bool) +
#              data["negative_credits_opponent"] (Bool).
# Does NOT call save!. Idempotent (second call produces identical data).

BkParamResolver.resolve(param, table_monitor:, effective_discipline_form: nil)
# => Bool. First explicitly-set value (data.key?(param.to_s)) walking from
#    highest priority (TableMonitor) down to lowest (Discipline). Falls back
#    to false (D-04) when no level sets it.
```

**Private helpers** (internal, may change):
- `compute_effective_discipline(table_monitor)` — String. Identity for non-BK-2kombi; multiset_components[(set_index - 1) % 2] for BK-2kombi.
- `lookup_discipline(free_game_form)` — Discipline AR record or nil. Uses `TableMonitor::GameSetup::BK_NAME_TO_FORM.invert`.
- `parsed_discipline_data(discipline)` — Hash. JSON.parse with rescue → {}.

## Walk Order Decision + TournamentPlan Reservation

The walk visits levels **highest-priority FIRST** and returns the first level where `lvl.key?(key)` is true:

| # | Level | Source | In Phase 38.5 |
|---|-------|--------|---------------|
| 7 | TableMonitor | `tm.data` | ✓ active (highest) |
| 6 | Detail-Form | (folds into 7) | deferred per D-16 — values land in tm.data via controller params |
| 5 | Quickstart-Preset | (folds into 7) | deferred per D-16 — same as 6 |
| 4 | TournamentMonitor | `tm.tournament_monitor&.data` | ✓ active (slot reserved per D-14, no UI yet) |
| 3 | TournamentPlan | **SKIPPED — no `data` column** | reserved, will need migration in follow-up phase if override needed |
| 2 | Tournament | `tm.tournament_monitor&.tournament&.data` | ✓ active (slot reserved per D-14, no UI yet) |
| 1 | Discipline | `JSON.parse(discipline.data)` | ✓ active (default; lowest) |

Fallback if all levels miss the key: **false** (D-04 — Karambol-Semantik for backward-compat).

**TournamentPlan gap** is the most consequential research finding (RESEARCH.md Pitfall 1, Critical Gap section): `tournament_plans` table has columns `name, rulesystem, players, tables, more_description, even_more_description, executor_class, executor_params, ngroups, nrepeats, created_at, updated_at` — and no `data` column. Per D-14 ("keine Migrationen") the resolver does NOT call `.tournament_plan.data` anywhere; the level is silently skipped with a comment. **Test D1** is the regression guard — it constructs a Tournament+TournamentPlan reference and asserts `assert_nothing_raised { resolve(...) }`.

## D-06 Sparse-Override Regression Guard

The single most important test in this suite is **C1**:

```ruby
test "C1: explicit false at TableMonitor level overrides true at Discipline (D-06 sparse override)" do
  # BK-2plus default at Discipline: allow_negative_score_input=true.
  # TableMonitor explicitly sets it to FALSE.
  tm = build_tm(free_game_form: "bk_2plus", overrides: {"allow_negative_score_input" => false})
  result = BkParamResolver.resolve(:allow_negative_score_input, table_monitor: tm)
  assert_equal false, result,
    "D-06 regression: explicit FALSE at TableMonitor level must override Discipline default TRUE. " \
    "If this test fails, somebody used .present? instead of .key? in the resolver walk."
end
```

The failure message specifically names the bug class (`.present?` vs `.key?`) so a future regression is self-diagnostic.

## bake! Contract

| Property | Behavior | Test |
|----------|----------|------|
| Writes `data["effective_discipline"]` | Always (even for non-BK-2kombi → identity with free_game_form) | F1, F4 |
| Writes each REGISTERED_PARAM as `data[param.to_s]` | Always | F1, F4 |
| Calls `tm.save!` | **NO** — caller is responsible | F3 (reloads from DB, asserts key absent) |
| Idempotent | Yes — second call produces identical data | F2 |

The "does NOT save" property is load-bearing for Plan 05's bake-hook integration: `start_game` and the set-boundary path already call `save!` exactly once after their respective `deep_merge_data!`, so inserting `BkParamResolver.bake!(@tm)` between them must NOT introduce a second save (would double up CableReady broadcasts and redundant TableMonitorJob enqueues).

## Decisions Made

- **Top-level BkParamResolver namespace** (not `Bk2::ParamResolver`): the resolver also services the default-false path for non-BK disciplines, so it's orthogonal to Bk2 scoring. Matches RESEARCH.md recommendation.
- **TournamentPlan silently skipped**: Schema gap accepted (D-14 mandates no migrations). Comment in code names the gap explicitly so a future phase can pick it up. Test D1 is the regression guard.
- **Levels 5 + 6 collapse into Level 7**: D-16 defers UI toggles — Quickstart-Preset and Detail-Form values already land in `tm.data` via controller params, so the TableMonitor level captures them. Documented inline in the resolver.
- **`private_class_method` on the three helpers** (compute_effective_discipline, lookup_discipline, parsed_discipline_data): minimal public surface — bake!, resolve, REGISTERED_PARAMS, MULTISET_DEFAULT_*. Tests use `.send(:compute_effective_discipline, tm)` to reach the helper for Group A — acceptable shortcut for unit testing.
- **F3 reloads from DB instead of `refute tm.persisted?`**: Discovered during Task 2 — TableMonitor.new persists itself implicitly via AASM `state :new, after_enter: :reset_table_monitor` callback that calls `save!` (table_monitor.rb:1593). The resolver's "does NOT save" contract is preserved by reloading from DB and asserting `effective_discipline` is absent on the persisted row. Test comment documents this for future readers.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] TableMonitor.new persists itself (AASM after_enter callback)**
- **Found during:** Task 2 (initial test run had F3 failing with "fresh build_tm TM is not persisted")
- **Issue:** The plan's suggested test pattern `assert refute tm.persisted?` after `BkParamResolver.bake!` doesn't work because `TableMonitor.new` triggers AASM initial-state `:new` → `after_enter :reset_table_monitor` → `save!` (table_monitor.rb:1583–1597). The TM is already persisted before bake! runs.
- **Fix:** Rewrote F3 to call `tm.save!` to establish a clean baseline, run bake!, assert `tm.changed?` is true (proves bake! mutated in-memory), and reload from DB to assert `effective_discipline` is absent on the persisted row (proves bake! did not save). The contract "bake! does NOT save" is verified more rigorously this way than the original `refute persisted?` approach.
- **Files modified:** `test/services/bk_param_resolver_test.rb`
- **Commit:** `417a9a27`

**2. [Rule 3 - Blocking] TableMonitor.new(data: {...}) drops the data Hash**
- **Found during:** Task 2 (initial Group A test run — `compute_effective_discipline` returned nil because `tm.data` was empty hash, not the constructed data)
- **Issue:** `TableMonitor.new(data: {"free_game_form" => "karambol"})` creates a TM with `data == {}` (the AASM initial-state callback `reset_table_monitor` calls `assign_attributes(data: {})` and `save!`, wiping the constructor-passed data). All Group A and several other tests would fail.
- **Fix:** `build_tm` helper now uses `tm = TableMonitor.new; tm.data = data; tm` — assigning data AFTER construction lands after the AASM callback fires. Documented inline in the helper.
- **Files modified:** `test/services/bk_param_resolver_test.rb`
- **Commit:** `417a9a27` (folded into Task 2 commit; the helper was authored with this pattern from the start of Task 2 after the discovery)
