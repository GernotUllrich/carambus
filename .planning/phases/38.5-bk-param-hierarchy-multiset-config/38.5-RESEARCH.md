## Summary

Phase 38.5 makes two orthogonal BK scoring booleans (`allow_negative_score_input`, `negative_credits_opponent`) first-class data values resolved from a 7-level hierarchy chain and baked into `TableMonitor.data` at two fixed lifecycle points. It also converts BK-2kombi from a "peer Discipline" to a "multiset config record" that routes per-set to BK-2plus or BK-2. No DB migrations. No UI toggles (derived-only).

**Primary recommendation:** Place `BkParamResolver` as a top-level `app/services/bk_param_resolver.rb`. Bake at (1) end of `GameSetup#perform_start_game` after `deep_merge_data!` + before `save!`, and (2) inside `ResultRecorder#perform_switch_to_next_set` after `deep_merge_data!(options)` + before `save!` guarded by `bk2_kombi?` check. ScoreEngine predicate rewrite is a 2-line body-swap with no call-site changes.

**Codebase Status:** No existing usage of `allow_negative_score_input` or `negative_credits_opponent` anywhere in `app/`. All changes are additive (new service + seed update + 2-line predicate swap). [VERIFIED: grep across app/]

## Architecture Patterns

### Resolver Hierarchy Walk

The 7-level hierarchy maps to these concrete lookups, in order from lowest-priority (top) to highest (bottom):

| Level | Ruby access from a `TableMonitor` | Notes |
|-------|------------------------------------|-------|
| 1. Discipline | `Discipline.find_by(name: @tm.data["playera"]["discipline"])` | Raw JSON text — must `JSON.parse`; see "Data Access" section below |
| 2. Tournament | `@tm.tournament_monitor&.tournament&.data` | `serialize :data, coder: JSON, type: Hash` → string keys |
| 3. TournamentPlan | **MISSING COLUMN — see critical gap below** | `tournament_plans` table has NO `data` column |
| 4. TournamentMonitor | `@tm.tournament_monitor&.data` | `serialize :data, coder: JSON, type: Hash` → string keys |
| 5. Quickstart-Preset | `Carambus.config.quick_game_presets` entries | YAML config (OpenStruct/Hash) — no DB, no `data.key?` semantics |
| 6. Detail-Form | Submitted via `params` → written into `@tm.data` before bake runs | In training, user selections land in `@tm.data["bk2_options"]`; not a separate level |
| 7. TableMonitor | `@tm.data` | `serialize :data, coder: JSON, type: Hash` → string keys |

**Training mode (no TournamentMonitor):** `@tm.tournament_monitor` is `nil`. Levels 2–4 are skipped entirely — `@tm.tournament_monitor&.anything` returns `nil` safely. Level 1 (Discipline) is reached via `Discipline.find_by(name: @tm.data["playera"]["discipline"])`.

**How to reach the Discipline AR record:** `TableMonitor#discipline` (line 608–609) returns `data["playera"]["discipline"]` as a String (e.g., `"BK2-Kombi"`), NOT a Discipline AR object. The resolver must call `Discipline.find_by(name: @tm.data.dig("playera", "discipline"))`. [VERIFIED: table_monitor.rb:608]

**How to reach the current set index (for multiset_components mapping):**
`set_index = Array(@tm.data["sets"]).length + 1`
This is the same formula used in `bk2_kombi_current_phase` (table_monitor.rb:1112). `data["sets"]` is pushed on each set close by `ResultRecorder#perform_save_current_set` (result_recorder.rb:179). Before any set closes, `data["sets"]` is nil/absent → `set_index = 1`. [VERIFIED: table_monitor.rb:1112, result_recorder.rb:179]

Critical: `data` column serialization is NOT uniform across the hierarchy.

| Model | Serialization | Key type returned | `data.key?("foo")` works? |
|-------|---------------|-------------------|---------------------------|
| `Discipline` | **None** (raw `text`, no `serialize`) | String (after manual `JSON.parse`) | Only after parsing |
| `Tournament` | `serialize :data, coder: JSON, type: Hash` | String | YES |
| `TournamentPlan` | **No `data` column** | — | N/A |
| `TournamentMonitor` | `serialize :data, coder: JSON, type: Hash` | String | YES |

[VERIFIED: discipline.rb:397–428]

**Discipline.data uses string keys.** The `seed_bk2_disciplines.rb` script writes `attrs[:data].to_json` — Ruby symbol keys → JSON string keys → string keys on read. [VERIFIED: script/seed_bk2_disciplines.rb:36]

### Effective Discipline Computation

For BK-2kombi, `effective_discipline` per set:
```ruby
# multiset_components order (D-01):
# first_set_mode="direkter_zweikampf" → ["bk_2plus", "bk_2"]
# first_set_mode="serienspiel"        → ["bk_2", "bk_2plus"]

set_index = Array(@tm.data["sets"]).length + 1
components = @tm.data["multiset_components"] ||
  (first_set_mode == "direkter_zweikampf" ? ["bk_2plus", "bk_2"] : ["bk_2", "bk_2plus"])
effective_disc_form = components[(set_index - 1) % 2]  # 2-element cycle
```

For non-BK-2kombi: `effective_discipline = data["free_game_form"]` (identity).

After resolving `effective_discipline`, the resolver fetches the Discipline record for the **effective** form (e.g., `Discipline.find_by(name: "BK-2plus")`), reads its `data["allow_negative_score_input"]` etc., and walks up the remaining hierarchy. [VERIFIED: table_monitor.rb:1109–1113, advance_match_state.rb:89–92]

## Common Pitfalls

### Pitfall 1: TournamentPlan Has No `data` Column

**What goes wrong:** Resolver tries `@tm.tournament_monitor&.tournament&.tournament_plan&.data&.key?(...)` → `NoMethodError: undefined method 'data'` because the column does not exist.
**Why it happens:** CONTEXT.md D-05 lists TournamentPlan as a hierarchy level, but the schema has no `data` column on `tournament_plans`.
**How to avoid:** Skip TournamentPlan level in the resolver — return `nil` immediately for that level. Document the gap.
**Warning signs:** Any code that calls `.tournament_plan.data` will raise. [VERIFIED: db/schema.rb:1255–1268]

### Pitfall 2: Discipline.data Is NOT Auto-Deserialized

**What goes wrong:** `discipline.data["allow_negative_score_input"]` raises `NoMethodError: undefined method '[]' for String`.
**Why it happens:** `disciplines` table has no `serialize :data` declaration in `discipline.rb`. The `data` attribute is a raw `text` column; AR returns a String.
**How to avoid:** Always parse: `JSON.parse(discipline.data) rescue {}`. Follow the `ballziel_choices` / `nachstoss_allowed?` pattern (discipline.rb:408–428).
**Warning signs:** `discipline.data.class` returns `String` not `Hash`. [VERIFIED: discipline.rb — no serialize declaration; db/schema.rb:232]

### Pitfall 3: Resolver Bake Must Run AFTER `free_game_form` Is Written

**What goes wrong:** `BkParamResolver.bake!(@tm)` called before `@tm.deep_merge_data!(result)` → `data["free_game_form"]` is nil → resolver looks up wrong Discipline (or no Discipline).
**Why it happens:** `free_game_form` is set inside `build_result_hash` which feeds `deep_merge_data!`. The `initialize_game` call before it does NOT set `free_game_form` reliably for the training path.
**How to avoid:** Insert bake call after line 343 (`deep_merge_data!(result)`) and before line 345 (`save!`) in `perform_start_game`. [VERIFIED: game_setup.rb:339–345]

### Pitfall 4: Set-Boundary Bake Must Guard Against Match-End

**What goes wrong:** `BkParamResolver.bake!` called at match end (when `perform_switch_to_next_set` is called but match is finishing) → resolver tries `Discipline.find_by(name: ...)` on a potentially stale `@tm`.
**Why it happens:** `perform_evaluate_result` (result_recorder.rb:288–341) calls `perform_switch_to_next_set` at line 314 and 333/340. When `max_number_of_wins >= sets_to_win`, the match may end; `perform_switch_to_next_set` can still be called for the automatic next-set path even at match end.
**How to avoid:** Guard: `if @tm.data["free_game_form"] == "bk2_kombi"` — this is already BK-2kombi specific. Additionally verify `@tm.data["playera"]` is present before baking. The `!!data["negative_credits_opponent"]` fallback in ScoreEngine ensures `false` for nil/missing keys, so even a missed bake is safe at match end.
[VERIFIED: result_recorder.rb:288–341]

### Pitfall 5: String vs Symbol Keys in `data.key?`

**What goes wrong:** `data.key?(:allow_negative_score_input)` returns `false` even when `data["allow_negative_score_input"]` is set.
**Why it happens:** `serialize :data, coder: JSON, type: Hash` on Tournament, TournamentMonitor, TableMonitor always yields string keys. Symbol key lookups silently miss the value.
**How to avoid:** Use string keys everywhere at `data[]` boundaries. D-07 says Constants use symbols internally (`REGISTERED_PARAMS = [:allow_negative_score_input, ...]`), but convert to strings when calling `data.key?(param.to_s)` or `data[param.to_s]`. [VERIFIED: tournament.rb:110, tournament_monitor.rb:48, table_monitor.rb:290]

### Pitfall 6: `allow_negative_score_input: false` Must Override, Not Pass-Through

**What goes wrong:** A Tournament explicitly overrides with `false` (to disable negative scores), but `false.present?` returns `false`, so the code treats it as "not set" and falls through to Discipline default (`true`), silently ignoring the explicit override.
**Why it happens:** D-06 requires `data.key?("allow_negative_score_input")` not `.present?`. If the resolver uses `.present?` instead of `.key?`, an explicit `false` override at any level is invisible.
**How to avoid:** Strictly use `data.key?("allow_negative_score_input")` for the "is this level explicitly setting the param?" check. `data["allow_negative_score_input"]` may be `false` — that is a valid override meaning "force false here".

## Code Examples

### BkParamResolver — Recommended Structure

```ruby
# Source: CONTEXT.md specifics + codebase patterns
# app/services/bk_param_resolver.rb
# frozen_string_literal: true

module BkParamResolver
  REGISTERED_PARAMS = %i[allow_negative_score_input negative_credits_opponent].freeze

  # Resolve all REGISTERED_PARAMS through the hierarchy and write results
  # (plus effective_discipline) into table_monitor.data.
  # Does NOT save — caller is responsible for tm.save!
  def self.bake!(table_monitor)
    eff_disc_form = resolve_effective_discipline(table_monitor)
    table_monitor.data["effective_discipline"] = eff_disc_form

    REGISTERED_PARAMS.each do |param|
      table_monitor.data[param.to_s] = resolve(param, table_monitor: table_monitor,
                                                effective_discipline_form: eff_disc_form)
    end
  end

  # Walk hierarchy for a single param. Returns first explicitly-set value,
  # falls back to false (D-04).
  def self.resolve(param, table_monitor:, effective_discipline_form: nil)
    eff_form = effective_discipline_form || resolve_effective_discipline(table_monitor)
    discipline_record = Discipline.find_by(name:
      TableMonitor::GameSetup::BK_NAME_TO_FORM.invert[eff_form])

    levels = [
      discipline_data(discipline_record),            # Level 1 — lowest priority
      table_monitor.tournament_monitor&.tournament&.data,          # Level 2
      nil,                                           # Level 3 TournamentPlan: no data column
      table_monitor.tournament_monitor&.data,        # Level 4
      nil,                                           # Level 5 Quickstart: derived-only (D-16)
      nil,                                           # Level 6 Detail-Form: already in tm.data
      table_monitor.data                             # Level 7 — highest priority
    ]

    key = param.to_s
    # Walk highest priority (Level 7) first, then down
    levels.reverse.each do |lvl|
      next if lvl.nil?
      return lvl[key] if lvl.key?(key)
    end

    false  # D-04 fallback
  end

  private

  def self.resolve_effective_discipline(table_monitor)
    form = table_monitor.data["free_game_form"]
    return form unless form == "bk2_kombi"

    # D-01: multiset_components per set index
    first_mode = table_monitor.data.dig("bk2_options", "first_set_mode").presence ||
                 "direkter_zweikampf"
    components = table_monitor.data["multiset_components"] ||
                 (first_mode == "direkter_zweikampf" ? ["bk_2plus", "bk_2"] : ["bk_2", "bk_2plus"])
    set_index = Array(table_monitor.data["sets"]).length + 1
    components[(set_index - 1) % 2]
  end

  def self.discipline_data(discipline)
    return {} if discipline&.data.blank?
    JSON.parse(discipline.data)
  rescue JSON::ParserError
    {}
  end
end
```

## Open Questions

1. **TournamentPlan data column gap**
   - What we know: `tournament_plans` table has no `data` column; CONTEXT.md D-05 lists it as a hierarchy level.
   - What's unclear: Is the intent to add the column (DB migration) or to silently skip the level?
   - Recommendation: Skip TournamentPlan in the resolver for Phase 38.5. Add comment: "Level 3 TournamentPlan: no data column in schema — add migration in Folge-Phase if override needed."

2. **Lazy bake vs. migration for in-flight TableMonitors**
   - What we know: In-flight BK-* TableMonitors lack the two new keys; `!!nil == false` gives safe fallback but breaks intended BK-2plus DZ scoring until rebake.
   - What's unclear: Are there active BK-2kombi matches on BCW at deploy time?
   - Recommendation: Implement lazy bake in `ScoreEngine#add_n_balls` — if `free_game_form` is BK-family AND `data["effective_discipline"]` is absent, call `BkParamResolver.bake!(table_monitor_reference)`. Requires ScoreEngine to have a reference to the TM (it currently only has the `data` hash). Alternatively: one-line migration task `TableMonitor.where("data LIKE '%bk2_kombi%' OR data LIKE '%bk_2%' OR data LIKE '%bk50%' OR data LIKE '%bk100%'").find_each { |tm| BkParamResolver.bake!(tm); tm.save! }`.

3. **`Bk2::AdvanceMatchState` naming vs. actual hook location**
   - What we know: CONTEXT.md D-03 says "bake in `Bk2::AdvanceMatchState`". The actual set-open transition is in `ResultRecorder#perform_switch_to_next_set`.
   - Recommendation: Either (a) add `Bk2::AdvanceMatchState.rebake_at_set_open!(tm)` class method that calls `BkParamResolver.bake!(tm)`, called from `ResultRecorder`; or (b) call `BkParamResolver.bake!(tm)` directly from `ResultRecorder` guarded by `bk2_kombi?`. Option (a) keeps all BK logic in the Bk2 namespace, is more testable.
