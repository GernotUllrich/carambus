## Delegation Rationale (D-03 Mental Model + Research Finding 3)

CONTEXT.md D-03 says "rebake in `Bk2::AdvanceMatchState`" — but RESEARCH.md finding 3 confirms the actual set-open file path is `ResultRecorder#perform_switch_to_next_set`. After 38.4 cleanup, `Bk2::AdvanceMatchState` was trimmed to a config seeder only.

**Resolution:** keep the orchestration named under the Bk2 namespace via a thin delegate (`Bk2::AdvanceMatchState.rebake_at_set_open!(tm)` → 1-line call to `BkParamResolver.bake!`), then call it from `ResultRecorder`. Preserves:

- The D-03 mental model (BK lifecycle stays under Bk2:: namespace; readers and future maintainers see the namespace correctly)
- The actual file path semantics (ResultRecorder remains the hook site, no logic redirection)
- A natural home for future enhancements (CableReady event on DZ/SP flip, audit logging) without further refactoring

This is exactly the extend-before-build shape from `.agents/skills/extend-before-build/SKILL.md` — small guard on existing structures, no parallel state machine. The 8-round BK2 experiment cautionary tale validates this approach (a new state machine would have over-engineered the 2-line behavior change).

## BK-2kombi Guard Rationale

The guard `if @tm.data["free_game_form"] == "bk2_kombi"` is at the consumer (`ResultRecorder`), NOT inside `rebake_at_set_open!`. Reasons:

- For non-BK-2kombi families (BK-2, BK50, BK100, BK-2plus, karambol, snooker, pool), `effective_discipline` is stable across sets (it equals `free_game_form` per BkParamResolver.compute_effective_discipline). Re-baking would write the same values it wrote at start_game — a no-op.
- The guard makes intent explicit: future readers see exactly which family triggers per-set re-bake.
- If a future BK family (e.g., BK3 with 3-mode multiset) needs per-set re-bake, the guard updates here only.
- The match-end safety concern (Pitfall 4) is implicitly addressed: at match-end for non-kombi the guard skips; for BK-2kombi the bake is idempotent and safe even on the unreachable code path.

Pure delegate design (no guard inside `rebake_at_set_open!`) preserves the method as a 1-line API that future callers can reuse without re-implementing the guard.

### Stub-and-restore Pattern (Load-bearing Test Decision)

The first test of advance_match_state_test.rb stubs `BkParamResolver.bake!` to record arguments. The original draft used `remove_method` in `ensure` to undo the stub — but `define_singleton_method` doesn't add a method that `remove_method` can cleanly remove without clobbering the original module method. This caused test 2 (idempotency, calls real `bake!`) to crash with `NoMethodError: undefined method 'bake!' for BkParamResolver:Module`.

**Fix (Rule 1 - Bug):** save the original method via `.method(:bake!)` BEFORE stubbing, then in `ensure` re-bind via `BkParamResolver.define_singleton_method(:bake!, original)`. The original method is preserved across the test boundary.

This pattern is now used in all 3 of the 4 new tests that stub a singleton method (test 1 of advance_match_state, both tests of result_recorder).

## Decisions Made

- **rebake_at_set_open! lives on Bk2::AdvanceMatchState:** preserves D-03 mental model (BK lifecycle in Bk2:: namespace) while ResultRecorder remains the actual hook site. Thin one-line delegate satisfies both perspectives.
- **BK-2kombi guard at the call site, not inside the delegate:** explicit intent at the consumer location; rebake_at_set_open! stays a pure delegate; future BK families update the guard in one place.
- **Stub-and-restore via .method(:name) + define_singleton_method(name, original):** load-bearing test pattern; avoids `remove_method` clobbering real module methods between sibling tests.
- **Idempotency test exercises the real BkParamResolver.bake! end-to-end:** validates D-01 (multiset_components) + D-02 (effective_discipline) + D-03 (re-bake hook timing) at the unit-test level using the disciplines fixture.
