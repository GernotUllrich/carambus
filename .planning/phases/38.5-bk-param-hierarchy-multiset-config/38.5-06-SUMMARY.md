   - Walks levels 7→4→2→1: TableMonitor (no key) → tournament_monitor=nil (skip) → tournament=nil (skip) → Discipline level returns true / true.

## No-Lazy-Bake Decision (Open Question 2 closed)

`38.5-RESEARCH.md` Open Question 2: "Lazy bake vs. migration for in-flight TableMonitors". Plan 06 records the decision: **NEITHER**.

### Rationale (full text in 38.5-DEPLOYMENT-RUNBOOK.md)

1. **D-15 forbids migration code.** Phase 38.5 ships zero `db/migrate/*` files for data backfill.
2. **Lazy bake in ScoreEngine would require an out-of-scope refactor.** ScoreEngine receives a `data` Hash by reference and a `Discipline` AR object, but no `TableMonitor` AR reference (`table_monitor.rb:405`). Adding a lazy bake on first `add_n_balls` would require plumbing the TM reference through ScoreEngine's constructor (touches every caller — out of scope) or a Thread-local hack (anti-pattern).
3. **Pre-Phase-38.5 in-flight BK-* matches are SAFE.** They lack the new `data["allow_negative_score_input"]` / `data["negative_credits_opponent"]` keys → ScoreEngine `!!data["..."]` = `false` per D-04 → identical to today's pre-38.5 behaviour (which had the latent bugs D-11 / D-12). **No regression** because the latent-bug behaviour is exactly today's behaviour.
4. **Next `start_game` on each BK-* TableMonitor bakes correctly.** No data lost; no manual migration needed.

### Consequence in the field

During the sync window (carambus_api seed → Version sync → BCW), in-flight matches behave **identically to today**. After sync arrives, the next `start_game` writes the correct keys. From that point, D-11 / D-12 fixes activate.

**Operator action required:** None for in-flight matches. Just deploy and let the bake-hooks fire on natural lifecycle events.

- **Service-level dispatch over real DOM clicks:** The plan's `<action>` block suggested clicking through `click_negative_three_button_for_player_a` etc., but explicitly noted "If a particular DOM interaction (e.g., -3 button) does not exist in the Karambol-parallel bottom bar after Phase 38.3, the test executor MUST stop and verify the actual DOM structure before fabricating a click target." End-to-end coverage at the data-path level WITHOUT JS-stack flake. Documented in test docstring.
- **No `visit table_monitor_path(tm)` in the new tests:** A fresh `TableMonitor.create!` without an associated Game record causes the show action to redirect (`table_monitors_controller.rb:48`). The existing `create_bk_family_table_monitor` helper at line 1391 also has no Game and is used WITHOUT visit by all 9 service-level tests. Plan 06 mirrors the same. UI shape is covered by 7+ existing visit-based tests.
- **`innings_redo_list[-1]` semantics:** Tested live during initial test run that `add_n_balls` does NOT append to `innings_redo_list`; it accumulates in the LAST element. So a single -2 input from 0-base writes -2 to `innings_redo_list[-1]` (signed-add path) or 0 (clamped path) — that's the binary signal the test asserts on. Adjusted the D-12 test's assertion accordingly during initial run; documented inline in test code.
- **Bk2::CommitInning errors deferred:** 9 pre-existing tests in `bk2_scoreboard_test.rb` reference the removed `Bk2::CommitInning` constant (deleted in commit `4c3abdf4`). Per Plan 06 scope boundary, they're out of scope. Recorded in `deferred-items.md` for a future test-suite hygiene task.
