## Pre-discuss conversation summary

User identifizierte: "die differenzierung der Negativ-Werte ist noch nicht correkt.
überall müssen negative werte zugelassen werden. überall bleiben die negativen werte
auf der Seite des Spielers - nur bei BK-2plus wird der Betrag dem Gegner zugeschrieben."

Folgte: "es sind zwei parameter — 1. erlaube negative Werte beim Score in der Aufnahme
und im Spiel; 2. Übertrage positiv an Gegner oder ziehe vom eigenen Score ab."

Plus Hierarchie-Anforderung: "Für alle Parameter in Carambus sollte eine Hierarchie
gelten — Disziplin → Tournament → TournamentPlan → TournamentMonitor → Quickstart →
Detail-Page → TableMonitor."

Korrektur mid-discuss: "bk-2kombi ist ja in dem Sinne keine eigene Disziplin. Es ist
ein Name für eine spezielle Mehrsatzkonfiguration. negative_credits_opponent=false
gilt in dem Fall z.B. nicht für alle phasen - nur für die BK-2 Phase."

Vor-Discuss-Decision (in Chat): Option 1 (Effective-Discipline-Switching pro Satz).

## BK-2kombi-Modellierung

| Option | Description | Selected |
|--------|-------------|----------|
| Discipline 107 bleibt | BK-2kombi behalten + data[:multiset_phase_map] Felder. Resolver lookup nach mapping. Minimal disruptive. | |
| Discipline 107 entärrt | BK-2kombi hört auf Discipline zu sein. Neue MatchConfiguration/MultisetConfiguration. start_game wählt Discipline=BK-2plus oder BK-2 pro Satz. Sauberer aber Migration für 4 in-flight Stellen. | |
| Hybrid | Discipline 107 bleibt + data[:multiset_components]=["bk_2plus","bk_2"]. Pro Satz löst sich in eine effective_discipline auf. start_game speichert effective_discipline auf TableMonitor. bk2_state.next_set tauscht. | ✓ |

**User's choice:** Hybrid
**User's notes:** "tendiere zu hybrid. Scheint auch am einfachsten realisierbar, oder?"

## Resolver-Timing

| Option | Description | Selected |
|--------|-------------|----------|
| Bake-once + Set-Boundary | Resolver läuft (a) bei start_game, (b) am Satzwechsel in Bk2::AdvanceMatchState. Werte stehen in data[] eingefroren. ScoreEngine schnell + ohne Resolver-Logik. | ✓ |
| Live pro Score-Event | Resolver bei jedem Punkt-Eintrag. Kein Cache. Immer korrekt, mehr Aufwand. | |
| Bake-once strikt | Nur start_game, fix. Würde Hybrid-BK-2kombi-Modellierung untergraben. | |

**User's choice:** Bake-once + Set-Boundary
