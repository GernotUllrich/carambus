<objective>
Phase 38.5 D-03 + D-11: wire BkParamResolver.bake! into the two lifecycle hooks.

Hook 1 (start_game initial bake): writes effective_discipline + both BK params into
TableMonitor.data when a game is started. Produces correct values for ALL
matches — BK-* gets the resolver-derived defaults, non-BK gets fallback false +
identity effective_discipline.

Hook 2 (set-boundary re-bake for BK-2kombi): re-runs the resolver when a new set
opens. Only relevant for BK-2kombi where effective_discipline alternates between
bk_2plus and bk_2 per set. For non-BK-2kombi the bake would be a no-op, so the
call is guarded.

Critical research finding 3: CONTEXT.md D-03 says "rebake in Bk2::AdvanceMatchState"
but the actual set-open code path lives in ResultRecorder#perform_switch_to_next_set
(result_recorder.rb:202-266). After the 38.4 cleanup, AdvanceMatchState was trimmed
to a config seeder. Resolution: keep the orchestration named under the Bk2 namespace
(Bk2::AdvanceMatchState.rebake_at_set_open!(tm) — class method that delegates
to BkParamResolver.bake!), then call it from ResultRecorder. Preserves the
D-03 mental model AND lands the call at the correct file path.

extend-before-build skill: this is a small guard on existing structures, NOT a
new state machine. ResultRecorder keeps its existing perform_switch_to_next_set
flow; AdvanceMatchState gains ONE new class method orthogonal to its existing
initialize_bk2_state!. No parallel lifecycle.

Output: 3 source files modified + 2 test files updated.
</objective>
