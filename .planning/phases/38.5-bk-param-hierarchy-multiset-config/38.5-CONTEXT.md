# Phase 38.5: BK-Param-Hierarchy + Multiset-Config - Context

**Gathered:** 2026-04-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Two orthogonal BK scoring parameters become first-class, configurable, and resolvable
via a 7-level hierarchy chain:

1. **`allow_negative_score_input`** (Bool) — negative Werte sind in Aufnahme/Spiel als
   Score-Eingabe zugelassen (kein Clamp auf 0).
2. **`negative_credits_opponent`** (Bool) — wenn TRUE, wird der absolute Betrag einer
   negativen Eingabe positiv beim Gegner verbucht; eigener Score bleibt unverändert.
   Wenn FALSE, wird der negative Wert signed dem eigenen Spieler addiert.

Resolver-Kette (oben = Default, unten = override):
Discipline → Tournament → TournamentPlan → TournamentMonitor → Quickstart-Preset →
Detail-Form → TableMonitor.

**BK-2kombi-Architektur-Korrektur:** BK-2kombi wird zur **Hybrid-Multiset-Config**.
Discipline 107 (BK-2kombi) bleibt bestehen, trägt aber neu `data[:multiset_components]
= ["bk_2plus","bk_2"]` (oder umgekehrt je first_set_mode). Pro Satz löst sich das in
eine echte "effective discipline" auf (BK-2plus oder BK-2), die für die Param-Auflösung
herangezogen wird. Damit wird BK-2kombi konzeptuell vom "eigenen Regelset" zum "Wechsel
zwischen zwei BK-Disziplinen pro Satz".

**Scope dieser Phase:**
- **Training-Pfad** komplett: Discipline → Quickstart-Preset → Detail-Form → TableMonitor.
- **Resolver-Infrastruktur** alle 7 Ebenen vorbereitet (Storage-Slots auf
  Tournament/TournamentPlan/TournamentMonitor sparse/default-leer; Resolver-Logik
  durchgängig).
- **Latente Bugs als Fall-out gefixt:**
  - BK-2kombi DZ-Phase schreibt heute negatives signed beim eigenen Spieler statt beim
    Gegner (`bk_credit_negative_to_opponent?` matcht nur `bk_2plus`).
  - BK-2 / BK50 / BK100 clampen heute negative Werte auf 0 (`allow_negative_scores?`
    matcht nur `bk2_kombi`).

**Out of scope:**
- **Tournament-Pfad UI:** Override-Felder in Tournament-Wizards / TournamentPlan-Editor /
  TournamentMonitor-Form. Folge-Phase. Storage-Slots werden hier vorbereitet, UI nicht.
- **Generischer Param-Resolver für ALLE Params** (balls_goal, sets_to_win, …):
  Infrastruktur ist generisch-genug aufgebaut, aber nur diese 2 Params werden registriert
  und verdrahtet. Weitere Params kommen in eigenen Phasen.
- UI-Toggles für die zwei Flags im Detail-Form / Quickstart-Preset-Editor: Derived-only
  in dieser Phase.

</domain>

<decisions>
## Implementation Decisions

### BK-2kombi-Modellierung

- **D-01:** **Hybrid.** Discipline.find(107) BK-2kombi bleibt als Discipline-Record.
  Erweiterung: `data[:multiset_components]` Array mit den beiden Phase-Disziplin-Keys.
  Reihenfolge richtet sich nach `bk2_options[:first_set_mode]`:
  - `first_set_mode=direkter_zweikampf` → `["bk_2plus","bk_2"]`
  - `first_set_mode=serienspiel` → `["bk_2","bk_2plus"]`
- **D-02:** **TableMonitor.data["effective_discipline"]** (String, einer von
  `bk_2plus` / `bk_2` / sonstige `free_game_form`-Werte). Wird vom Resolver
  pro Satz geschrieben. Für Nicht-BK-2kombi-Disziplinen ist
  `effective_discipline == data["free_game_form"]` (Identität).

### Resolver-Timing (Bake-once + Set-Boundary)

- **D-03:** **Resolver läuft an exakt zwei Punkten:**
  1. `start_game` (initial bake): setzt `effective_discipline`, `allow_negative_score_input`,
     `negative_credits_opponent` in `TableMonitor.data` aus der Hierarchie.
  2. `Bk2::AdvanceMatchState` am Satzwechsel (re-bake): aktualisiert
     `effective_discipline` für den neuen Satz und re-resolved die zwei Flags. Nur
     für BK-2kombi-Matches relevant — andere Disziplinen ändern `effective_discipline`
     nicht.
- **D-04:** **ScoreEngine liest aus data[]**, nicht aus dem Resolver direkt. Predicates
  `allow_negative_scores?` und `bk_credit_negative_to_opponent?` werden auf
  `data["allow_negative_score_input"]` / `data["negative_credits_opponent"]`
  umgestellt. Fallback bei nil/missing-key: `false` (= Karambol-Semantik).

### Param-Storage-Shape

- **D-05:** **Flacher Top-Level-Slot pro Param** auf jeder der 5 Ebenen
  (`Discipline.data`, `Tournament.data`, `TournamentPlan.data`, `TournamentMonitor.data`,
  `TableMonitor.data`):
  - `data["allow_negative_score_input"]` (Bool oder nicht-gesetzt)
  - `data["negative_credits_opponent"]` (Bool oder nicht-gesetzt)
- **D-06:** **Sparse override semantics.** Schlüssel ist nur gesetzt, wenn die Ebene
  explizit override willst. Nicht-gesetzt = "fall through to upper level". Resolver
  prüft pro Ebene `data.key?(param_name)`, NICHT `data[param_name].present?`
  (sonst wäre `false` nicht von "nicht gesetzt" unterscheidbar).

### Param-Namen

- **D-07:** Konkrete Bezeichner: `allow_negative_score_input` und
  `negative_credits_opponent`. Konstante `BkParamResolver::REGISTERED_PARAMS`
  (oder ähnlich) listet sie für generische Iteration.

### Defaults-Migration (Discipline-Seed-Update)

- **D-08:** Discipline-Defaults seeden via Update der zentralen Discipline-Records
  (gleicher Pfad wie 38.4 — central record + sync via `Version#update_from_carambus_api`,
  D-05 von 38.4 sync-bug bereits gefixt):
  - **BK-2plus:** `data["allow_negative_score_input"]=true`, `data["negative_credits_opponent"]=true`
  - **BK-2:** `data["allow_negative_score_input"]=true`, `data["negative_credits_opponent"]=false`
  - **BK50, BK100:** `data["allow_negative_score_input"]=true`, `data["negative_credits_opponent"]=false`
  - **BK-2kombi (id 107):** *KEINE* Werte für die zwei Params in `data[]` — Resolver
    benutzt für BK-2kombi-Matches die `effective_discipline`-Auflösung (Discipline-Lookup
    nach `bk_2plus` oder `bk_2`). Plus: `data["multiset_components"]=["bk_2plus","bk_2"]`
    Default; first_set_mode-spezifische Reihenfolge wird im Resolver/start_game-Bake
    angewandt.
  - **Karambol, Snooker, Pool, Biathlon, etc.:** unverändert (kein neuer Schlüssel) —
    Fallback `false`/`false` per D-04 erhält das heutige Verhalten exakt.

### ScoreEngine-Integration

- **D-09:** Body von `score_engine.rb#allow_negative_scores?` (line 1308) wird zu
  `!!data["allow_negative_score_input"]`. Body von `bk_credit_negative_to_opponent?`
  (line 1315) wird zu `!!data["negative_credits_opponent"]`. Die alten free_game_form-
  String-Equality-Checks werden ersetzt.
- **D-10:** Drei betroffene Codestellen in score_engine.rb sind durch D-09 automatisch
  korrekt:
  - line 84 negative-input-Gate (live add)
  - line 148 negative-Clamp auf 0 (live add)
  - line 706 protocol-modal Validierung "Negative Punktzahlen sind nicht erlaubt"

### Latente-Bugs als Fall-out

- **D-11:** **BK-2kombi DZ-Phase credit-opponent** wird durch D-08 + D-02 + D-09
  automatisch korrigiert: am Satzanfang setzt der Resolver `effective_discipline=bk_2plus`
  für DZ-Sätze, liest BK-2plus-Defaults `allow_negative_score_input=true,
  negative_credits_opponent=true`, schreibt sie in TableMonitor.data, ScoreEngine routet
  korrekt zum Gegner.
- **D-12:** **BK-2/BK50/BK100 negative-Clamp** wird durch D-08 + D-09 automatisch
  korrigiert: Discipline-Default `allow_negative_score_input=true`, ScoreEngine clampt
  nicht mehr.
- **D-13:** **Regression-Tests** sichern beides explizit ab — RED-Test vor Fix für
  beide Bugs (Charakterisierung erst aktuelles falsches Verhalten, dann nach Migration
  GREEN mit korrektem Verhalten).

### Scope Boundaries

- **D-14:** **Training-Pfad zwingend, Tournament-Pfad vorbereitet**:
  - Storage-Slots auf Tournament/TournamentPlan/TournamentMonitor werden in Phase 38.5
    angelegt (in `data[]`-Schema dokumentiert), aber leer. Es gibt kein UI um sie zu
    setzen.
  - Resolver-Logik geht alle 7 Ebenen durch (Discipline → Tournament → TournamentPlan
    → TournamentMonitor → Quickstart-Preset → Detail-Form → TableMonitor).
  - In Trainings-Matches (kein TournamentMonitor) sind Tournament/TournamentPlan/
    TournamentMonitor-Ebenen einfach `nil` und werden übersprungen.
  - In Tournament-Matches (mit TournamentMonitor) sind die Override-Slots leer →
    Resolver fällt auf Discipline durch wie heute.
- **D-15:** **Match-End-Verhalten Training:** Bestehendes Restart-with-swapped-Anstoß
  unverändert. Neuer TableMonitor erbt die heute resolved Werte als Discipline-Default
  (Resolver läuft erneut bei start_game). Kein migration code.
- **D-16:** **UI-Surface in dieser Phase = derived-only.** Keine UI-Toggles in
  Detail-Form (`scoreboard_free_game_karambol_new.html.erb`) oder Quickstart-Preset
  (`config/carambus.yml.erb`). Override geht nur via Console oder DisciplineController
  (admin-only, selten). UI-Override-Surfaces für Trainer (Detail-Form-Toggles) und
  Tournament-Setup-Wizards kommen in Folge-Phase falls echter Bedarf.

### Claude's Discretion

- Genaue Resolver-API: ob `BkParamResolver.resolve(param_name, table_monitor:)` mit
  einer Param-Liste-Iteration oder `BkParamResolver.bake!(table_monitor)` schreibt alle
  registrierten Params auf einmal. Empfehlung: Bake-API für die zwei Hooks (start_game,
  set-boundary), plus einzelne `resolve(...)`-Methode für Tests.
- Test-Strategie: Unit-Tests für Resolver pro Ebene (Discipline-only, Tournament-override,
  TableMonitor-override), Integration-Tests für Bk2::AdvanceMatchState set-boundary
  re-bake mit BK-2kombi DZ→SP-Wechsel, ScoreEngine-Tests mit gemockten data-Werten für
  alle 4 Kombinationen (true/true, true/false, false/false; false/true ist nicht-sinnig).
- Konstante für die zwei Param-Namen: `BkParamResolver::REGISTERED_PARAMS = [
  :allow_negative_score_input, :negative_credits_opponent].freeze`. Symbol vs. String —
  Symbol für interne API, Stringkonvertierung an data[]-Boundary.
- Backward-Compat im score_engine: Falls in-flight TableMonitor.data noch keine zwei
  Keys gesetzt hat (z.B. weil pre-Phase-38.5 erstellt), Fallback `false` per D-04 ist
  sicher für Karambol/Snooker und gibt Pre-38.5-Verhalten. Für aktive BK-2kombi-Matches
  könnte ein einmaliges `bake!` beim ersten Score-Event nach Deploy nötig sein —
  Planner entscheidet ob "lazy bake on first add_n_balls if data missing" oder
  "data migration script".
- Service-Modul-Name: `BkParamResolver` vs. `Carambus::ParamResolver` vs. inline in
  `Bk2`-Namespace. Empfehlung: `BkParamResolver` als Top-Level-Service (orthogonal zu
  Bk2 da auch Karambol/Snooker default-Fall durchläuft).

## Specific Ideas

- **multiset_components Reihenfolge** richtet sich nach `bk2_options[:first_set_mode]`:
  start mit DZ → `["bk_2plus","bk_2"]`, start mit SP → `["bk_2","bk_2plus"]`. Resolver
  bestimmt effective_discipline am Satzanfang als `multiset_components[(set_index-1) % 2]`.

- **Param-Resolver-API** (Empfehlung):
  ```ruby
  module BkParamResolver
    REGISTERED_PARAMS = [:allow_negative_score_input, :negative_credits_opponent].freeze

    def self.bake!(table_monitor)
      # Resolves all REGISTERED_PARAMS through the hierarchy and writes
      # results into table_monitor.data. Also writes effective_discipline.
    end

    def self.resolve(param_name, table_monitor:)
      # Walks Discipline → Tournament → TournamentPlan → TournamentMonitor →
      # Quickstart-Preset → Detail-Form → TableMonitor; returns first-found
      # explicit value, falls back to false. Used by tests + ad-hoc.
    end
  end
  ```

- **Latent-Bug Regression Tests** — explizite RED-Tests **vor** dem Fix:
  - Test 1: BK-2kombi DZ-Phase, Spieler A schießt -3 → erwartet: Player B Score +3,
    Player A Score unchanged. Heute: Player A Score -3 (BUG).
  - Test 2: BK-2-Match (single set), Spieler A schießt -2 → erwartet: Player A Score
    -2 (signed). Heute: Player A Score 0 (CLAMPED, BUG).
  - Test 3: BK50/BK100 dito.

- **first_set_mode-Default**: BK-2kombi seed-Default `first_set_mode=direkter_zweikampf`
  entspricht aktueller Tournament-Praxis. Trainer kann via Quickstart oder Detail-Form
  umschalten (existiert schon ab 38.3 D-19).
