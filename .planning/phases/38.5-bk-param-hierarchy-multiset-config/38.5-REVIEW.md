# Phase 38.5: Code Review Report

**Reviewed:** 2026-04-29T00:00:00Z
**Depth:** standard
**Files Reviewed:** 13

## Summary

The Phase 38.5 changes are well-scoped, well-tested, and the predicate rewrite + bake-hook pattern is a clean correctness improvement over the previous string-equality dispatch in `ScoreEngine`. The new `BkParamResolver` is small, has a clear contract (sparse-override via `data.key?`), and is exercised through three layers of tests (unit, integration, end-to-end system). No critical security or correctness issues found.

The findings below are logic robustness concerns and code-quality improvements:

- Two `ResultRecorder` warnings around behavior of `perform_save_current_set` when `@tm.game.blank?` (returns `nil` vs. an empty hash) and a guard that may incorrectly cap `kickoff_switches_with: "winner"` if the just-closed `Innings1` array is empty.
- The bake-hook is gated on `data["free_game_form"] == "bk2_kombi"` in `ResultRecorder#perform_switch_to_next_set` — non-`bk2_kombi` BK-* (BK-2/BK-2plus/BK50/BK100) never re-bake on set boundaries. Consistent with the documented design (their `effective_discipline` is stable), but this assumption is implicit and worth pinning.
- Dead `effective_discipline` write — the resolver writes `data["effective_discipline"]` but no production code reads it (only tests). Either harmless metadata for debugging or a forgotten consumer.
- `Discipline.find_by(name:)` runs synchronously on every `bake!` call (per param) — minor; not an issue at current call frequency.
- Several Info items: documentation inconsistencies, dead-code comments, brittle test stubs, and one ERB-injection-style risk in test rescue logging that's actually fine as-is.

## Warnings

### WR-01: Bake hook silently no-ops for non-`bk2_kombi` BK-* on set close

**File:** `app/services/table_monitor/result_recorder.rb:271-273`
**Issue:** The set-boundary re-bake is gated only on `data["free_game_form"] == "bk2_kombi"`. For BK-2/BK-2plus/BK50/BK100 multi-set matches, `perform_switch_to_next_set` does NOT call `BkParamResolver.bake!`. The phase docs claim this is intentional ("Non-BK-2kombi families have stable effective_discipline across sets"), and that's true today — but the BK-* discipline data is read into `tm.data` only at `start_game` (Hook 1). If a Tournament or TournamentMonitor row's overrides change mid-match (manual edit, config push, etc.), those changes do not propagate at set boundaries for non-kombi BK-*. For BK-2kombi, Hook 2 picks them up.

This is more brittle than it appears: the comment says the bake "would be a no-op" for non-kombi, but it actually wouldn't be a no-op if the upstream Tournament/TournamentMonitor data changed. Today this isn't reachable, but the asymmetry is silent.

**Fix:** Either make the rebake unconditional for any BK-* family, or pin the assumption with an explicit comment + spec:

```ruby
# Phase 38.5 D-03 hook 2: re-bake at set-boundary for BK-2kombi only.
# CONTRACT: For non-BK-2kombi BK-* families, the resolver's hierarchy walk is
# assumed stable across the match — Tournament/TournamentMonitor overrides
# captured at start_game (Hook 1) remain authoritative. Re-bake here would be
# safe-but-redundant for non-kombi; we skip to keep the call narrow.
# If upstream override semantics change, widen the guard to `bk_family?`.
if @tm.data["free_game_form"] == "bk2_kombi"
  Bk2::AdvanceMatchState.rebake_at_set_open!(@tm)
end
```

A safer alternative: re-bake for ALL BK-* families on set close. The work is cheap (4 hash writes) and the resulting state is provably correct.

### WR-02: `perform_switch_to_next_set` crashes when `kickoff_switches_with: "winner"` and last set's `Innings1`/`Innings2` arrays are empty

**File:** `app/services/table_monitor/result_recorder.rb:209-211`
**Issue:** The "winner" branch reads:

```ruby
current_kickoff_player = (@tm.data["sets"][-1]["Innings1"][-1].to_i > @tm.data["sets"][-1]["Innings2"][-1].to_i) ? "playera" : "playerb"
```

For BK-2kombi DZ-Phase or simple_set games (Snooker, Pool), the `Innings1`/`Innings2` arrays in `data["sets"][-1]` may legitimately be empty (the winning condition is `result`, not the last inning's score). In that case `[][-1]` returns `nil`, then `.to_i` returns 0, both sides equal, "playerb" silently selected — not a crash, but wrong. If the array key itself is missing (`["Innings1"]` returns nil), `nil[-1]` raises `NoMethodError`. This is also an existing legacy issue but was inherited by Phase 38.5's set-close path.

**Fix:** Guard with explicit nil-coalescing and prefer `"Ergebnis1"`/`"Ergebnis2"` when arrays are empty:

```ruby
when "winner"
  last = @tm.data["sets"][-1] || {}
  e1 = last["Ergebnis1"].to_i
  e2 = last["Ergebnis2"].to_i
  current_kickoff_player = (e1 > e2) ? "playera" : "playerb"
```

### WR-03: `perform_save_result` returns `{}` (not `nil`) when `@tm.game` is absent — `perform_save_current_set` then proceeds

**File:** `app/services/table_monitor/result_recorder.rb:52-105` and `:177-185`
**Issue:** When `@tm.game` is blank, `perform_save_result` returns `game_set_result = {}` (the local default at line 53). `perform_save_current_set` calls `perform_save_result`, then unconditionally pushes the `{}` into `data["sets"]`, calls `update_ba_results_with_set_result!({})` (which early-returns, OK), then `save!`. The result: an empty `{}` sentinel gets pushed into `data["sets"]`. Subsequent calls to `compute_effective_discipline` count this as a "closed set" via `Array(data["sets"]).length + 1`, which would advance the multiset_components index — but in practice only happens when `@tm.game` is blank, an unreachable path during normal matches.

This is brittle: if a test or admin tool ever calls `perform_save_current_set` on a TM without a game, it silently corrupts `data["sets"]` and `effective_discipline`.

**Fix:** Mirror the existing `perform_save_current_set` already-saved guard with an explicit early-return when no game:

```ruby
def perform_save_current_set
  unless @tm.game.present?
    Rails.logger.info "[save_current_set] m6[#{@tm.id}] no game — skipping"
    return
  end
  # ... existing body ...
end
```

The current code already has the equivalent of this check at line 186 (`else: ignored - no game`), but only AFTER the simple_set_game duplicate detection at 151-175 — so for simple_set_games, the body runs partially before the `else` branch is reached.

### WR-04: `score_engine.rb` `set_n_balls` has nested unreachable branches and a duplicated `if data["biathlon_phase"] != "3b" || n_balls <= to_play_3b` test

**File:** `app/models/table_monitor/score_engine.rb:206-268`
**Issue:** This method is inherited (not authored in Phase 38.5), but the predicate rewrite passes through it untouched and the structure is now harder to reason about — same condition is checked at lines 211 and 220, with an `else` branch on line 263 that can never fire (its outer condition at line 211 already matched). Lines 213 and 217 both have a `return :goal_reached` at the same depth, which the test suite seems to depend on.

This is dead code from the Biathlon path that won't be triggered for any BK-* discipline (none of which set `biathlon_phase`). The risk: a future refactor that simplifies the duplicated check could accidentally drop the `set == to_play` early return, breaking BK-2kombi `set_n_balls` calls.

**Fix:** Out of scope for Phase 38.5 (no behavioural change requested), but flag for follow-up. Add a regression test that exercises `set_n_balls` for BK-2 + BK-2kombi + Biathlon to lock the current shape.

### WR-05: Tests stub class methods on `BkParamResolver` and `Bk2::AdvanceMatchState` without `Mocha`/`Minitest::Mock` — leaks if the test fails before `ensure`

**File:** `test/services/bk2/advance_match_state_test.rb:74-88`, `test/services/table_monitor/result_recorder_test.rb:307-317, 325-335`
**Issue:** Three tests use `define_singleton_method` to stub `BkParamResolver.bake!` / `Bk2::AdvanceMatchState.rebake_at_set_open!`, with the original captured via `method(:name)` and restored in an `ensure` block. The pattern works, BUT:

1. If the `original = X.method(:name)` line raises (it can if the method is missing), the `ensure` block restores `nil` — the stub stays in place for subsequent tests in the file.
2. If a sibling test runs in parallel, the singleton-method override is process-global and races.
3. Minitest uses `define_singleton_method(name, original)` to restore — that uses the `Method` object as a block, which IS valid Ruby but obscure; `original.bind(...).call` is more conventional.

**Fix:** Use `Mocha` (already in the project per Gemfile presence-check) or `Minitest::Mock`:

```ruby
test "Phase 38.5 D-03: rebake_at_set_open! delegates to BkParamResolver.bake!" do
  BkParamResolver.expects(:bake!).with(@tm).once
  Bk2::AdvanceMatchState.rebake_at_set_open!(@tm)
end
```

Or, if Mocha isn't wanted, structure the ensure block defensively:

```ruby
original = BkParamResolver.method(:bake!) rescue nil
recorded = []
BkParamResolver.define_singleton_method(:bake!) { |arg| recorded << arg }
begin
  Bk2::AdvanceMatchState.rebake_at_set_open!(@tm)
  # assertions
ensure
  if original
    BkParamResolver.singleton_class.send(:remove_method, :bake!)
    BkParamResolver.define_singleton_method(:bake!, original)
  end
end
```


### IN-02: `BkParamResolver.lookup_discipline` does a per-call DB hit in `bake!`

**File:** `app/services/bk_param_resolver.rb:106-113`
**Issue:** `bake!` calls `resolve` for each registered param (currently 2). Each `resolve` calls `lookup_discipline`, which calls `Discipline.find_by(name: bk_name)`. The `effective_discipline_form:` cache parameter avoids the lookup walk repetition for the SAME bake call BUT the Discipline AR record is still re-fetched once per param (2x). At each `start_game` and each set boundary that's 2 queries — negligible at current load.

**Fix:** Cache the Discipline record inside `bake!`:

```ruby
def self.bake!(table_monitor)
  eff_form = compute_effective_discipline(table_monitor)
  table_monitor.data["effective_discipline"] = eff_form
  discipline_record = lookup_discipline(eff_form)
  REGISTERED_PARAMS.each do |param|
    table_monitor.data[param.to_s] = resolve(
      param,
      table_monitor: table_monitor,
      effective_discipline_form: eff_form,
      discipline_record: discipline_record  # new param, optional
    )
  end
end
```

Out of scope for v1 (performance), but worth noting for follow-up.

### IN-03: `lookup_discipline` returns nil for non-BK families silently — by design but undocumented at call site

**File:** `app/services/bk_param_resolver.rb:106-113` and `:69`
**Issue:** When `lookup_discipline` returns nil, `parsed_discipline_data(nil)` returns `{}`, which has no keys, so the walk falls through to `false`. The chain works, but the "non-BK families return nil → fall through to false fallback" path is documented in the resolver header comment, not at the line where it matters. A developer scanning Level 1 of the walk:

```ruby

```

…would not see why karambol/snooker/pool return `false`. The documentation in the module header (lines 102-105) is good, but a one-liner at the call site would help.
