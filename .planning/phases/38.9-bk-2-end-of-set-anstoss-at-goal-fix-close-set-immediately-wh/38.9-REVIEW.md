## Summary

Phase 38.9 adds a fourth sub-branch to `TableMonitor#end_of_set?` (lines 1529–1552) that closes a BK-2 / BK-2kombi-SP set immediately when Anstoss-Spieler reaches `balls_goal` in inning ≥ 2. This mirrors the close-side of the Erste-Aufnahme-Gate already enforced on the open-side in `follow_up?` (lines 1205–1210). Two tests were appended to `test/models/table_monitor_test.rb`: SC-1 (RED → GREEN, inning 2 must close immediately) and SC-2 (regression guard, inning 1 must NOT over-fire).

The change is **semantically correct, surgically scoped, and well-guarded**:

- The new branch is wrapped inside the existing `bk_with_nachstoss` block (line 1510), so it cannot fire on non-BK disciplines (karambol, snooker, pool, BK-2plus, BK50, BK100, BK-2kombi-DZ-Phase). Over-firing on non-BK disciplines is structurally impossible.
- The boundary condition `anstoss_innings >= 2` is correct: inning 1 is the Erste-Aufnahme-Gate happy path (Nachstoß permitted, banner renders, set stays open). SC-2 pins this boundary with a `refute`.
- Reuses already-computed locals (`anstoss_role`, `nachstoss_role`, `anstoss_innings`, `anstoss_at_goal`) — zero recomputation, no new helper method, no parallel state machine. Aligns with the `extend-before-build` skill (validated 2026-04-29 with -1463 LOC rollback on this exact surface).
- Logger output is informative and follows the established style of the D-02 branch.

## Info

### IN-01: Minor asymmetry between `follow_up?` and the new 38.9 branch in determining the Anstoss role

**File:** `app/models/table_monitor.rb:1516`
**Issue:** The new branch determines the Anstoss role with:

```ruby
anstoss_role = data["current_kickoff_player"].presence || "playera"
```

`follow_up?` (line 1193), whose Erste-Aufnahme-Gate this branch is supposed to mirror, computes the equivalent identifier slightly differently:

```ruby
left_player_id = data["fixed_display_left"].blank? ? data["current_kickoff_player"] : data["current_left_player"]
```

Both predicates are intended to identify the same role (the kickoff player). In standard BK-2 fixtures the two formulas agree because `fixed_display_left` and `current_left_player` are kept aligned with `current_kickoff_player`. However, in any scenario where `fixed_display_left` is set AND `current_left_player` differs from `current_kickoff_player`, the open-side gate (`follow_up?`) and the close-side gate (the new branch) would disagree about which player counts as Anstoss — defeating the symmetry the comment claims to preserve.

Note: this asymmetry was inherited from the existing D-02 branch (line 1516), so the new 38.9 branch is consistent with its immediate neighbour. The risk is therefore low.

**Fix:** No action required for this phase. If a future phase tightens the symmetry, consider extracting a single `kickoff_role_for(data)` helper consumed by both `follow_up?` and `end_of_set?`. Document the chosen identifier convention as part of the BK-2 invariants doc.

The risk that the predicate behaves differently for `bk2_kombi` is low (the predicate is identical and `bk_with_nachstoss` is the only branching point), but a test would lock the contract and protect against future churn in `bk2_kombi_current_phase`.

### IN-03: TR-B baseline test (line 121) now reaches `end_of_set? == true` via the new 38.9 branch instead of the legacy parity gate

**File:** `test/models/table_monitor_test.rb:121–128`
**Issue:** The Plan 38.7-02 TR-B baseline test (`"end_of_set? closes BK-2 set when both reach balls_goal at equal innings (TR-B baseline)"`) uses `playera_innings=5, playerb_innings=5, both at goal`. Walking through the predicates with the new branch in place:

1. `bk_with_nachstoss = true`
2. D-02 branch: `nachstoss_finished_followup = (5 == 5 + 1) = false` → not fire
3. **New 38.9 branch:** `anstoss_at_goal = true && anstoss_innings (5) >= 2 = true` → returns true

The assertion still passes (the test asserts `end_of_set? == true`), but the code path that produces the truthy result has shifted. Before 38.9, the legacy karambol gate at line 1555–1559 would fire (innings parity 5==5 + both at goal). Today, the 38.9 branch fires first and the legacy branch is never reached for this scenario.

The behavioral contract (`end_of_set? == true`) is preserved. The log message that observers will now see is `[TableMonitor#end_of_set?] Phase 38.9 BK-2-immediate-close: ...` instead of `Game[…] ended: balls_goal reached`. This may surprise operators / log scrapers reading production logs against an old playbook.

Note also that the new log message is technically slightly misleading for this scenario: the comment in the new branch says "no Nachstoss-Aufnahme permitted" — but in the TR-B baseline scenario, Nachstoss has already played (and reached the goal). The branch still produces the correct decision (set closes), but the log message is framed around the inning-2 latent-defect scenario rather than the both-at-goal-equal-innings scenario.

**Fix:** No action required for this phase — the contract is preserved. If log clarity becomes a concern, consider distinguishing the two cases in the log line, e.g.:

```ruby
reason = nachstoss_innings.zero? ? "Erste-Aufnahme-Gate fails (no Nachstoss-Aufnahme permitted)" : "Anstoss past inning 1 with both already played"
Rails.logger.info "[TableMonitor#end_of_set?] Phase 38.9 BK-2-immediate-close: ... — #{reason}"
```

Alternatively, reorder the branches so the legacy karambol gate at lines 1555–1564 (innings parity) is checked **before** the 38.9 branch. This would keep the existing TR-B log message and reserve the 38.9 log for the genuinely new scenario. The semantic outcome is identical either way.
