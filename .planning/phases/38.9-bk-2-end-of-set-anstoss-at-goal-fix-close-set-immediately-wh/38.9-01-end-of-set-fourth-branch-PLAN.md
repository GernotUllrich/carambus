<objective>
Close the latent BK-2 / BK-2kombi-SP "Anstoss-at-goal-in-inning-2" defect (introduced
together with the Erste-Aufnahme-Gate in commit `79328663`, 2026-04-28).

Today, when the Anstoss-Spieler reaches `balls_goal` in his 2nd (or later) inning of a
BK-2 game, `TableMonitor#end_of_set?` returns false because none of its three existing
branches match this case:
  - `no_followup_phase` excludes BK-2.
  - Plan 38.7-02's D-02 branch requires `nachstoss_innings == anstoss_innings + 1` —
    Nachstoss has not played yet, so this is `0 != 3`.
  - The legacy karambol gate requires innings parity OR `!allow_follow_up` — both false
    in this scenario.

The set stays open. `ScoreEngine#terminate_inning_data` (`score_engine.rb:1306`) swaps
`active_player` unconditionally. Player B plays an unauthorized inning. The legacy parity
gate finally closes the set after Player B's inning increments to 1 — without ever
showing the red "Nachstoß" banner (which `follow_up?` correctly suppresses via the
Erste-Aufnahme-Gate at `table_monitor.rb:1205-1210`).

This plan adds a single guard branch to `end_of_set?` that mirrors the Erste-Aufnahme-Gate
on the close-side: when `bk_with_nachstoss` is true AND Anstoss reached `balls_goal` AND
Anstoss already played 2 or more innings, the set closes immediately. Per
`.agents/skills/extend-before-build/SKILL.md` (validated 2026-04-29: -1463 LOC after
rolling back a parallel state machine on this exact surface), the fix is an additive
branch on the existing predicate — no new state machine, no helper class, no parallel
predicate.

This plan ships RED-then-GREEN: the failing characterization test lands first (commits
to the regression baseline), then the production fix lands and turns it GREEN.

Output: 2 new tests in `test/models/table_monitor_test.rb` + a small additional branch
in `TableMonitor#end_of_set?` in `app/models/table_monitor.rb`.
</objective>

Per CONTEXT.md and the linked debug document `.planning/debug/bk2-nachstoss-banner-missing.md`,
this is a three-way semantic mismatch:

1. `follow_up?` correctly enforces the Erste-Aufnahme-Gate (banner suppressed when
   Anstoss reaches goal in inning 2+) — KEEP AS-IS.
2. `terminate_inning_data` switches `active_player` unconditionally — KEEP AS-IS (it
   only runs if end_of_set? returns false; the fix at end_of_set? prevents that path).
3. `end_of_set?` is missing a branch that mirrors the Erste-Aufnahme-Gate semantic on
   the close-side — THIS IS THE FIX.

predicate situation. The SKILL was validated 2026-04-29 on this exact surface — adding a
parallel state machine cost +1463 LOC of garbage that had to be reverted. The fix is a
single conditional branch on the existing `end_of_set?` predicate. NO new helper class,
NO new predicate, NO parallel state machine.

Per CONTEXT.md "Out of scope" list: this plan does NOT touch follow_up?, does NOT touch
terminate_inning_data, does NOT touch the banner conditional in _player_score_panel.html.erb,
does NOT replan phase 38.7-02's D-02 branch, does NOT replan phase 38.8's operator-gate.
The fix is surgical and localized.

Per `.planning/STATE.md` v7.1 Phase 38.7 decisions: the existing tiebreak path (tied
BK-2 sets at equal innings) flows through the legacy karambol parity branch (innings_a
== innings_b) into Plan 38.7-04's tiebreak detection. The new 4th branch slots BEFORE
