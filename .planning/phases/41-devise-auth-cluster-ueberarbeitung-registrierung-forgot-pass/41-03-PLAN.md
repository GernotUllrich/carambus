<objective>
Forgot-Password-Flow auf D-41-C-Standard heben: Reset-Password-Use rotiert den JTI automatisch via after_update-Callback im User-Modell. Damit werden ALLE bestehenden JWT-Sessions des Users invalidiert — ein kompromittierter Account, der per Reset gesichert wird, kann von Angreifern mit gestohlenem JWT NICHT weiter genutzt werden.

Implementation-Approach: Model-Callback (NICHT Controller-Override) — laut RESEARCH.md Section "Architecture Patterns" Option B. Vorteile: greift bei jedem Passwort-Change-Pfad (Controller, Admin, Console, direkter Save) — defense in depth; kein neuer Controller noetig (Anti-Pattern: "Don't rewrite Devise"); keine devise_for-Route-Aenderung in routes.rb.

Cross-Repo-Sicherheit: Callback feuert NUR bei `saved_change_to_encrypted_password?` — bei Routine-Updates (Preferences, Email, Confirmation) bleibt der JTI stabil → bcw MCP-JWTs werden NICHT unbeabsichtigt revoked.

Purpose: D-41-D Flow 2 (Forgot/Reset-Password) + D-41-C (Hard-Revoke). Plan-04 wird denselben Callback fuer Change-Password nutzen (kein zusaetzlicher Codepfad noetig).

Output: User-Modell mit JTI-Rotation-Callback; Layer-1-Unit-Tests (Trigger und Non-Trigger); Erweiterung der Layer-2 Reset-Password-Controller-Tests um JTI-Assertion.
</objective>

```ruby
module Devise::JWT::RevocationStrategies::JTIMatcher
  # Vom Warden-Hook aufgerufen, prueft ob payload['jti'] zum User.jti passt
  def self.jwt_revoked?(payload, user)
    payload['jti'] != user.jti
  end

  # Rotiert den jti — alle alten JWTs werden ungueltig
  def self.revoke_jwt(_payload, user)
    user.update_column(:jti, generate_jti)
  end

  # Beim User-Insert (after_initialize?): jti wird gesetzt falls leer
  def self.generate_jti
    SecureRandom.uuid
  end
end
```
