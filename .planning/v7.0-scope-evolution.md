## The three meta-findings

### Meta-1: Wizard-Schritt ≠ AASM-State ≠ UI-Screen (F-36-11, F-36-15)

The walkthrough's numbered "Schritte 1–14" don't map linearly to either AASM states or UI screens. Current reality:

| Doc Schritte | Actual UI | Actual AASM |
|--------------|-----------|-------------|
| 1 (invitation) | no UI | — |
| 2–5 (Meldeliste / Setzliste / Teilnehmerliste) | `TournamentsController#index` wizard partial | one state with action-links |
| 6 (Turniermodus) | separate mode-selection screen | state transition |
| 7–8 (start params + tables) | **same** `tournament_monitor` param form | one screen |
| 9+ (start / warmup / games) | `TableMonitor#show` and downstream views | multiple states |

The linear "Step for Step" framing suggests more manager interaction than really exists — during the actual game (Schritte 10–12) the Turnierleiter barely touches Carambus because everything happens at the scoreboards.

**Consequence:** Phase 36b will need to make the walkthrough honest about this structure OR the UI will need to be consolidated (bigger work). The SME quote: *"Alle diese Seiten sind historisch gewachsen und haben leider kein einheitliches UI-Konzept."*

### Meta-2: ClubCloud-Upload model incomplete (F-36-23)

FIX-02 as originally written ("auto_upload_to_cc checkbox location") was a narrow reading of a much larger gap:

- Carambus supports two upload paths: per-game upload and end-of-tournament CSV
- Both require the Teilnehmerliste to be finalized in ClubCloud first
- **Finalizing the Teilnehmerliste via the CC API is not currently implemented**
- Only certain ClubCloud roles (typically Club-Sportwart) have permission to add missing players, and they aren't always present
- The doc describes none of this
- The doc also mentions a "Schaltfläche 'Ergebnisse nach ClubCloud übertragen'" that **does not exist in the Carambus UI**

**Consequence:** The ClubCloud integration story needs its own phase. Phase 36c (or v7.1) must cover the actual upload model, credential delegation for Club-Sportwart rights, and possibly CC-API finalization code.

### Meta-3: The Walkthrough overstates manager activity during play (F-36-30, F-36-31)

Schritt 11 "Spielbeginn freigeben" describes actions the manager doesn't actually perform — all game starts are automatic at the scoreboards. Schritt 12 describes a Turnier-Monitor view with "Spielbeginn" buttons that exist only as a dead-code fallback. The real manager role during play is **observation** plus occasional intervention via browser-tab oversight on individual table links.

**Consequence:** The walkthrough gliederung needs restructuring into "phases with and without manager action" rather than a linear 14-step sequence. This is tier-A doc work but affects the whole document layout.

## Agreed restructuring: Phase 36 → 36a / 36b / 36c

### Phase 36a — Turnierverwaltung Doc Accuracy
**Focus:** All tier-A doc corrections from the review.
**Scope:** ~60 doc-level actions mapped to the 7 review blocks.
**Input artifact:** `.planning/phases/36-small-ux-fixes/36-DOC-REVIEW-NOTES.md`
**Output:** Corrected `tournament-management.{de,en}.md` + new glossary entries + new troubleshooting recipes + new appendix sections (special cases, ClubCloud CSV handling).
**EN handling:** Corrections derive 1:1 from DE; no separate EN review pass. Both language files updated in the same plans.
**Requirements (new, replacing original FIX-01..04 for 36a):**
- DOC-ACC-01: Begriffshierarchie (Setzliste / Meldeliste / Teilnehmerliste) consistent across walkthrough + glossary
- DOC-ACC-02: Ballziel / Aufnahmebegrenzung correction + all other factual corrections from blocks 1–7
- DOC-ACC-03: New glossary entries for logischer/physikalischer Tisch, TableMonitor, Turnier-Monitor, Meldeliste, Teilnehmerliste, Trainingsmodus, Freilos, T-Plan vs. Default-Plan
- DOC-ACC-04: New appendix sections (no-invitation flow, missing-player flow, player registration flow, CC CSV upload handling, Rangliste manual maintenance)
- DOC-ACC-05: Walkthrough restructure — Meta-finding 3 (honest "phases with/without manager action")
- DOC-ACC-06: "Mehr zur Technik" entfernt

**Estimate:** 1 medium phase, ~5–8 plans (one per review block plus a consolidation plan).

### Phase 36b — UI Cleanup & Kleine Features
**Focus:** UI cleanup items (U1–U5) + the small features (F6, F7) + the original FIX-01..04 as far as they remain relevant after Phase 36a.
**Scope:**
- U1: Tooltip feature on parameter form fields
- U2: i18n corrections for English labels
- U3: Remove manual round-change control (parameter and UI)
- U4: Remove dead-code input UI from Turnier-Monitor
- U5: Delete unused `_wizard_steps.html.erb`
- F6: Reset safety dialog
- F7: Parameter verification dialog before start
- Original FIX-01..04: evaluated for relevance — FIX-01 (help block open by default) is still valid and can fold into U1 tooltip work; FIX-02 (auto_upload_to_cc location) is resolved as "code and docs already match" per F-36-23; FIX-03 (step names) and FIX-04 (AASM state badge) remain, but F-36-15 suggests the badge work should happen alongside UI consolidation, not in isolation

**Requirements (new):**
- FIX-01 (original, still valid): Help block open by default for active wizard step
- FIX-03 (original, still valid): Step names instead of bare numbers
- FIX-04 (original, flagged): AASM state badge prominence — dependency on Meta-1 UI consolidation
- UI-01: Tooltips on parameter form fields
- UI-02: i18n corrections for English labels in start form
- UI-03: Remove dead-code manual round-change control
- UI-04: Remove "Aktuelle Spiele" dead-code input UI
- UI-05: Delete `_wizard_steps.html.erb` (per F-24)
- UI-06: Reset safety dialog at `tournament_started` or later
- UI-07: Parameter verification dialog before start

**Estimate:** 1 medium phase, ~6–8 plans. Fits within v7.0.

**Note:** FIX-02 is **closed as no-op** because the code and docs already agree; the larger CC-upload issue is moved to Phase 36c. REQUIREMENTS.md will be updated to reflect this.

### Phase 36c — v7.1 Preparation / ClubCloud Integration Groundwork
**Focus:** The large code features (F1–F5, F8) that exceed v7.0 scope.
**Scope:** Not a full implementation — a **design + planting-seed** phase that produces:
- A v7.1-ClubCloud-Integration milestone roadmap for features F1, F4, F5
- A v7.1-Shootout milestone roadmap for F2 (potentially its own milestone, v7.2)
- A planting seed for F3 (Match-Abbruch)
- A scoping document for F8 (UI consolidation)
- Optionally: Phase 36c may execute F6, F7 (small items) as preview features if they're isolated enough

**Output artifacts:** 
- `.planning/milestones/v7.1-*.md` skeleton files
- `.planning/seeds/*.md` entries for long-term items
- No code changes (or minimal: F6, F7 if they fit)

**Estimate:** 1 small phase, 2–4 plans. Mostly planning artifacts, not implementation.

**Why 36c instead of jumping straight to v7.1:**
The v7.0 milestone should close cleanly before the team takes on the big features. 36c does the planning work that makes v7.1 executable without blocking v7.0 closure. Plus: some of the findings (F-36-23 ClubCloud-Upload complexity) benefit from one more round of SME clarification before commitment.
