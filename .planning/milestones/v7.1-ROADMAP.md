## v7.1 Milestone Goal

Carambus takes over every ClubCloud-interaction that today forces the
Turnierleiter into a second browser tab: Teilnehmerliste finalization,
missing-player handling, and Endrangliste upload. At the end of v7.1 a
Carambus-run tournament that is backed by ClubCloud should complete
end-to-end without the Turnierleiter ever opening the CC admin UI
(unless their Carambus role explicitly lacks a CC permission, in which
case Carambus produces a concrete handoff).

## Proposed Phases

Phase numbering is **provisional**. Actual numbers depend on whether
v7.0 extends further before v7.1 starts.

### Phase A: Endrangliste Calculation & Preview (CCI-01, CCI-02)

**Goal:** Carambus computes the Endrangliste in-memory for every
supported tournament mode, and a preview screen lets the Turnierleiter
verify it before any upload.

- **Scope:** Pure ruby + ERB/Stimulus; no CC API calls yet.
- **Depends on:** Nothing v7.1-specific.
- **Risk:** Medium. The "supported tournament mode" set covers KO,
  round-robin, group+finals, and possibly league-style — each mode's
  ranking rule must be codified explicitly.
- **Plans:** ~4–6 **[TBD: v7.1]**
  - Ranking calculator PORO with one strategy per AASM tournament mode
  - Preview view with tiebreaker manual-override UI
  - Characterization tests from historical tournament fixtures
  - Integration with existing tournament_monitor parameter form

### Phase B: Endrangliste Upload via CC API (CCI-03)

**Goal:** The Turnierleiter clicks an "Upload Endrangliste" button; the
computed ranking goes to CC via the same API surface the per-game
upload already uses.

- **Scope:** CC API client extension + error handling + success/failure
  UI states.
- **Depends on:** Phase A (Endrangliste must exist in memory).
- **Risk:** Low-medium. Leverages existing CC upload authentication.
- **Plans:** ~2–3 **[TBD: v7.1]**

### Phase C: Teilnehmerliste Finalization via CC API (CCI-04, CCI-05)

**Goal:** Carambus can call the CC API endpoint that marks a
Teilnehmerliste as finalized, guarded by a preflight check that every
participant has a matching CC player record.

- **Scope:** CC API client extension, preflight match logic, UI flow
  for the finalization action (probably a second confirmation-modal
  variant).
- **Depends on:** Nothing v7.1-specific, but **[TBD: v7.1]** on the
  specific CC endpoint — SME needs to confirm the API surface exists
  and describe its auth/payload contract.
- **Risk:** High. If the CC endpoint doesn't exist or requires auth
  Carambus cannot currently produce, this phase blocks and the whole
  upload model needs a different approach.
- **Plans:** ~3–5 **[TBD: v7.1]**

### Phase D: Missing-Player Flow (CCI-06)

**Goal:** When a Carambus-known participant has no CC match, Carambus
either creates the CC player on-the-fly (if credentials allow) or
produces a concrete handoff to a human with the right role.

- **Scope:** CC player creation API (if any) + handoff report + resume-
  point persistence so the Turnierleiter can return to the exact step
  they were blocked on.
- **Depends on:** Phase C (preflight logic surfaces the missing players).
- **Risk:** High. Depends on whether the CC API exposes player creation
  at all — SME to confirm. **[TBD: v7.1]**
- **Plans:** ~3–4 **[TBD: v7.1]**

### Phase E: Credentials Profile & Permission Handoffs (CCI-07, CCI-08)

**Goal:** Carambus knows what CC role the current user has and tells
them up-front what they can and cannot do in the CC integration.
Missing-permission actions produce concrete handoffs.

- **Scope:** User model extension for CC role metadata, middleware that
  surfaces the role in tournament actions, handoff report generator.
- **Depends on:** Phases C and D (the handoff reports describe specific
  flows that exist in those phases).
- **Risk:** Medium. Most of the risk is UX — how to communicate role
  limits without nagging users who never hit them.
- **Plans:** ~2–3 **[TBD: v7.1]**

### Phase F: Organizational Documentation (CCI-09)

**Goal:** A written procedure (`docs/managers/clubcloud-admin-handling.md`)
explains how Club-Sportwart credential delegation works in practice —
shared account vs. per-tournament token vs. impersonation — and what
the venue team does when the right person isn't present.

- **Scope:** Doc-only, no code. Ground truth comes from SME interview
  during v7.1 discuss-phase.
- **Depends on:** Phase 36c `.planning/clubcloud-admin-appendix-DRAFT.md`
  as starting point (see Phase 36c PREP-04 deliverable).
- **Risk:** Low (doc work) but **[TBD: v7.1]** on SME availability.
- **Plans:** 1 **[TBD: v7.1]**

## Execution order (proposed)

`A → B → C → D → E → F`

- A and B form the Endrangliste vertical; they can ship as a sub-milestone.
- C, D, E form the participant-list-and-permissions vertical; C gates
  both D and E.
- F is doc work that should happen in parallel (in a branch that doesn't
  touch code) and land before v7.1 ships.
