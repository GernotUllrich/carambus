# Roadmap: Carambus API v7.2 — Shootout / Stechen Support

**Status:** SKELETON — produced during Phase 36c (2026-04-14)
**Predecessor:** v7.1 ClubCloud Integration Redesign (skeleton)

> This document is a **skeleton**. Whether v7.2 becomes a standalone
> milestone or folds into v7.1 is **[TBD: v7.1-discuss]**. Until that
> decision, treat the phases below as an independent unit of work.

## Milestone Goal

By the end of v7.2, a KO-style tournament with a decisive game that
ends in a tie is resolved live on the scoreboards via Shootout / Stechen
mode — AASM tracks the state, the tournament plan routes the winner
into subsequent rounds automatically, and the final ranking (v7.1
CCI-01) attributes the result to the correct player without a human
editing a spreadsheet.

## Proposed Phases

Phase numbering provisional; finalized during v7.2 discuss-phase.

### Phase A: AASM Foundations (SHO-01, SHO-02)

**Goal:** Tournament and Game AASM gain the states and transitions for
Shootout without changing any runtime behavior — purely additive.

- **Scope:** AASM diagram update, transition callbacks, PaperTrail
  version baseline, characterization tests using existing tournament
  fixtures to prove the non-Shootout path is unchanged.
- **Depends on:** Nothing v7.2-specific.
- **Risk:** Medium. AASM changes are invasive, and the Tournament
  model had just come through the v2.1 refactoring — we need to be
  careful not to regress any existing state-transition tests.
- **Plans:** ~3 **[TBD: v7.2]**

### Phase B: Discipline Shootout Rules (SHO-08)

**Goal:** A new `Discipline#shootout_rules` column (or similar) holds
the per-discipline Shootout rules. Seed data for the 5–6 disciplines
Carambus actually uses (Einband, Dreiband klein, Dreiband groß, Cadre,
Free-game, 5-pin).

- **Scope:** Migration, seed data, model method, unit tests.
- **Depends on:** SME ground truth on the actual rules **[TBD: v7.2]**.
- **Risk:** Low (data structure) + high (SME availability).
- **Plans:** ~2 **[TBD: v7.2]**

### Phase C: Tournament Plan Integration (SHO-03, SHO-04, SHO-05)

**Goal:** The tournament plan distribution logic knows that a KO leaf
game can have a Shootout appendix whose result replaces the tied leaf
result for downstream routing. The Endrangliste calculator (v7.1
CCI-01) agrees.

- **Scope:** `Tournament::RankingCalculator` extension,
  `TournamentMonitor::PlayerGroupDistributor` update, integration with
  v7.1 CCI-01.
- **Depends on:** Phase A (AASM states) and v7.1 CCI-01 (Endrangliste
  calculator must exist or be designed with a Shootout placeholder).
- **Risk:** Medium-high. Cross-service changes where regression in the
  non-Shootout path is easy and catastrophic.
- **Plans:** ~4–6 **[TBD: v7.2]**

### Phase D: Scoreboard UI (SHO-07, SHO-09)

**Goal:** When a game enters `shootout_pending`, both physical-table
scoreboards switch to Shootout mode with distinct visual state, and
the Turnier-Monitor view distinguishes Shootout-in-progress from
regular games.

- **Scope:** TableMonitor view update, new Stimulus controller state,
  scoreboard CSS for the Shootout banner, Turnier-Monitor grouping
  change.
- **Depends on:** Phases A, B, C (state exists, rules exist, routing
  works).
- **Risk:** Medium. Visual work that can look correct in isolated
  testing but fail on the actual scoreboard hardware — needs live
  venue verification.
- **Plans:** ~3–4 **[TBD: v7.2]**

### Phase E: Abort Semantics (SHO-09 open question)

**Goal:** Define and implement what happens when a Shootout starts but
one player can't complete it (left, injured, disqualified). This is
the intersection of SHO-05 and the Match-Abbruch backlog seed.

- **Scope:** Decision document + implementation + tests.
- **Depends on:** Phase A (AASM state is what we're extending), plus
  SME ground truth on the abort rules **[TBD: v7.2]**.
- **Risk:** High, not because of code but because of rules: some
  disciplines may have definitive abort rules, others may leave it
  to the referee.
- **Plans:** ~1–2 **[TBD: v7.2]**

## Execution order (proposed)

`A → B → C → D → E`

- A ships first and independently (non-Shootout path is unchanged).
- B can ship in parallel with A as a separate data-only phase.
- C requires both A and B, plus a decision on v7.1 CCI-01 coordination.
- D is the last dependency — it lights everything up for users.
- E is a cleanup phase but should ship within v7.2, not deferred again.

## Provisional totals

- **Phases:** 5
- **Plans:** ~13–17 **[TBD: v7.2]**
- **Estimated duration:** 2–3 weeks, dominated by SME-question turnaround.

## Cross-milestone coordination

- **v7.1 CCI-01 (Endrangliste calculator)** — must be designed to
  accept Shootout results as first-class data, not as a post-hoc
  patch. Coordinate during v7.1 Phase A (ranking strategy design).
- **Match-Abbruch backlog seed** — SHO-09 and Match-Abbruch overlap
  conceptually. Whoever picks up v7.2 first should re-read the seed
  and decide whether to resolve them together or separately.

## Out of scope for v7.2

- v7.1 CC integration (separate milestone).
- Match-Abbruch general handling (backlog seed).
- Discipline additions beyond the 5–6 already in use.
