Test name: `"CONC-01: rapid-fire AASM transitions — no broadcast bleed under high-frequency alternating broadcasts"`

Implementation per D-01, D-02, D-03:

1. Open Session A on TM-A scoreboard (`in_session(:scoreboard_a)`), confirm `#full_screen_table_monitor_#{@tm_a.id}` present, call `wait_for_actioncable_connection`, install the `window._mixupPreventedCount` console.warn interceptor (same pattern as ISOL-01 test above).

2. Open Session B on TM-B scoreboard (`in_session(:scoreboard_b)`), confirm `#full_screen_table_monitor_#{@tm_b.id}` present, call `wait_for_actioncable_connection`, install a similar `window._mixupPreventedCount` interceptor for Session B.

3. Define `RAPID_FIRE_COUNT = 6` (even number so both TMs fire equally — 3 each).

4. Loop `RAPID_FIRE_COUNT.times do |i|`:
   - If `i.even?`: `@tm_a.update_columns(state: "ready")` then `TableMonitorJob.perform_now(@tm_a.id)`
   - If `i.odd?`: `@tm_b.update_columns(state: "ready")` then `TableMonitorJob.perform_now(@tm_b.id)`

   Use `update_columns(state: "ready")` (not AASM events) because AASM `whiny_transitions: true` prevents repeated `ready!` calls from the same state. `update_columns` bypasses callbacks and resets state for each broadcast cycle. This is documented in Research Pitfall 1.

5. Positive assertion — Session B: `assert_selector "#full_screen_table_monitor_#{@tm_b.id}", text: /Frei/i, wait: 10` (last TM-B broadcast updated DOM).

6. Positive assertion — Session A: `assert_selector "#full_screen_table_monitor_#{@tm_a.id}", text: /Frei/i, wait: 10` (last TM-A broadcast updated DOM).

7. Negative assertion — Session A (CONC-01 proof): `sleep 2`, then check `window._mixupPreventedCount >= 3` (3 TM-B broadcasts should each have been filtered). Then `refute_selector "#full_screen_table_monitor_#{@tm_b.id}"`.

8. Negative assertion — Session B (CONC-01 proof): `sleep 2`, then check `window._mixupPreventedCount >= 3` (3 TM-A broadcasts should each have been filtered). Then `refute_selector "#full_screen_table_monitor_#{@tm_a.id}"`.

Add thorough inline comments explaining the rapid-fire simulation approach and why `update_columns` is used instead of AASM events. Reference CONC-01 requirement.
