**4. Known Architectural Gap** — Document the structural risk:
- Server broadcasts to global `table-monitor-stream` via CableReady
- ALL connected clients receive EVERY broadcast regardless of which table they are viewing
- Isolation is enforced ENTIRELY client-side in `shouldAcceptOperation` (line ~130-170 of `table_monitor_channel.js`) and the `score:update` DOM event listener (line ~10-40)
- A JS bug, unhandled selector pattern, or new operation type could cause broadcast bleed
- This architecture is correct for the current use case but carries inherent risk

**5. Known Limitations of Testing Approach** — Document:
- `TableMonitorJob.perform_now` is synchronous — cannot simulate true parallel race conditions (real threading)
- Real production load involves multiple Puma threads handling concurrent AASM events simultaneously
- The `sleep 2` negative assertion window is a timing assumption — theoretically a very slow broadcast could arrive after the window
- System tests use a single-threaded Puma server — concurrent requests are serialized

**6. Phase 18 Development Findings** — Extract from 18-01-SUMMARY.md and 18-02-SUMMARY.md deviations:
- `score:update` DOM event IS dispatched to all scoreboard sessions before the channel listener can filter it (CableReady.perform fires the event; the channel's listener returns early for wrong tableMonitorId)
- `User.scoreboard` (scoreboard@carambus.de) does not exist in the fixture DB — caused 500 error on `table_scores` page until `login_as` workaround applied
- `refute_selector` does not accept a failure message string as second argument (Capybara API difference from Minitest assert)
- `update_columns` with a serialized JSON column still invokes Rails serializer — required raw SQL bypass for `data` column

**7. Deferred Fixes** — Verbatim from REQUIREMENTS.md v2 section:
- **FIX-01**: Server-side targeted broadcasts scoped per-table (replace client-side filtering)
- **FIX-02**: Refactor `TableMonitorChannel` to use per-table stream names instead of global stream
- Note: These are tracked in `.planning/REQUIREMENTS.md` under "v2 Requirements"

**8. Recommendation** — Brief recommendation for future milestone planning:
- FIX-01 + FIX-02 should be addressed together (per-table stream names enable server-side targeting)
- The client-side filter code should be retained as a defense-in-depth layer even after server-side fix
- Consider load testing with concurrent Puma threads after server-side fix is implemented
