# Requirements: Carambus API v7.2 — Shootout / Stechen Support

**Status:** SKELETON — produced during Phase 36c (2026-04-14) as v7.1+ groundwork
**Core Value:** KO-style tournaments no longer drop into manual workarounds when two players tie at the end of a decisive game — Carambus owns the Shootout / Stechen resolution flow from AASM through UI.

> This document is a **skeleton** written ahead of v7.2. The v7.1
> discuss-phase will decide whether Shootout work folds into v7.1 or
> becomes a standalone v7.2 milestone. Until then, items here are
> treated as a separate unit of work. **[TBD: v7.1-discuss]** whenever
> the resolution depends on that decision.

## Context

v7.0 Phase 36 review (F-36-35) identified Shootout / Stechen support as
the single largest code gap in Carambus: **there is no mechanism for
resolving a tie at the end of a decisive KO game**. Today the tie is
resolved off-system (paper, verbal agreement, manual CC entry), which
defeats the point of Carambus as a live tournament workflow tool.

The scale of the work justifies a dedicated milestone:
- **AASM changes** — a new state (or set of states) for "tie detected,
  awaiting Shootout resolution"
- **Tournament plan modifications** — the tournament plan PORO must
  know that a KO bracket can have a Shootout appendix and how that
  appendix participates in subsequent rounds
- **Scoreboard UI** — the scoreboards at the physical tables need a
  Shootout mode with different scoring rules, timeout handling, and
  display

Source: `.planning/v7.0-scope-evolution.md` §"New code feature gaps"
row F2.

## v7.2 Requirements

Each requirement uses the `SHO-` prefix (ShootOut) to distinguish from
v7.0 and v7.1.

### AASM state machine

- [ ] **SHO-01**: Tournament AASM gains a `shootout_pending` state (or
  equivalent) that a game can transition into when two players tie at
  the end of a decisive set/match and the tournament mode requires a
  resolution.
- [ ] **SHO-02**: Game AASM gains transitions for `shootout_start`,
  `shootout_complete`, and `shootout_abort` with PaperTrail versioning
  so a Shootout can be replayed or audited.
- [ ] **SHO-03**: The Shootout resolution updates the parent KO
  bracket's "winner of game X" pointer so downstream games see the
  correct player without manual intervention.

### Tournament plan modifications

- [ ] **SHO-04**: The `Tournament::RankingCalculator` and
  `tournament_plan` distribution logic know that a KO bracket's leaf
  games may have a Shootout appendix, and that the appendix result
  replaces the tied leaf result for all downstream purposes.
- [ ] **SHO-05**: The Endrangliste (see v7.1 CCI-01) correctly
  attributes Shootout wins/losses to the winning player, never to
  "both players" or the pre-Shootout tie.
- [ ] **SHO-06**: **[TBD: v7.2]** Does Shootout apply only to KO
  tournaments, or also to round-robin tie-breakers? SME to confirm
  during v7.2 discuss-phase.

### Scoreboard UI

- [ ] **SHO-07**: When a game enters `shootout_pending`, both
  scoreboards at the physical table switch to a dedicated "Shootout
  mode" that displays (a) the current innings/ball count under
  Shootout rules, (b) a clear "Shootout in progress" banner so anyone
  glancing at the board knows this isn't a regular inning, and (c) the
  remaining Shootout attempts if the rules are attempt-based.
- [ ] **SHO-08**: The Shootout mode's scoring rules are configurable
  per discipline (via `Discipline#shootout_rules` or similar) because
  Dreiband, Einband, Cadre, and Free-game all resolve ties differently.
- [ ] **SHO-09**: The Turnier-Monitor view shows Shootout-in-progress
  games as a distinct visual category, not lumped in with "games in
  progress", so the Turnierleiter can glance at it and know whether
  the tournament is running normally or waiting on a tiebreaker.

## Dependencies

- **v7.1 CCI-01 (Endrangliste calculation)** — Shootout results feed
  directly into the final-ranking calculator. v7.2 either depends on
  v7.1 shipping first OR the Endrangliste calculator must be designed
  with a Shootout-shaped placeholder from day one. **[TBD: v7.1-discuss]**
- **v7.0 Phase 36b reset/parameter modals** — Shootout start may need
  a "are you sure this is a tiebreaker?" dialog that reuses the shared
  confirmation modal from 36B-05.

## Open questions (resolve during v7.2 discuss-phase)

1. **Shootout rules per discipline** — Dreiband, Einband, Cadre, Free-
   game, 5-pin each have historical Shootout conventions. Who is the
   ground-truth source? Tournament regulations? SME? **[TBD: v7.2]**
2. **UI split** — is the Shootout a new AASM "game" with its own
   TableMonitor session, or is it a mode toggle inside the existing
   game's TableMonitor? **[TBD: v7.2]**
3. **Historical data** — do existing tournament fixtures contain any
   ties that could be used as characterization data, or does v7.2 need
   to build synthetic test cases? **[TBD: v7.2]**
4. **Scorekeeping authority** — who enters Shootout scores? The same
   referee as the regular game, or a separate Shootout operator?
   **[TBD: v7.2]**
5. **Abort semantics** — what happens if a Shootout starts but one
   player leaves? Fall back to Match-Abbruch handling? **[TBD: v7.2]**
   See `.planning/seeds/match-abbruch-freilos-handling.md`.

## Out of Scope

| Feature | Reason |
|---------|--------|
| ClubCloud integration of Shootout results | Handled by v7.1 CCI-01..03 once Shootout produces structured results. No separate CC work in v7.2. |
| Historical Shootout data backfill | v7.2 is forward-looking. Backfilling old tournaments is a separate data-quality effort. |
| Non-KO tiebreakers (if SHO-06 says "no") | Depends on SHO-06 resolution. |
| New discipline support | v7.2 adds Shootout to existing disciplines. New disciplines are a separate milestone. |
