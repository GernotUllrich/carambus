**Section content requirements:**

### Setup Patterns (per D-04, D-05, CONS-01)
- Standard: Fixtures are the primary setup mechanism. Every test class inherits `fixtures :all` from test_helper.rb.
- Fixture access pattern: `players(:one)`, `tournaments(:imported)`, etc.
- `Model.create!` is acceptable for one-off complex setups where fixtures are insufficient (e.g., KO tournament brackets, state-dependent records needing specific attribute combinations).
- FactoryBot stays in Gemfile but is NOT the default. Existing FactoryBot usage is not a violation but new tests should prefer fixtures.
- Non-compliant: Using `build(:factory_name)` or `create(:factory_name)` when a fixture would suffice.
- Include a decision table:

| Situation | Use |
|-----------|-----|
| Standard test data (players, tournaments, users) | Fixtures |
| Complex multi-record setup (KO brackets, scoring sequences) | Model.create! with explicit attributes |
| Existing FactoryBot usage | Tolerate, don't expand |

### Assertion Style (per D-07, D-08, CONS-02)
- Standard: MiniTest assertions are the baseline (`assert_equal`, `assert_nil`, `assert_includes`, `assert_difference`, `assert_not_nil`, `assert_raises`, `refute`, `refute_nil`, `refute_equal`).
- shoulda-matchers are available and encouraged for concise validation/association tests (`should validate_presence_of`, `should belong_to`, `should have_many`).
- Non-compliant: Using only `assert true` or `assert_not_nil result` when a more specific assertion is possible (e.g., `assert_equal expected, result`).
- Weak assertion checklist (issues to flag in audit):
  - `assert true` — always passes, tests nothing
  - `assert_not_nil x` when x's value should be checked specifically
  - `assert x.valid?` without checking specific error messages
  - Empty test methods (no assertions at all)
  - `assert_nothing_raised` as the ONLY assertion (acceptable in scraping smoke tests)

### Test Naming (per D-06, CONS-03)
- Standard: `test "description of behavior" do` blocks (Rails default).
- Non-compliant: `def test_method_name` style.
- Test descriptions should describe behavior, not implementation: "creates a game when started" not "test_start_game_creates_game".
- Existing `def test_` methods are flagged for conversion but are not blocking issues.

### Helper & Support Files (per CONS-04)
- Document each support file from Task 1 findings:
  - `test/support/scraping_helpers.rb` — purpose, public methods, which tests use it
  - `test/support/snapshot_helpers.rb` — purpose, public methods, which tests use it
  - `test/support/vcr_setup.rb` — purpose, configuration details
  - `test/support/ko_tournament_test_helper.rb` — purpose, public methods, which tests use it
- Note: ScrapingHelpers and SnapshotHelpers are included globally in ActiveSupport::TestCase. Flag if any methods are unused.
- Standard: Helpers should be included only where needed, not globally (unless genuinely used everywhere).

### File Structure
- Standard test file template:
```ruby
# frozen_string_literal: true

require "test_helper"

class ModelNameTest < ActiveSupport::TestCase
  # setup block for shared state
  setup do
    @record = model_names(:fixture_key)
  end

  test "describes expected behavior" do
    # arrange (if needed beyond setup)
    # act
    result = @record.some_method
    # assert
    assert_equal expected_value, result
  end
end
```
- System tests inherit from `ApplicationSystemTestCase`
- Controller tests inherit from `ActionDispatch::IntegrationTest`
- Characterization tests may deviate from standard patterns (they pin existing behavior, not test new behavior)
