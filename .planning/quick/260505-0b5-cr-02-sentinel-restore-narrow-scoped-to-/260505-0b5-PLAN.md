<objective>
Restore the CR-02 thread-local sentinel that was reverted by commit 0dbe45c4 on 2026-05-01.
The revert was made on the basis of a BK-2 single-set validation that proved insufficient:
training-mode BK-2 single-set never enters `TournamentMonitorState#finalize_round` (no
tournament), so the recursion path stayed dormant. Tournament-mode finals after group play
DOES enter `finalize_round` and the recursion is real — confirmed live on
2026-05-05T00:01:45Z, Tournament[17416] / TournamentMonitor[50000028]: `SystemStackError`
after ~323 nested savepoints when an operator clicked "Nächstes Spiel" on a tied 10:10
Einband finals game.

Recursion path:
`TableMonitor#close_match!` (AASM)
  → after-callback `advance_tournament_round_if_present` (table_monitor.rb:1761)
  → `TournamentMonitor::ResultProcessor#advance_round_after_match_close` (result_processor.rb:122)
  → `@tournament_monitor.finalize_round` (lib/tournament_monitor_state.rb:54)
  → loops over `tabmons` calling `tabmon.close_match!` for each
  → re-enters `advance_tournament_round_if_present` for the SAME outer TM → recurse → SystemStackError.

The fix is one small additive guard (SKILL extend-before-build), narrow-scoped to per-TM
identity (`Thread.current[:_advancing_round_for_tm] = self.id`) so a legitimate cross-TM
cascade where Plan 38.8-04's deferred logic happens to invoke close_match! on a DIFFERENT
TM in the same flow is NOT blocked.
