## What Was Done

### Task 1 — Engine fix (`perform_switch_to_next_set`)

Inserted between the BK-2kombi `rebake_at_set_open!` hook and the AASM state assignment:

```ruby
if @tm.game.present?
  g_data = @tm.game.data
  if g_data["tiebreak_winner"].present?
    g_data.delete("tiebreak_winner")
    @tm.game.data = g_data
    @tm.game.save!
  end
end
```

**Note on the read-mutate-assign-back pattern:** `Game#data` (app/models/game.rb#114) is a custom getter that calls `read_attribute_before_type_cast(:data)` and decodes a NEW hash on every invocation. The plan's straightforward `@tm.game.data.delete("tiebreak_winner") + @tm.game.save!` did not actually persist the deletion (mutated a transient hash). The corrected pattern reads into a local, mutates, and assigns back via `data=`, then saves. This was discovered during Task 2 test execution (Test 1's `assert_nil` failed with the stale `"playera"` value).

Two commits captured this:
- `12978591` — initial guard block (intent + lifecycle ordering rationale).
- `86ca3e62` — corrected hash-mutation pattern (read-mutate-assign-back).
